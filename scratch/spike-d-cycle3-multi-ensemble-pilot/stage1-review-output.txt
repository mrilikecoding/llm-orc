# Code Review: Session Budget Tracking Module

## Executive Summary

This patch introduces `SessionBudget` and `SessionBudgetRegistry` for per-session token budget enforcement. The diff is small (90 lines) but contains several issues that should be addressed, including one **critical cross-file consistency problem** and a **real security concern** with API key logging.

---

## 1. Logic / Semantic Issues

### 🔴 CRITICAL: Unchecked `get()` will raise unhandled `KeyError`

**Lines 90–92:**
```python
def get(self, session_id: str) -> SessionBudget:
    """Look up the budget for a session."""
    return self._budgets[session_id]
```

`SessionBudgetRegistry.get()` performs a direct dictionary lookup with no guard. Any caller requesting a non-existent session will receive an unhandled `KeyError`. This is problematic because:

- `add_usage()` (line 94–98) calls `get()`, propagating the crash upward
- The Orchestrator Runtime calling `check_limit()` before each iteration could crash if a session was somehow unregistered

**Recommendation:** Either raise a descriptive custom exception (e.g., `SessionNotFoundError`) or return `None` with proper null handling by callers.

---

### 🟡 Potential Integer Overflow

**Lines 44–46:**
```python
def add_tokens(self, input_count: int, output_count: int) -> None:
    self.input_tokens_used += input_count
    self.output_tokens_used += output_count
```

Token counters accumulate indefinitely without overflow protection. For long-running sessions with high-throughput LLM calls, these values could exceed typical integer ranges (though Python's arbitrary-precision integers mitigate this). Consider adding a sanity check or overflow guard.

---

### 🟡 `remaining()` Returns Negative Values

**Lines 66–67:**
```python
def remaining(self) -> int:
    return self.limit - self.total_used()
```

If `check_limit()` returns `False`, `remaining()` will return a negative integer. This may be unexpected for callers. Document this behavior or consider returning `max(0, ...)` with a note.

---

### 🟢 `check_limit()` Logging Side Effect

**Lines 52–62:**
The `check_limit()` method logs a warning as a side effect when the budget is exceeded. This conflates checking with reporting. Callers may call `check_limit()` purely for the boolean return value but unexpectedly trigger logging.

**Recommendation:** Consider separating the check from the log, or document this dual behavior clearly.

---

## 2. Security / Sensitive-Data Issues

### 🔴 HIGH PRIORITY: API Key Logged on Budget Exceeded

**Lines 58–61:**
```python
logger.warning(
    f"Session {self.session_id} exceeded budget: "
    f"{self.total_used()}/{self.limit} tokens. "
    f"API key: {self.api_key}"
)
```

The `api_key` attribute is being written to the log stream when a session exceeds its budget. This is a **real security concern** confirmed by the script-agent's detection. API keys should never appear in logs because:

- Logs are often aggregated to external services (Datadog, CloudWatch, ELK)
- Access to log aggregation systems is typically broader than access to credentials
- Log retention policies may keep sensitive data longer than intended
- This violates least-privilege principles for logging

**Recommendation:** Remove `f"API key: {self.api_key}"` from the log message entirely. If correlation is needed for debugging, log a truncated hash or masked version of the key instead.

---

### 🟢 Token Count in Logs (False Positive)

**Lines 60:** The script-agent flagged "token" as a sensitive term in the log message. This is a false positive — token *counts* (integers) are not sensitive data. Only the `api_key` logging is a genuine issue.

## 6. Other Concerns

### 🟢 In-Memory Registry — No Persistence or Cleanup

**Line 76:**
```python
self._budgets: dict[str, SessionBudget] = {}
```

The registry is purely in-memory. Per the comment: "the registry is in-memory only, so test coverage is not blocking." This is acceptable for v1 but note:
- Sessions are never removed from the registry
- No TTL or expiration mechanism exists
- Server restarts wipe all session budget state

If sessions can be long-lived, this could cause unbounded memory growth. Consider adding a `unregister()` method or implementing periodic cleanup.

### 🟢 No Bounds Checking on `add_tokens` Inputs

**Lines 44–46:** Negative token counts could produce invalid totals. Validate that `input_count >= 0` and `output_count >= 0`, or document the assumption that callers are trusted.

### 🟢 Missing `__repr__` for Debugging

Neither `SessionBudget` nor `SessionBudgetRegistry` implements `__repr__`. Given that this system deals with session IDs and API keys, a debug-friendly representation would aid troubleshooting.
