## Code Review: `src/llm_orc/agentic/session_budget.py`

### 1. Logic / Semantic Issues

**`remaining()` returns negative on overflow** (line 72 diff)
```python
def remaining(self) -> int:
    return self.limit - self.total_used()
```
If tokens exceed limit, this returns a negative value. Callers who treat this as unsigned will underflow. Add a guard: `return max(0, self.limit - self.total_used())`.

**`add_tokens()` accepts negative counts** (line 52 diff)
```python
def add_tokens(self, input_count: int, output_count: int) -> None:
    self.input_tokens_used += input_count
    self.output_tokens_used += output_count
```
Negative tokens are silently accepted, allowing callers to artificially inflate a budget. Add: `if input_count < 0 or output_count < 0: raise ValueError(...)`.

**`get()` raises `KeyError` on missing session** (line 82 diff)
```python
def get(self, session_id: str) -> SessionBudget:
    return self._budgets[session_id]
```
No graceful handling. A missing session_id should return `None` or raise a typed `SessionNotFoundError`, not `KeyError`. Use `self._budgets.get(session_id)` and decide on the null-handling strategy.

**No limit validation in `SessionBudget.__init__`** (line 40 diff)
A zero or negative limit is silently accepted. `check_limit()` will return `False` immediately, but the session is registered as if functional.

---

### 2. Security / Sensitive-Data Issues

**API key logged in warning message** (lines 60–63 diff)
```python
logger.warning(
    f"Session {self.session_id} exceeded budget: "
    f"{self.total_used()}/{self.limit} tokens. "
    f"API key: {self.api_key}"   # ← PII in logs
)
```
This is a direct credential leak. If these logs flow to a log aggregator (Datadog, CloudWatch, etc.), the API key is exposed. The `session_id` alone is sufficient for debugging. Remove `api_key` from the log message entirely.

**API key stored in `SessionBudget` with no encryption** (line 39 diff)
The dataclass holds `api_key: str` in plain memory for the lifetime of the session. If threat model includes memory-scraping attacks, this is an attack surface. The attribute is included in `__repr__` if dataclass uses default `repr`, potentially leaking the key to debug output. Consider:
- Omitting `api_key` from the dataclass and using it only at the billing layer.
- Using `repr=False` on the field if dataclass generates `__repr__`.

---

### 3. Type-Safety / API-Contract Issues

**Wrong default value annotation** (line 78 diff)
```python
def register(self, session_id: str, api_key: str, limit: int = None) -> SessionBudget:
```
Should be `limit: int | None = None`. In Python < 3.10 this raises `TypeError` at call time if someone passes no argument and strict type-checking is enabled. Correct the annotation.

**`get_metadata()` returns internal dict reference** (line 74 diff)
```python
def get_metadata(self) -> dict[str, Any]:
    return self.metadata
```
Returns the raw internal `dict`. Callers can mutate `self.metadata` directly. Return `dict(self.metadata)` (a shallow copy) to enforce immutability of internal state.

**`all_sessions()` returns internal dict reference** (line 88 diff)
```python
def all_sessions(self) -> dict[str, SessionBudget]:
    return self._budgets
```
Same issue as `get_metadata()`. Return `dict(self._budgets)` to prevent callers from modifying registry state.

**Inconsistent limit type handling** (line 79 diff)
```python
limit=limit if limit else DEFAULT_BUDGET_LIMIT,
```
`if limit` is falsy for `0`. If a caller legitimately passes `limit=0`, this evaluates as falsy and silently uses the default. Use `if limit is not None` instead.

---

### 4. Test Coverage

The PR message explicitly defers tests, citing that "the budget logic is straightforward." The issues above contradict that claim:

- The API key logging bug requires a test that asserts no credential appears in log output.
- The negative-token path requires a test for rejection behavior.
- `KeyError` vs typed error on missing session requires explicit test coverage.
- The `if limit else` falsy trap needs a test with `limit=0`.

Without tests, these bugs remain latent. The statement that tests are "not blocking" should be reconsidered given the severity of the credential leak in particular.

---

### 5. Cross-File Consistency

**Critical: `DEFAULT_BUDGET_LIMIT` vs `DEFAULT_TOKEN_LIMIT` vs `DEFAULT_MAX_TOKEN_LIMIT`** (line 18 diff vs `orchestrator_config.py:38,43`)

The diff comment explicitly flags this as a silent-divergence risk. The actual values confirm the drift:

| Constant | Value |
|---|---|
| `session_budget.DEFAULT_BUDGET_LIMIT` | 100,000 |
| `orchestrator_config.DEFAULT_TOKEN_LIMIT` | 10,000,000 |
| `orchestrator_config.DEFAULT_MAX_TOKEN_LIMIT` | 50,000,000 |

These are 100× apart. Either the new module's default is absurdly restrictive (sessions die at 100K tokens vs. 10M elsewhere) or the existing defaults are absurdly permissive. They cannot both be correct. This needs resolution before merge. The new module should import from `orchestrator_config` rather than duplicating the constant.

**Dual budget systems** — `budget_controller.py` enforces turn/token limits via `BudgetController.check()`, and `session_budget.py` enforces token limits via `SessionBudget.check_limit()`. Both exist, both do token limiting, but they are independent code paths. If they diverge, a session could pass the `BudgetController` check but fail `SessionBudget`, or vice versa. Document which wins, or consolidate into one.

**`add_usage()` has a race condition** (lines 84–87 diff)
```python
def add_usage(self, session_id: str, input_count: int, output_count: int) -> bool:
    budget = self.get(session_id)
    budget.add_tokens(input_count, output_count)
    return budget.check_limit()
```
If two coroutines/tasks call `add_usage` concurrently for the same `session_id`, token counts can be added and checked in an interleaved manner, leading to over-consumption beyond the limit. The in-memory `dict` is also not thread-safe. Consider `asyncio.Lock` or `threading.Lock` per session.

---

### 6. Other Concerns

**`metadata: dict[str, Any]` field with no schema** — Arbitrary dict injected into `SessionBudget` with no type constraints or validation. If this is used for logging/persistence downstream, schema drift will be silent. Consider a typed schema or `TypedDict`.

**No session cleanup / unregistration** — The registry grows indefinitely. Long-running processes will accumulate stale `SessionBudget` objects. Add a `unregister(session_id)` method.

**Missing `__init__.py` export** — `src/llm_orc/agentic/__init__.py` was read; `session_budget` is not exported or mentioned. Consumers importing from `llm_orc.agentic` won't find `SessionBudget` or `SessionBudgetRegistry` without a direct import. Consider re-exporting from the package `__init__` for discoverability.
