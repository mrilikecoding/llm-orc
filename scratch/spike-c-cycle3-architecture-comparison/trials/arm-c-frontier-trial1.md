### 1. Logic / Semantic Issues

**Line 57 — Off-by-one in budget enforcement.**
`check_limit` returns `False` when `total_used() > self.limit`. This means a session that has consumed exactly `limit` tokens is considered within budget and the next iteration is allowed to proceed, potentially pushing usage to `limit + iteration_cost`. The correct guard is `>=`:

```python
if self.total_used() >= self.limit:
```

Whether "at the limit" means "stop" or "continue" is a design decision, but the current behavior silently permits overage by one iteration's worth of tokens. The docstring says "exceeded" but the code allows equality, which is a contract mismatch regardless.

**Line 74 — `limit=None` default with a falsy guard.**
`register` has `limit: int = None` and then uses `limit if limit else DEFAULT_BUDGET_LIMIT`. Two distinct problems:

1. If a caller explicitly passes `limit=0` (a valid, if aggressive, budget), the condition is falsy and the value is silently replaced with `DEFAULT_BUDGET_LIMIT`. `0` is a legitimate sentinel meaning "block all usage immediately"; it will be ignored.
2. `None` is the correct sentinel for "not provided," but the guard should be `if limit is not None` rather than truthiness.

**Lines 20–22 — Hardcoded constant with a silent divergence risk.**
The comment itself flags the issue: `DEFAULT_BUDGET_LIMIT` must stay in sync with `DEFAULT_MAX_TOKEN_LIMIT` in `orchestrator_config.py`. There is no import, assertion, or shared constant enforcing this. When they drift, both enforcement paths run without error — one just silently permits more usage than the other intends. The constant should be imported from `orchestrator_config.py`, not redeclared here.

**Lines 63–86 — `get()` and `add_usage()` raise `KeyError` on unknown session.**
`get` does a bare dict lookup with no error handling. Any caller that passes an unregistered `session_id` (race condition, replay, deserialization mismatch) gets an unhandled `KeyError`. This should raise a domain-typed exception (`UnknownSessionError` or similar) so callers can distinguish "session not found" from an unexpected programming error.

**Lines 83–86 — `add_usage` records tokens before checking the limit.**
The usage is committed to the budget before `check_limit` determines the session is over budget. If the caller treats `False` as "reject this iteration," the tokens from the rejected call are still recorded, artificially inflating usage. The more defensible pattern is to check the limit *before* committing, or at minimum document that token recording is unconditional regardless of the return value.

**Line 88 — `all_sessions()` returns the live internal dict.**
Callers receive a direct reference to `self._budgets`. Mutations through this reference bypass all registry logic. Should return a shallow copy: `return dict(self._budgets)`.

### 2. Security / Sensitive-Data Issues

**Lines 53–57 — API key logged at WARNING level.**
```python
logger.warning(
    f"Session {self.session_id} exceeded budget: "
    f"{self.total_used()}/{self.limit} tokens. "
    f"API key: {self.api_key}"
)
```
This emits the raw API key to the application log on every budget overrun. Depending on log aggregation (stdout capture, Datadog, Sentry, CloudWatch), this surfaces a billing credential in plaintext wherever logs flow. Log only a masked or truncated form (e.g., `api_key[:8] + "..."`) or omit the key entirely — the session ID is sufficient to correlate with billing records.

**Line 31 — `api_key` stored as a plain string field on the dataclass.**
The dataclass is not marked frozen, so `api_key` can be mutated after construction. More importantly, `api_key` participates in the default `__repr__` generated by `@dataclass`, meaning `repr(budget)` or any debug print of the object will emit the key. Add `field(repr=False)` to suppress it from `__repr__` and `__str__`.

### 3. Type-Safety / API-Contract Issues

**Line 74 — `limit: int = None` violates the declared type.**
The parameter is annotated `int` but defaults to `None`, which is `int | None`. With `mypy --strict` (mandated by project CLAUDE.md), this is a type error. Fix:

```python
def register(self, session_id: str, api_key: str, limit: int | None = None) -> SessionBudget:
```

**Line 78 — `get()` return type is `SessionBudget` but may raise.**
The annotation implies a non-optional return. The `KeyError` is undocumented and untyped. Either annotate the exception in a docstring and raise a named exception, or return `SessionBudget | None` and let callers handle absence explicitly.

**Line 14 — `from typing import Any` is unused if dropped.**
`metadata: dict[str, Any]` on the dataclass and `get_metadata()` return type use `Any`. This is acceptable, but worth noting that `metadata` has no documented schema — consider `dict[str, str | int | float | bool]` if the expected payloads are known, to avoid burying type errors in an `Any` escape hatch.

**Line 11 — `from __future__ import annotations` interacts with `dataclass`.**
With `from __future__ import annotations`, all annotations are strings at runtime. `@dataclass` in Python 3.10+ handles this correctly, but `field(default_factory=dict)` combined with string-annotations can produce surprising behavior in some third-party tooling. Not a bug here, but worth verifying against the runtime Python version in CI.

### 4. Test Coverage

The commit message explicitly defers tests to a follow-up PR and asserts the logic is "straightforward." This is the most significant process concern in the diff.

The following cases are not straightforward and have real defect potential without tests:

Shipping logic with security implications (credential logging, billing enforcement) without tests, against a project standard (CLAUDE.md) that mandates the TDD Red-Green-Refactor cycle, is not defensible on "straightforward" grounds. ### 6. Other Concerns

**`remaining()` can return a negative value.**
Once the budget is exceeded, `remaining()` returns a negative integer. Callers that use this value as a "tokens available for next call" guard may not handle negative correctly. Either clamp to zero (`return max(0, self.limit - self.total_used())`) or document that negative values are expected and meaningful.

**No eviction or TTL on `SessionBudgetRegistry`.**
The registry is in-memory and unbounded. Long-running processes with high session turnover will accumulate `SessionBudget` objects indefinitely. This is noted as "in-memory only" in the PR description, but there is no cleanup path, no `deregister` method, and no documentation of the expected cardinality. If the registry is singleton-scoped in the Orchestrator Runtime, this is a slow memory leak. At minimum, a `deregister(session_id)` method and a note on ownership should be present.

**`add_tokens` accepts negative values silently.**
`add_tokens` does not validate that `input_count` and `output_count` are non-negative. A caller that passes a negative value (accounting correction, bug) will reduce `total_used`, potentially pulling a session back under budget after it was flagged. Add a guard:

```python
if input_count < 0 or output_count < 0:
    raise ValueError(f"Token counts must be non-negative: {input_count=}, {output_count=}")
```

**Commit message defers tests against project standards.**
CLAUDE.md mandates "Only commit when: ALL tests pass." A new module with zero test coverage does not satisfy this gate, regardless of complexity. The "follow-up PR" pattern for tests is a known anti-pattern in TDD-mandated codebases — the follow-up rarely arrives before the module is depended upon.
