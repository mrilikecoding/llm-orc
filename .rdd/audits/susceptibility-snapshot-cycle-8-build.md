## Interpretation

Split this the way the ARCHITECT-phase snapshot did: separate what's earned from what's carried forward unexamined.

**Gates 1-3 read as earned agreement, not rubber-stamping, and I checked rather than took the commit messages' word for it.** Gate 2's reorder claim (step 2 before step 0) is a dependency-ordering claim about code, not a taste call, and it's the kind of thing that either works or breaks the build; it worked (2307 unit + 31 integration green, mypy/ruff clean). Gate 3's three judgment calls are checkable against the actual diff, and I checked: `test_plexus_adapter.py` shows in the deletion commit's stat as a rename (`{agentic => core/session}/test_plexus_adapter.py`), matching the "recategorized from delete to migrate" claim exactly; `test_orchestrator_config.py` and `test_tool_dispatch_composition.py` show as full deletions, matching the other two claims; the `TOOL_NAMES`/ADR-003 reserved-name references disappear alongside the dissolved orchestrator surface they protected, matching the "vestigial, entailed by ADR-046" reasoning. None of this was narrated-but-unverifiable; it is independently reproducible from `git show 6974176 --stat`. That is what earned confidence looks like at the BUILD phase position: claims that survive an outside check.

That verification also has a cycle-level precedent that matters for reading the "(Recommended)"-every-time pattern correctly. The ARCHITECT-phase snapshot for this cycle (`docs/agentic-serving/housekeeping/audits/susceptibility-snapshot-cycle-8-architect.md`) flagged two unexamined module-boundary questions (Serving Registry, Accept Gate) and recommended a Grounding Reframe before BUILD scaffolded code around them. Both were actually run, not just logged: the WP-C8 gate note (`.rdd/gates/cycle-8-c8-gate.md`) records a genuine Challenge move where the agent surfaced an operator-correctness tension the "verification-shape map" framing concealed, and the practitioner resolved it onto a substantive good-defaults-vs-extension boundary rather than accepting the framing whole; WP-D8's roadmap entry records the ADR-048 fixture rerun Grounding Reframe as executed and complete (2026-07-03). So by the time WP-F8 opened, the practitioner had two recent, first-hand data points that this agent's "(Recommended)" markings, when the stakes are architectural or code-level, come with real alternatives and get checked. Accepting the recommendation at gates 1-3 of WP-F8 is consistent with calibrated trust built on that track record inside this same cycle, not blind deference.

**Gate 4 is where that track record doesn't transfer, and it's where a real gap surfaced.** Documentation completeness isn't something the test suite grades, which is exactly the residual-risk category this dispatch asked me to watch for. The doc-sweep scope wasn't a BUILD-time invention to scrutinize; it was inherited from ARCHITECT's Amendment #21 three weeks earlier, and nobody at any of the four WP-F8 gates re-asked "is that list still the right list." It wasn't: `architecture-map.md`, built 2026-06-25 specifically as a current-state reconciliation with code-line citations ("Built... from three grounding reads: the design docs, the code, the ADRs"), is not in the six files the in-progress sweep is touching (confirmed via `git status`), and it still describes `ClientToolActionTerminal → LoopDriver.decide()` as the ACTIVE path with citations like `loop_driver.py:498-536` and `loop_driver.py:905` to a file this same session just deleted. `ORIENTATION.md`, itself mid-edit in this sweep, points readers to `architecture-map.md` as supporting reference material (line 43), so the sweep is actively editing the document that sends readers to the one that will be wrong.

This is a scope question the scenarios never specified (`WP-F8`'s own roadmap entry says "Scenarios covered: none directly"), and it closed without anyone examining the alternative of a corpus-wide grep against the docs tree, which is exactly the check that would have caught it and which I ran in about a minute. That's the finding that fits this dispatch's framing precisely: not a test failure, not a code defect, but a decision-gate commitment (the doc-sweep's scope) that inherited a framing from three weeks and one phase transition earlier without re-examination, at the one moment (the deletion landing) when that framing's completeness had a concrete, checkable answer.

## Recommendation

**Grounding Reframe recommended: narrow, concrete, and actionable before the doc sweep commits and Cycle-8 BUILD closes.**

**What is uncertain:** whether `architecture-map.md` (and, lower-priority, the `single_step_enforcer.py:17-19` line citation in `scenarios.md`'s refactor scenario) was deliberately scoped out of the current-state sweep, or simply never re-examined since ARCHITECT fixed the doc list three weeks before this session ran. Nothing in the corpus (roadmap, gate notes, ARCHITECT amendment) records a decision to exclude it; `ORIENTATION.md`, which the sweep is actively editing, still points to it as reference material.

**The concrete grounding action:** before committing the in-progress doc sweep, run a corpus-wide grep for the retired module names (`orchestrator_runtime`, `loop_driver`, `calibration_gate`, `calibration_signal_channel`, `delegation_rate_meter`, `client_tool_action_terminal`, `artifact_bridge`, `result_summarizer`, `single_step_enforcer`, `tier_escalation_router`, `operator_terminal_event_sink`, `orchestrator_context_event_sink`, `sibling_interface_extractor`, `dispatch_event_substrate`, `inference_wait_heartbeat`, `budget_controller`, `autonomy_policy`) restricted to current-state-facing docs (`system-design.md`, `system-design.agents.md`, `ORIENTATION.md`, `field-guide.md`, `architecture-map.md`, `roadmap.md`, `scenarios.md`, `cycle-status.md`), and reconcile every hit outside the historical-record documents (ADRs, cycle-archive, essays, audits, which are legitimately point-in-time). At minimum, add a banner to `architecture-map.md` noting it describes the pre-Cycle-8 dissolved architecture and is superseded by `system-design.md` §Cycle 8, the same treatment already given to other retired artifacts in this corpus.

**What the practitioner is building on without this grounding:** `ORIENTATION.md` is being edited right now to reflect the post-deletion state while still pointing readers to a document that contradicts it. Any future reader (human or agent) who follows that pointer inherits a wrong current-state picture with specific, dead code-line citations, exactly the kind of stale-documentation trap this corpus's own graduate-time discipline (clean main / research-branch split) is meant to prevent. The fix is cheap right now, while the sweep commit is still open; it gets more expensive once Cycle-8 BUILD closes and the corpus moves to graduate.
