# ADR-033: Layer-A Loop-Driver as the Multi-Turn Agentic Control Structure (Callee)

> **Updated by ADR-043 on 2026-06-18.** Cycle 7 loop-back #9 supersedes **§Decision 1** (surface-mode discrimination): the two-surface split collapses to one — all chat-completions requests engage this loop-driver, validated by Spike ι. Decisions 2–6 and the four fitness criteria are unchanged and now govern all requests, not only tool-driven ones. The §Decision 1 discriminator (`_is_tool_driven` selecting between two callers) is retired.

**Status:** Proposed

**Date:** 2026-06-01

---

## Context

Cycle 7 reached BUILD WP-A (the chat-completions handler swapped to the ADR-027 framework-driven dispatch pipeline), and a BUILD reflection-gate surfaced that the pipeline emits text only and is single-turn: it returns `ContentDelta` + `Completion`, never `finish_reason: "tool_calls"`, and has no stage that participates in a client's multi-turn tool loop. The practitioner's north-star is to drive a tool-rich client (OpenCode confirmed) against agentic-serving while a long-horizon process (RDD or a similar skill framework) runs, trusting that the work is delegated to ensembles. A text-only single-turn pipeline cannot serve that flow. The cycle looped back BUILD → RESEARCH → DISCOVER → MODEL, then re-entered DECIDE.

The loop-back established (essay-outline-006 Amendment B; spikes π/ρ/σ; domain-model OQ #26 / OQ #27 + AS-9 scope boundary):

- **The north-star session is a multi-turn agentic loop, and ADR-027's `plan → dispatch → synthesize` pipeline is single-turn.** Spike σ named two layers the session needs: **layer A** drives the loop (decide the next agentic step per turn: read / edit / write / bash / finish), and **layer B** generates the per-turn content. ADR-027's pipeline is a layer-B mechanism. **No ADR-027 component holds layer A** (the routing-planner decides *which capability*, not *what the next agentic step is*; the synthesizer produces a response from given context, it does not drive). The layer-A loop-driver is the load-bearing role the loop-back surfaced as missing.
- **The wrapper-vs-callee fork (OQ #26) was held genuinely open for DECIDE**, with the "ADR-027 is incomplete, not wrong; the fix is additive" framing flagged as the wrapper reading and deliberately not inherited as the default (it minimizes upstream rework, which is not evidence of correctness). Every multi-turn run the loop-back produced (σ.2) was callee-shaped; no wrapper-as-loop was ever spiked, so the fork could not be settled on evidence without a wrapper-shaped probe.
- **The grounded-loop hypothesis (OQ #27) was a working inference, not a finding** (σ.1/σ.2: n=1, short batchable task; σ.2 batched all three actions in one planning turn, so a real decide-act-observe chain was never exercised). Its conditional reopening of AS-9's scope was widened at the MODEL gate to two axes: axis 1 (grounding: the driver slips into ungrounded batch planning) and axis 2 (sequential composition: trajectory drift over many decisions even with per-turn grounding intact).

DECIDE-entry ran two free local probes (research log `essays/research-logs/cycle-7-spike-tau-upsilon-decide-entry-probes.md`; $0; OpenCode 1.15.5 + Ollama qwen3:14b/qwen3:8b):

- **Spike τ (grounded-loop falsification, OQ #27 axis 1; n=4):** on a non-collapsible task (a random code visible only in stdout, which the driver must observe and carry into a `write`), the unconstrained qwen3:14b driver batched `bash`+`write` in one turn every time and committed the write to an unobserved value (3× a literal placeholder `${bash_output}`, 1× a fabricated code). 0/4 stepped to observe. This is exactly the axis-1 ungrounded-batch failure OQ #27 names. It also reinterprets σ.2: that loop succeeded only because the calc task was batchable; it was never a grounded chain.
- **Spike τ′ (structural mitigation; n=3):** when the framework forces one action per turn (execute the first tool call, return its result, force re-planning), the same driver carried the exact observed code into the write every time. Grounding is achievable cheaply, but as a framework-enforcement property, not a model assumption.
- **Spike υ (wrapper-shaped, OQ #26; n=1):** running the full `plan → dispatch → synthesize` pipeline as the per-turn generation subroutine worked end-to-end (the test passed), but each write cost ~35-50s (three serialized model calls) vs the callee shape's single ensemble call (~10-15s), roughly 3× per-turn latency. The synthesizer returned the dispatched code verbatim both times (`synth_differs_from_dispatch=False`): for tool-call content the synthesize stage is redundant, and the planner stage re-decides a capability the loop-driver already chose.

The practitioner resolved the fork toward callee at DECIDE-entry (2026-06-01), with the loop-driver's long-horizon coherence (axis 2) accepted as a named BUILD/PLAY validation target rather than a blocker.

This ADR records the layer-A control-structure decision. ADR-034 records the client-tool-action terminal that the loop-driver uses to deliver work to the client.

## Decision

**The agentic-serving chat-completions surface engages a layer-A loop-driver as the control structure for multi-turn, tool-driven clients, and delegates per-turn generation to a single capability ensemble (the callee resolution of OQ #26).**

1. **Surface-mode discrimination.** When a chat-completions request carries client `tools[]` (a tool-driven client prepared to execute tool calls, e.g., OpenCode's build agent with `write`/`edit`/`bash`/`read`), the surface engages the layer-A loop-driver. When a request carries no client tools (a non-agentic answer-a-question request), the surface continues to use ADR-027's `plan → dispatch → synthesize` pipeline as the single-turn mechanism. The two surface-modes coexist; the discriminator is the presence of client tools in the request. *(Per the Provenance check: this discriminator is a drafting-time design choice, grounded in Spike π Phase 0's observation that tool-driven clients send `tools[]` with `tool_choice: "auto"` on the agent turn, but not itself settled by spike evidence. ARCHITECT/BUILD validates the discriminator, including the edge case of a tool-capable client that wants a plain answer on a given turn — the loop-driver handles this by finishing with text, so engaging the driver when tools are present is safe, but the branch is a design commitment to confirm, not a measured result.)*

2. **The layer-A loop-driver drives, the ensembles generate (callee).** Each turn, the loop-driver decides the next agentic step (which client tool to call, or finish). When a step requires generated content (file content for a `write`, etc.), the loop-driver delegates that generation to a **single capability ensemble** selected for the step's task. The `plan → dispatch → synthesize` pipeline is **not** the per-turn generation primitive on this surface: per Spike υ it triples per-turn latency and its planner and synthesize stages are redundant when a loop-driver already decides the step and the deliverable is tool-call content rather than a chat response.

3. **Framework-enforced single-action-per-turn.** The framework executes one client tool call per turn, returns its result to the loop-driver, and forces re-planning before the next action. The driver is structurally prevented from emitting a batch whose later steps presuppose unobserved outputs. Per Spike τ vs τ′, this enforcement is what makes grounded driving reliable on a cheap-tier driver: grounding is a property the framework guarantees structurally, not a behavior assumed of the model. **This is a fitness criterion** (see below). *(Scope note: the τ′ result was produced by a scratch-proxy stand-in that truncated the driver's batch to its first tool call. The production enforcement mechanism is unresolved and is ARCHITECT/BUILD design: candidates are batch-truncation, a re-planning prompt that requests one action at a time, or a one-tool `tool_choice` constraint on the driver. Batch-truncation is the technique the τ′ probe actually used, so it carries direct evidence and is the starting prior; the other two are untested candidates. The decision here is that single-action-per-turn is enforced by the framework; the enforcement technique is not yet selected.)*

4. **Grounded-loop hypothesis: Conditional Acceptance (ADR-097 pattern).** The hypothesis that a cheap structurally-bounded role can reliably drive a grounded loop is **conditionally accepted, conditional on the framework-enforced single-action-per-turn constraint above.** Empirical basis: Spike τ (n=4 falsification of the unconstrained driver) + Spike τ′ (n=3 grounded pass under enforcement), on short tasks at qwen3:14b. **Axis 2 (long-horizon trajectory coherence over many turns, the regime AS-9's single-decision basis never tested) is designated a BUILD/PLAY validation target** against the real built loop-driver on a real long task; the short DECIDE probes do not and cannot settle it. The Conditional Acceptance is discharged to full acceptance when BUILD/PLAY produces sustained-long-horizon evidence, or revised if that evidence surfaces axis-2 failure.

5. **Seat-filler and the load-bearing open risk.** The loop-driver occupies the client's "model" seat (Spike σ candidate (a)): the model the client talks to *is* the loop-driver, with the framework interposing single-step enforcement and per-turn ensemble delegation. Which model fills the seat is ARCHITECT/BUILD selection work. **The recorded load-bearing open risk is the driver-capability bet:** whether a cheap-tier driver sustains long-horizon coherence (preserving the cost-distribution value proposition), or a more capable driver is required (reviving the cost-distribution tension and the hybrid question). τ/τ′ established the grounding floor (short-task grounding under enforcement), not the long-horizon ceiling. This risk is carried explicitly into BUILD/PLAY, backstopped by the Conditional Acceptance above; it is not resolved here.

6. **Gate-folded constraints (2026-06-01, from evaluating the settled/open split against the north star).**
   - **(a) Seat-filler swappability is settled.** The loop-driver's model is selected via a swappable Model Profile (the ADR-011 pattern). The *choice* of model is open (the cheap-vs-capable bet); the *swappability* is a settled ARCHITECT constraint — it is the structural precondition that lets BUILD/PLAY resolve the bet by a configuration change rather than a re-architecture. Without it, the bet the whole callee resolution depends on cannot be run cleanly.
   - **(b) Axis-2 validation is north-star-shaped.** Axis 2 is validated by a real skill-framework / RDD-style session driven through a tool-rich client (OpenCode), not a synthetic long task. That run is the outcome that validates the cycle's prerequisite claim (cheap orchestration + ensemble delegation matching a frontier single model on long-horizon agentic work); a synthetic task would muddy the signal, the same logic that rejected the wrapper-callee hybrid.
   - **(c) Full client tool surface.** A real north-star run presents OpenCode's full declared tool surface (`skill`, `task`, `todowrite`, plus `read`/`write`/`edit`/`bash` — observed in Spike π Phase 0, which flagged `skill`/`task` as "the surface a future 'run RDD via OpenCode' flow would use"). The loop-driver must handle that surface, not only the file-action tools the spikes exercised. The file-action tools (`write`/`edit`/`bash`) are the BUILD focus; the richer surface is north-star context ARCHITECT/BUILD must not be blindsided by.

### Fitness criteria introduced

- **FC (single-step enforcement):** on the tool-driven surface, the framework emits at most one client tool call per assistant turn; any additional tool calls the driver proposes in the same turn are not dispatched until the first call's result is returned and the driver re-plans. Refutable: a turn that dispatches two client tool calls before returning the first's result violates this.
- **FC (callee generation):** per-turn content generation routes to a single capability ensemble invocation, not through the `plan → dispatch → synthesize` pipeline, on the tool-driven surface. Refutable: a per-turn generation that invokes the routing-planner + synthesizer stages violates this.
- **FC (grounded carry):** when a turn's action depends on a prior turn's observed tool result, the action uses the observed value, not a placeholder or fabricated value. Refutable: a tool-call argument containing an unresolved template reference to an unobserved output (the τ failure signature) violates this.
- **FC (seat-filler swappability):** the loop-driver's model is selected via a Model Profile; substituting the driver model (cheap-tier ↔ capable-tier) requires no change to the loop-driver control structure, the client-tool-action terminal, or the ensemble-delegation path. Refutable: a driver-model change that requires editing the loop-driver / terminal / delegation code violates this.

### Relationship to ADR-027

ADR-027's `plan → dispatch → synthesize` pipeline is **scoped to the single-turn (non-tool-driven) answer-a-question surface and the layer-B generation role.** It is not superseded: it remains the correct mechanism for a client that sends a request expecting a synthesized text answer, and the response-synthesizer's strict-fidelity rules (ADR-029) continue to govern that surface. ADR-033 adds the layer-A control structure above it for the tool-driven multi-turn surface, where per-turn generation is callee-delegated to a single ensemble rather than run through the full pipeline. ADR-027 carries a dated `> Updated by ADR-033 on 2026-06-01.` partial-update header recording this scoping; its body is preserved.

### Relationship to AS-9

AS-9 (structurally-bounded LLM roles produce reliable output on single-decision-shaped tasks) is honored and extended in shape, not amended. The loop-driver is a structurally-bounded role in the AS-9 sense, with the bound made explicit as single-action-per-turn. AS-9's scope-boundary annotation (per-turn agentic-loop-driving held outside AS-9's validated single-decision scope until validated) is updated by the domain-model Amendment Log to record that axis 1 (grounding) is conditionally validated under single-step enforcement, and axis 2 remains a BUILD/PLAY target. No invariant claim changes; the annotation records the conditional resolution.

## Rejected alternatives

### Wrapper: the `plan → dispatch → synthesize` pipeline as the per-turn generation subroutine

A layer-A loop-driver runs above ADR-027's pipeline, and each turn's generation invokes the full pipeline (planner + dispatch + synthesizer). The "ADR-027 is incomplete, not wrong; the fix is additive" reading.

**Rejected because:** Spike υ ran this shape deliberately (to avoid settling the fork by callee-skew) and measured its cost: ~3× per-turn generation latency (three serialized model calls per write vs one), with the synthesize stage producing output identical to the dispatched ensemble content (redundant, and a corruption risk since the synthesizer is shaped for chat responses, not tool-call content) and the planner stage re-deciding a capability the loop-driver already selected. *(Latency-comparison scope: the ~3× figure was measured on a batchable two-write task without single-action-per-turn enforcement on either shape. Under production single-step enforcement both shapes incur N driver round-trips; the per-generation 3-calls-vs-1 ratio is the stable comparison, and the direction — callee leaner — holds, but the absolute session numbers shift with horizon length and are not what υ measured.)* The wrapper is not strictly dominated: its pipeline stages would earn their keep if a *single agentic turn* genuinely needed multi-capability composition (search-then-summarize-then-write within one step) — a niche no spike tested. But in the north-star flow the loop composes across turns, so per-turn generation is almost always single-capability, and the wrapper's stages are dead weight for the common case. **Recorded residual (gate-adjudicated 2026-06-01, F3-1):** if production traffic surfaces a real need for per-turn multi-capability composition, the pipeline-as-subroutine remains available for that specific step type without changing the callee control structure. At the DECIDE gate the practitioner could not name a concrete reopening trigger, so this is a **recorded concession, not a watched contingency** — it is not elevated to a live design alternative and carries no monitoring obligation. The practitioner's rationale for keeping the commitment clean rather than admitting a wrapper-callee hybrid: a clean single approach produces an *interpretable* BUILD/PLAY outcome, whereas a hybrid would muddy the axis-2 long-horizon signal the cycle most needs to read. The practitioner noted a "spectrum" intuition (wrapper-vs-callee may not be strictly binary); it is recorded here as acknowledged but not pursued, to be revisited only if a BUILD/PLAY outcome demands it.

### Unconstrained loop-driver (no single-action-per-turn enforcement)

The loop-driver emits whatever tool calls it chooses per turn, batched or not, with no framework constraint.

**Rejected because:** Spike τ (n=4) showed the unconstrained cheap driver batches and commits later steps to unobserved values (placeholder `${bash_output}`, fabricated codes) on any task with a cross-step runtime dependency. This is the OQ #27 axis-1 failure. Without structural enforcement, grounded driving is not reliable on a cheap-tier driver. The enforcement is the load-bearing mechanism, not an optimization.

### Extend the routing-planner (ADR-028) to drive the per-turn loop

The routing-planner ensemble, already present for the single-turn surface, is extended to decide each turn's next agentic step.

**Rejected because:** the routing-planner's bounded role is *which capability matches a request* (a single-decision classification validated by Spike ζ). Deciding the next agentic step in a multi-turn loop (read vs edit vs write vs bash vs finish, conditioned on observed tool results) is a different task shape. Conflating the two would either over-scope the planner (re-introducing a bundled-reasoning surface AS-9 warns against) or under-serve the loop. The loop-driver is a distinct role; reusing the planner's name for it would blur the layer-A / layer-B distinction the loop-back established. The seat-filler may share a model tier with the planner, but the role is separate.

### Frontier-tier loop-driver as the default seat-filler

The loop-driver runs at frontier tier (Claude Opus, GPT-5, or comparable) to maximize long-horizon coherence.

**Rejected as the default because:** it defeats the cost-distribution value proposition (the loop-driver runs every turn; a frontier driver on every turn is the expensive case the architecture exists to avoid), and the Khanal et al. MOP finding (domain-model methodology vocabulary) weakens the assumption that frontier models are reliably better on long-horizon tasks. The default seat-filler is a cheap-tier driver under single-step enforcement (per τ′). Frontier-tier is the **named fallback** if BUILD/PLAY axis-2 validation shows a cheap driver cannot sustain the horizon; that contingency is the load-bearing open risk recorded in the Decision, not the starting commitment.

## Consequences

### Positive

- **The north-star multi-turn agentic flow becomes serviceable.** The surface can drive a tool-rich client's loop, delegate per-turn work to ensembles, and (with ADR-034's terminal) deliver that work to the client's filesystem with parity (Spike ρ/σ).
- **Callee is lean and matches the grain of the north-star process.** One driver call + one ensemble call per turn, vs the wrapper's three. Composition happens across turns via the loop, which is how the client already decomposes work.
- **Grounding is structurally guaranteed, cheaply.** Single-step enforcement makes a cheap-tier driver ground reliably (τ′), so the cost-distribution value proposition holds at the generation layer regardless of how the long-horizon driver-capability question resolves.
- **The layer-A / layer-B split clarifies the architecture.** Driving and generating are separated; each is a bounded role.

### Negative

- **The loop-driver's long-horizon coherence (axis 2) is unvalidated.** τ/τ′ proved the short-task grounding floor under enforcement; they did not prove a cheap driver holds a coherent trajectory over dozens of turns. This is the load-bearing open risk; it is carried to BUILD/PLAY under Conditional Acceptance, and it intersects the cost-distribution value proposition directly. **Compound-cost scenario:** if axis-2 validation forces a capable (frontier-tier) driver, the two cost dimensions compound rather than simply add — a higher per-turn cost (capable driver on every turn) combined with the higher turn count that single-action-per-turn enforcement produces on tasks a driver would otherwise have batched into one turn. (The turn-count inflation is task-dependent: it bites on batchable tasks the enforcement splits into N turns, not on tasks that were inherently N-step regardless.) The worst case for the cost-distribution value proposition is "capable driver required AND long, batchable-but-split horizon," where both factors compound. This is the scenario that would most strongly revive the hybrid question; it is the precise shape of the axis-2 risk BUILD/PLAY must surface.
- **Single-step enforcement multiplies turn count.** A task the driver would have batched into one turn becomes N turns, each a driver round-trip plus an ensemble call, serialized. Local models are slow; a long session is many minutes of wall-clock. Accepted under the parity-is-behavioral-not-latency commitment (the practitioner accepted local-model latency as a tuning tradeoff), but it is a real cost and compounds with horizon length.
- **Two surface-modes now coexist** (single-turn pipeline for non-tool requests; layer-A loop-driver for tool-driven requests). The discriminator (presence of client tools) is simple, but the surface carries two code paths. The single-turn pipeline (ADR-027) is the simpler mode; the loop-driver mode is the new BUILD load.
- **The seat-filler choice is deferred and consequential.** ARCHITECT/BUILD selects the driver model; the choice is partly a bet on the axis-2 question, which is not yet evidenced.

### Neutral

- **The OpenAI chat-completions API contract is preserved.** A tool-driven client sees standard `finish_reason: "tool_calls"` responses and standard tool-result follow-ups (ADR-034). The architectural change is internal.
- **The wrapper shape remains available for a narrow niche** (per-turn multi-capability composition), recorded as a residual rather than foreclosed.
- **`ClientToolCall` emission, removed from the handler in commit `0a7a822` (WP-A), returns** under ADR-034; the `OrchestratorChunk` union still includes `ClientToolCall`. ADR-033 establishes the control structure that decides when to emit it.
