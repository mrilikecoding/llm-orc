## Context

Finding H (cycle-status §"Finding H," 2026-06-08): the first live trajectory run validated the axis-1 mechanism end to end. Real OpenCode to a working-tree `llm-orc serve` to qwen3:14b seat and a `code-generator` ensemble built a 5-file dependent temperature-conversion library and the session routed, advanced through all five distinct deliverables with zero churn, delegated at rate 1.0, and converged COMPLETE. Capability-matched routing even emerged unprompted (the four code files routed to `code-generator`, the README to `prose-improver`). The orchestration mechanism Cycle 7 built holds live, at depth five.

But the generated *content* degraded across files, the axis-2 frontier (callee content fidelity) ADR-033 §6b scoped as the structurally-non-decomposable target. `cli.py` called `converters.convert_temperature(...)`, a function that does not exist in `converters.py` (which exposes three specific pairwise functions). `README.md` documented `fahrenheit_to_kelvin` and a Rankine scale that do not exist. `test_cli.py` asserted behavior `cli.py` does not implement.

Root cause, structural and not only model capability: the model issued **zero reads** across the whole run. Each deliverable was generated from the task description plus the framework digest, which records each prior action as `(action, path, result)` and **never the file's contents**. So the callee writing `cli.py` never saw `converters.py`'s actual API and guessed wrong. The sibling API exists on disk; it is simply not fed to the callee dispatch (the input the capability ensemble receives when the seat-filler delegates via `invoke_ensemble`).

This rhymes with Finding G / ADR-038 one layer down. Finding G was multi-file *progress* unanchored (the judge's "what remains" was computed and discarded); ADR-038 routed that already-computed signal forward. Finding H is multi-file *content* unanchored (the sibling API exists on disk and is not fed to the callee); the fix is the same shape, route the already-available signal forward.

Why this is a parity blocker, not a PLAY-deferrable nicety (practitioner disposition at the Finding H gate): the north star is the cheap-local orchestrator plus ensembles reaching "all that remains is ensemble iteration," and cross-file coherence is where that bites. A frontier seat was explicitly declined as a contrast: masking the gap with a more capable model proves it skippable for a model not being targeted, and qwen3:14b is not the production coder. The production `code-generator` coder runs at `agentic-tier-cheap-general`, **qwen3:8b** (the cheap tier; serve.log dispatch-0002/4/6/8 all `tier=cheap`, no Calibration-Gate escalation fired), so Finding H's content failure was produced by the cheap coder, and the cheap coder is where the fix must work.

Spike ξ (pre-registered, methods-reviewed before any run; research log `essays/research-logs/cycle-7-spike-xi-content-anchor.md`) measured the fix at the coder-generation layer, qwen3:8b, using the ensemble's real coder `system_prompt`. The primary outcome is AST-checkable: does a generated dependent file reference only symbols that exist in its produced siblings? Two bases (converters to cli, an impl-to-impl dependency; a non-guessable text-utility module to its test, a test-to-impl dependency), n=10 per cell:

- **Baseline reproduces Finding H via invention.** A_current (no anchor) resolved 3/10 (Base T, 7 invented) and 0/10 (Base V, 9 invented). Blind, the coder invented `convert_temperature` (T) and `clean_text` / `slugify` / `to_snake_case` (V), never the pinned API.
- **The signatures anchor fixes it.** B_signatures (the produced sibling's public API surface routed into the callee dispatch) resolved **10/10 on both bases**. The cheap coder, handed the real signatures, stopped guessing.
- **The specific content is the mechanism, causally isolated.** B 10/10 versus Control_decoy 0/10 (API-shaped signatures with wrong names) versus Control_filler 1/10 (length-matched non-API prose), on Base T. Neither API-shape alone nor extra tokens alone reliably breaks the guessing; the real sibling API does (B minus decoy = 1.0, B minus filler = 0.9, both far past the 0.3 isolation gate). The model's own priors are weak but insufficient (the filler arm resolved one trial fully and two only partially), so the anchor supplies what the priors cannot.
- **The fix is callee-agnostic (prose confirmed).** A practitioner-directed prose arm (Base P, the `prose-improver` coder generating `README.md`) measured the README failure Finding H also exhibited. Blind, A_current resolved 0/10 (every README invented functions — lower than code's 3/10, a comparison that crosses resolver types; the blind READMEs confidently document a whole conversion library, most of it nonexistent); the signatures anchor resolved 10/10, and signatures sufficed (B = C_full = 10/10). Nothing about the failure is code-specific: same cheap qwen3:8b coder, same blindness, same invention.

ADR-036 settled the delegation *decision* (the seat-filler delegates; the framework wins that decision by V3 user-turn composition and the seat names a capability ensemble with an input brief and a filePath). It did not give the callee the produced siblings' content. This ADR settles that information flow.

## Decision

**On a delegated generation that writes a file into a session with already-produced file deliverables, the framework augments the callee dispatch context with the produced siblings' interface, extracted by the framework from the real produced files. The default and universal path is the sibling's full content (type-blind, so it sources an anchor from any content type); where the framework has a structural extractor for the sibling's type, it compacts to the public API surface (signatures). The signal is sourced from the real files on disk, never model-composed and never guessed, and no sibling content type can break the mechanism.**

The information ADR-037's Session Action Record and the workspace already hold (the produced files) is no longer left out of the callee dispatch. It is read by the framework and routed forward to steer generation, the Finding-G shape applied to content rather than progress. **Content-agnosticism is the design commitment** (the north star is OpenCode + ensembles writing arbitrary local files): the mechanism is agnostic on both sides — it fires for any callee (deliverable side) and sources an anchor from any sibling type (source side, via the full-content baseline). Signature extraction is a frugality optimization layered on the agnostic baseline, never a precondition.

Concretely:

- **Form: full content is the content-agnostic baseline; signatures are a compaction optimization.** The universal path is the produced sibling's **full content** — type-blind (it is the file's bytes), so it sources an anchor from any sibling type: code, config, doc, anything. Where the framework has a structural extractor for the sibling's type (Python AST now; pluggable for other languages later), it compacts the anchor to the public API surface (signatures + one-line docstrings), the frugal path that does not bloat as the deliverable count grows. A sibling type with no extractor falls back to full content, so **no content type breaks the mechanism**. Spike ξ validated both paths: signatures resolved 10/10 (code, Base T/V) and 10/10 (prose, Base P); full content resolved 8/10 (code Base T — two misses: one inline reimplementation, one syntax error) and 10/10 (prose). For code, signatures win (frugal, and they avoid full-content's two misses); on prose B = C. So signatures are the optimization for extractable types and full content is the agnostic guarantee for the rest. (The earlier draft framed this "signatures, not full content," which inverted the priority for a content-agnostic goal: full content must be the default that guarantees agnosticism, with signatures layered on where extractable.)
- **Sourcing: framework-read, never guessed.** The framework reads the produced files (via the existing Session Artifact Store read-side, the `read_deliverable` accessor) and extracts the signature surface. Anchor correctness is load-bearing: Control_decoy (wrong signatures) resolved 0/10, *below* the 3/10 unanchored baseline, because the coder obeyed the wrong API and invented `convert_temperature` as instructed. A wrong anchor is worse than none, so the framework must source the anchor from the real produced file, which it structurally can.
- **Injection point: the callee dispatch, extending ADR-036.** ADR-036 wins the *decision* to delegate and the seat composes the input brief; ADR-039 augments *what the callee actually receives* with the framework-sourced sibling signatures, on the framework-to-callee hop. The seat-filler's model-composed brief is not the carrier (the model does not know the API, having issued no reads); the framework appends the signatures to the dispatch context before the capability ensemble runs.
- **Content-agnostic, both sides.** *Deliverable side:* the augmentation fires on the callee dispatch regardless of which capability ensemble is invoked. Finding H's failure appeared in both code (`cli.py` and the tests, via `code-generator`) and prose (`README.md`, via `prose-improver`), and the practitioner-directed prose arm confirmed the fix is identical (blind 0/10 → anchored 10/10; same cheap qwen3:8b coder, same blindness; the cross-resolver caveat noted below). *Source side:* the full-content baseline is type-blind, so it sources an anchor from any sibling type, not only code — guaranteed by construction (full content is the file's bytes; the framework does not parse the sibling on the universal path), and confirmed beyond code by the cross-type config probe (Base G — a non-code `settings.json` data sibling whose Python consumer must reference real keys, not functions). The mechanism does not special-case code on either side, because nothing about the cross-file-coherence failure is code-specific.

The spike validated injecting the **single relevant** sibling's signatures. Whether production injects all produced files' signatures or a dependency-inferred subset is a selection-policy detail deferred to BUILD, bounded by the same context-budget reasoning that selected signatures over full content (a many-file session should not carry every file's signatures if a subset suffices). The Decision commits the mechanism (framework-sourced signatures into the callee dispatch); it does not commit a specific multi-sibling selection policy, which is unmeasured here.

This is additive to ADR-036 (a new framework augmentation on the callee dispatch surface), not an amendment of its delegation-decision mechanism: V3 user-turn composition, the win-not-coerce property, and the delegation-rate instrumentation all stand. ADR-036 gains a dated "Extended by ADR-039" header on the callee-dispatch surface.

### Fitness criteria introduced

- **FC (content-anchor presence):** on a delegated write into a session with already-produced file deliverables, the callee dispatch context contains the relevant produced siblings' API signatures. Refutable from dispatch-context inspection. (The selection scope, all produced files versus a dependency-inferred subset, is a BUILD detail; Spike ξ validated injecting the single relevant sibling, not a multi-sibling selection policy.)
- **FC (anchor sourced from real files, never guessed):** every injected signature names a symbol that actually exists in the produced file it is drawn from (a structural extraction, not a model composition). Refutable: an injected signature naming a symbol absent from the produced file. This FC is the structural guarantee that the decoy failure mode (a wrong anchor, worse than none) cannot occur in production.
- **FC (signatures form, not full bodies):** the anchor carries signatures and docstrings, not full function bodies. Refutable from dispatch-context inspection.
- **FC (delegation and form preserved under the anchor):** the anchored dispatch still delegates generation (no Finding B inline-write regression) and the produced file still honors ADR-035's form contract (no regression of the deliverable form). Refutable: an inline write of generated content on an anchored dispatch, or a form-contract violation traceable to the added context.
- **FC (cross-file reference resolution, the outcome):** a generated dependent deliverable references only symbols that exist in its produced siblings, for code AND prose deliverables. Refutable: for code, an AST-resolvable cross-file reference to a nonexistent sibling symbol (the Finding H signature); for prose, a documented sibling-API identifier that does not exist (the README's invented `fahrenheit_to_kelvin`), resolved by a prose-robust identifier heuristic rather than AST. This promotes Spike ξ's primary outcome to a standing fitness criterion across both callee types.

## Rejected alternatives

### Full-content anchor (inject whole sibling files)

Measured, not assumed: C_full resolved 8/10 (Base T) and 10/10 (Base V), a working fix, but it does not beat signatures (10/10 on both) and was lower on Base T, its two misses being one inline reimplementation that sidestepped the dependency and one syntax error. Rejected on the measured margin plus the context-budget argument: full content bloats the dispatch as the deliverable count grows (a five-file session would carry five full files), where signatures stay compact. Held as the fallback if a future dependency needs more than the API surface (for example behavior or invariants that signatures do not carry); the n=10/cell caveat means 8-versus-10 is not a distinguishable population rate, so the preference rests on signatures' equal-or-better rate and the budget argument, not a demonstrated reliability gap.

### Induce the model to read (guidance-only read-back by the model)

The seat holds the read tool and used it zero times across the live run; a guidance nudge to read could in principle recover the behavior. Rejected as the primary mechanism, and not run, for three reasons. First, layer mismatch: the read decision is the qwen3:14b seat's, but the content failure is the qwen3:8b coder's, so even a successful read would still need its content routed to the coder, which is the framework injection this ADR specifies anyway. Second, it adds a read round-trip per dependent turn (latency that compounds across a multi-file session). Third, it relies on the model *choosing* to read reliably, where framework-sourced injection is unconditional. The framework-injection result was decisive on the cheap coder (10/10), so the lighter model-read path was deferred to a secondary probe rather than run. Held available if a future case needs content the framework cannot cheaply pre-extract.

### A more capable coder seat (escalate to qwen3:14b or a cloud model)

Rejected by the practitioner's Finding H disposition (the cloud-contrast was declined): masking a structural information-flow gap with a frontier seat proves it skippable for a model not being targeted, and the north star is the cheap-local coder. The evidence makes escalation unnecessary, not merely undesirable: the cheap qwen3:8b coder resolved 10/10 *with* the anchor, so the failure was the missing information flow, not a capability ceiling. The free-first capability ladder (qwen3:8b, then the in-stack qwen3:14b escalation tier at $0, then cloud) never had to climb past its first rung, and no cloud spend was incurred.

### Mere context perturbation, or any API-shaped context (the nulls the controls test)

Two controls, not one (the methods-review P1-A finding: a content-wrong decoy alone leaves "API-format-as-a-constraint" uncontrolled). Control_filler, length-matched non-API prose, resolved 1/10 (approximately baseline): extra tokens do not fix the guessing. Control_decoy, API-shaped signatures with wrong names, resolved 0/10: API-shape alone does not fix it either, and worse, the coder followed the wrong names into invention. So the specific sibling content is causally responsible, B minus filler = 0.9 and B minus decoy = 1.0, both far past the 0.3 gate. These are not rejected designs; they are the refuted nulls that license the mechanism claim "the real sibling API breaks the guessing, where neither plausible API-shaped content nor arbitrary tokens does." The decoy's sub-baseline result is itself a finding, recorded as the anchor-correctness FC: the framework must never hand the coder a guessed API.

## Consequences

**Positive:**
- Dependent deliverables reference real sibling APIs: cross-file resolution moved from 3/10 to 10/10 (Base T code) and 0/10 to 10/10 (Base V code), from 0/10 to 10/10 on the prose README (Base P), and from 0/10 to 9/10 on a non-code JSON data sibling (Base G, the full-content path), all on the cheap qwen3:8b coder, in the coder-generation harness. The Finding H blocker is removed at the dispatch layer in that harness across code, prose, and data siblings; the real-client confirmation is the discharge gate, not yet cleared.
- The fix reuses already-available information (the produced files on disk), no new model call and no new role; a framework read plus a signature extraction. It rhymes with ADR-038, route the already-available signal forward.
- The specific-content mechanism is causally isolated (decoy 0/10, filler 1/10 versus B 10/10), so the mechanism is understood, not a lucky perturbation.
- The fix works on the production cheap tier, so the cross-file-coherence piece of the cheap-local north star holds without a capability-floor escalation.
- The signature-compaction path is compact, so the dispatch does not bloat as the deliverable count grows on types with an extractor; the full-content fallback is heavier but bounded (it carries only the produced siblings, and the BUILD-deferred selection policy can cap it).

**Negative:**
- The framework now reads produced files on each dependent dispatch (full content, compacted to signatures where an extractor exists for the sibling's type), a new framework step and a new runtime dependency on the produced files being readable through the Session Artifact Store read-side.
- Validated at the coder-generation layer (Spike ξ harness, qwen3:8b, single-hop dependencies), not yet end to end against the real client on a multi-file trajectory. The real-OpenCode trajectory re-run is the Conditional Acceptance discharge gate; the Finding H lesson is exactly that a synthetic pass can hide a real-client gap.
- Anchor correctness is load-bearing (a wrong anchor resolved 0/10, below the 3/10 baseline). The sourcing-from-real-files FC is the structural guarantee, but a signature-extraction defect that misreports the API would actively harm, so the extractor is a correctness-critical component.
- Signature *extraction* is language-specific (the spike used Python AST), but this is a handled case, not a boundary: a sibling type with no extractor falls back to the full-content baseline (the universal, type-blind path), so content-agnosticism holds regardless. Adding an extractor for a new language is a frugality optimization, never a precondition. What remains a recorded boundary is the per-type *effectiveness* of full content beyond the measured types (code, prose, and the cross-type config/data sibling): structurally the full-content path is type-blind (it injects the real bytes), and the worst case for an exotic type is a benign no-op (the model ignores an unhelpful-but-correct anchor, degrading to the unanchored baseline, never below it, since the anchor is always the real file and so cannot trigger the decoy failure mode).
- Scope is the measured class: qwen3:8b coder, single-hop dependencies, across three structurally-different sibling types — Python code (signatures), prose README (full-text), and JSON data (keys). The remaining boundary: deeper dependency chains (a file depending on a file that depends on a file), non-file deliverables, exotic sibling types beyond the three measured (the full-content path is type-blind by construction, so these are an effectiveness question with a benign-no-op worst case, not a content-agnosticism gap), and the prose/config outcomes being identifier heuristics rather than AST-clean resolution (the 0/10-versus-9-10/10 gaps are far outside any plausible extraction-noise margin, but those FCs' refutation is heuristic, not exact). Prose-to-code coherence is no longer scoped out: it was measured (0/10 → 10/10), so the README is a discharge-gate criterion, not merely observed.

**Neutral:**
- The form-contract bleed (critic review prose leaking into `test_converters.py`, Finding H's secondary observation) is a separate ADR-035 FormGate question, not addressed here.
- The progressive task-shape ladder's content-coherence rung is now grounded at the harness layer; the ladder continues from deeper and prose shapes.
- Like ADR-036 and ADR-038, the win is a property of the validated (composition × qwen3:8b) pair; a coder-profile change re-validates the anchored dispatch (the same per-profile re-validation discipline ADR-036's profile-swap FCs already carry).

## Empirical grounding (ADR-097 filter)

**Grounding path: spike validation.** Spike ξ, pre-registered and methods-reviewed before any run (2 P1 / 4 P2 / 2 P3 applied; the second causal-isolation control `Control_filler` and the Base-V non-guessable re-pin are reviewer-directed), with an AST-checkable primary outcome for the code and config bases (the config consumer is Python — string-key subscripts resolved against the real keys; no adjudication subjectivity, the ρ P1-A class structurally avoided) and a regex-heuristic outcome for the prose base, full-n denominators, two code bases plus a practitioner-directed prose base and a practitioner-directed cross-type config/data base, a baseline-validity guard that passed (A_current reproduced Finding H predominantly by invention on every base, after Base G was re-pinned from a generic-loader task that the blind consumer dodged), and a three-way causal isolation refuting both the perturbation null and the any-API-context null. A post-review fidelity correction set the generation model to qwen3:8b, the production cheap coder where Finding H's content failure was produced. The prose arm (Base P, the `prose-improver` README callee) confirmed callee-agnosticism: A_current 0/10 → B_signatures 10/10, signatures sufficing (B = C_full). The practitioner-directed cross-type config arm (Base G, a non-code `settings.json` data sibling) confirmed source-side content-agnosticism: A_current 0/10 → B_content 9/10 via the full-content path, with causal isolation holding on the data type (Control_decoy 0/10). The four bases span two code domains (converters + text_tools) plus two structurally-different sibling types — prose README (full-text) and JSON data (keys, a distinct domain and file kind) — establishing content-agnosticism across the code/prose/data spread, not within code alone; exotic sibling types beyond these three remain a benign-no-op effectiveness boundary (the full-content path is type-blind by construction). Research log `essays/research-logs/cycle-7-spike-xi-content-anchor.md`; methods review `housekeeping/audits/research-methods-spike-xi.md`; artifacts `scratch/spike-xi-content-anchor/`.

**Conditional Acceptance: the discharge gate is the BUILD real-OpenCode multi-file trajectory re-run.** A single real-OpenCode session on the exact 5-file Finding H temperature-library task, with the content-anchor wired into the callee dispatch, in which the dependent deliverables reference real sibling APIs, verified by reading the landed files: `cli.py` calls the real `converters` functions, the tests match the implementation, AND the README documents real functions (no invented `fahrenheit_to_kelvin`, no Rankine scale), while the axis-1 properties hold (zero churn, convergence). The README is a discharge criterion, not merely observed: the prose arm established that a prose callee, given the sibling signatures, documents the real API (0/10 → 10/10 in the harness, where the anchor was injected directly into the generation). The discharge run confirms the *framework* actually delivers that anchor to `prose-improver` (not only `code-generator`) end-to-end under the real client. This is the same layer-match discipline ADR-037 and ADR-038 carried. The harness PASS is necessary, not sufficient: the synthetic ladder passed axis-1 honestly and the real run still exposed axis-2, so the in-harness 10/10 is confirmed under the real client before the status flips to full Acceptance.

Unlike ADR-038, whose Conditional Acceptance is already discharged (the WP-LB-L real-OpenCode multi-file run), ADR-039's discharge is pending. The route-the-signal-forward rhyme with ADR-038 is structural; it does not transfer ADR-038's cleared empirical confidence to this ADR, whose end-to-end behavior under the real client is still unmeasured.
