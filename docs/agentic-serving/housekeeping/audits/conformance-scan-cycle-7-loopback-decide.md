## Notes

**Pattern of non-conformance.** All twelve violations are BUILD-work or a single refactor-now (the stale docstring comment). There is no code that actively contradicts the ADRs in a way that would corrupt an already-running system — `0a7a822` removed the old `ClientToolCall` plumbing cleanly rather than leaving a broken half-implementation. The debt is entirely in the gap between what WP-A shipped (a correct ADR-027 pipeline for single-turn text) and what ADR-033/ADR-034 now require (a layer-A loop-driver for multi-turn tool-driven clients).

**What `0a7a822` removed that ADR-034 requires back.** Four specific pieces were excised and must return:

1. `ClientToolCall` import in `v1_chat_completions.py` (removed from lines 61–65).
2. `tool_calls` field on `_NonStreamingResult` and `finish_reason: "tool_calls"` logic in `_collect_non_streaming` (`v1_chat_completions.py:565–594`).
3. The `isinstance(chunk, ClientToolCall)` branch that set those fields.
4. The `message["tool_calls"] = result.tool_calls` + `content: null` assignment in `_build_completion_body()` (`v1_chat_completions.py:713–746`).

The streaming formatter case (`sse_format.py:66–72`) was **not** removed and is correct. Re-introduction targets are the non-streaming collector and body-builder.

**The artifact-bridge gap is the highest-priority BUILD design dependency.** The `SessionArtifactStore` has no `read_deliverable` method — this is the single missing API on an otherwise complete substrate. The ADR-034 §Decision 3 terminal chain (`loop-driver decides write → ensemble writes artifact → framework reads artifact → emit tool_call(content = artifact content)`) cannot close without it. BUILD scenario-writing should include a scenario that drives through this chain end-to-end, since fidelity at scale is acknowledged as unvalidated (spikes used trivially small content).

**The `dispatch_pipeline.py` `_extract_request()` silently drops `role: "tool"` messages.** At line 325–335 it extracts only the last `role: "user"` message. When a `role: "tool"` follow-up arrives (the client's tool result), the current pipeline would treat the prior user message as the request and discard the tool result. This is not a defect in ADR-027's single-turn pipeline (which this function correctly serves) but it is a concrete footgun for BUILD: the loop-driver's message routing must not call `_extract_request()` or must extend it to surface tool results.

**No server-side client-filesystem write exists.** The Spike π Phase A rejected shape (server writes directly to client workspace) was never introduced into the current codebase. ADR-034's FC (no server-side client-filesystem write) is satisfied by absence. No violation there.
