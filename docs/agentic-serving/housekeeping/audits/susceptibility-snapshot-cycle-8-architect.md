## Interpretation

Split the session into two populations, as the DECIDE-phase snapshot did, and the same split holds up.

**Most of the new content is formalization of DECIDE's already-converged text, not ARCHITECT-time invention.** classify→seat→marshal-as-a-sequence, the Topaz-keyed parts/shape-catalog vocabulary, the executor/judge/gate composition, the AS-2 admission gate, the six system-level fitness criteria — every one of these traces cleanly to ADR-044/046/047/048, and I checked each against its source ADR rather than taking the system-design table's provenance column on faith. This is the low-risk category and it is fine.

**One category of decision is genuinely ARCHITECT's to make, not inherited from DECIDE: how many modules, and where the boundary lines fall.** None of the five ADRs allocates module boundaries — they were written at the level of decisions (registry shape, gate composition, per-turn sequence), and the module table is the first place in the corpus that turns those decisions into a specific count and shape of architectural units. This is exactly the kind of "boundary serves the developer's mental model vs. the user's" risk the dispatch named, and it is where the session's engagement was uneven.

The marshal case is a genuine example of the warrant-probe process working, not a rubber stamp. The agent proposed consolidating two retired modules (Artifact Bridge, Client-Tool-Action Terminal) into one marshal node; the practitioner tested that against a stated principle rather than accepting or rejecting by feel; the agent re-derived a three-node split and, notably, did not stop at asserting it — it attached a specific, testable fitness claim (emit is caller-agnostic, verified by an integration test that drives emit from both the deliverable path and the Accept Gate's ODP-1 executor-delegation path). That is what earned convergence looks like: the conclusion changed, and the change came with a way to falsify it.

The same session did not apply that process to Serving Registry or Accept Gate, which are structurally the same kind of question (is this one architectural unit or several; what does the split cost or buy). Two observations sharpen this rather than just restating the dispatch's suspicion:

1. **Serving Registry's internal parts/shapes split is not the ARCHITECT-level risk** — that vocabulary is DECIDE-derived (ADR-047 §1, itself grounded in the operator's own llm-conductor pattern-library mining, keyed by "capability × shape × output-type"), and it maps reasonably onto the Ensemble Author/Operator's own stated mental model ("the system is a library of capabilities... the orchestrator draws from," `product-discovery.md` §Ensemble Author/Operator). The dispatch's worry that parts-vs-shapes might be the developer's split rather than the operator's is not well supported once you check where the vocabulary actually came from. The real open question is narrower than the dispatch posed it: not "is parts-vs-shapes the right operator model" but "does that DECIDE-level distinction need its own L1 module, or is it data/config the existing Composition Validator + Capability List Builder pairing could carry." That question was never separately argued.

2. **Accept Gate's module status is internally inconsistent, not just unexamined.** `system-design.md` gives it full peer treatment alongside classify/seat/marshal (a Module entry, two Fitness lines, dependency edges). `roadmap.md` WP-D8 calls the identical thing "a build-shape segment of the Serving Ensemble." Both documents were produced in the same ARCHITECT pass. That disagreement is direct textual evidence that no single moment settled whether Accept Gate is a peer architectural unit or an inline sub-graph — the two documents just inherited whichever template their section happened to use.

A smaller, concrete symptom of the same pattern: the Test Architecture table (`system-design.md`, the "marshal → I/O Envelope → client" row) still describes a single "marshal test" verifying both content-shaping and un-parseable-deliverable refusal — behavior that, after the split, belongs to two different nodes (shape and form-gate). The Integration Contracts section three paragraphs above it correctly describes the three-node sub-sequence. The late re-decomposition propagated to some sections of the document and not others, inside the same artifact, in the same session. This is a small, mechanical instance of exactly the compounding-without-verification risk the dispatch was asking about, visible by direct inspection rather than inference.

**Framing-adoption reflex, assessed as asked.** The "defer to BUILD" discipline has now been applied consistently across DECIDE and ARCHITECT without incident, and the practitioner has already parked an explicit graduate-time reconciliation note (main-branch-vs-research-branch) showing active tracking, not blind momentum. But the accumulation is visible in WP-F8, which now bundles four distinct kinds of work (the ~12.7K-LOC code deletion, six ADR supersession headers, migrating three cycles' worth of WPs to a Completed Work Log, and a four-document current-state sweep) under one work package with no internal size breakdown, sequenced only by the single hard WP-B8 dependency. That is not a defect in the discipline itself — deferring documentation-of-an-unbuilt-architecture is sound, and I re-derived the same conclusion the corpus already reached — but WP-F8 is the one Cycle-8 work package that does not get an Objective-scaled breakdown the way WP-A8 through E8 do ("Scenarios covered: none directly" is the tell). Worth a size estimate before BUILD opens it, not urgent given it is last-sequenced by construction (parity-before-delete).

**Product-facing reading, assessed as asked.** classify/seat/marshal staying invisible to the Tool User is fine — a real coding model does not narrate its own routing either, so invisibility here matches rather than violates the "endpoint is a model" abstraction. The one live product-facing thread worth naming forward: if ODP-1 resolves toward client-delegated execution, the Accept Gate's executor rounds will surface to the Tool User as ordinary-looking tool_calls (run tests, etc.) — which likely *helps* rather than hurts the "endpoint is a model" abstraction, since a real agentic coding session also runs tests and iterates visibly. That is a genuinely useful connection the corpus has not yet made explicit, but ODP-1 is correctly still open and deferred to BUILD/PLAY, so this is a note for that decision point, not a gap in the current artifact.

## Recommendation

**Grounding Reframe recommended — narrow scope, in-cycle applicable at the ARCHITECT→BUILD boundary.**

**What is uncertain:** whether Serving Registry warrants a dedicated L1 module, or whether the Topaz-keyed parts + operator-curated shape catalog it holds is better carried as owned state/config on the existing Composition Validator + Capability List Builder pairing; and whether Accept Gate warrants full peer-Module status with its own dependency edges and Fitness lines, or whether `roadmap.md`'s own "build-shape segment of the Serving Ensemble" framing is the more accurate one — i.e., three inline seats (executor/judge/gate) within the Serving Ensemble's build branch rather than a standalone module.

**The concrete grounding action:** apply the same warrant probe already used once this session, explicitly, to both boundaries before BUILD scaffolds code around them. The test is available in the corpus already (single-responsibility-per-node, the same lens that produced the marshal split): for Serving Registry, ask whether folding the catalog into Composition Validator (which already owns the AS-2 admission routine the registry reuses) and Capability List Builder (which already feeds it) breaks black-box seat-contract testing or creates a responsibility collision — if not, a standalone module may be unnecessary weight; if so, that reasoning belongs in the module entry, not just its existence. For Accept Gate, resolve the two-documents disagreement directly: decide whether it is a Module or a segment, and make `system-design.md` and `roadmap.md` agree. This is a documentation/design-review pass, not new spike infrastructure — it reuses the same reasoning move already applied to marshal, at the cost of one more warrant-probe conversation.

**Secondary, lower-priority fix:** the Test Architecture table's "marshal → I/O Envelope → client" row still names a single "marshal test"; split it (or explicitly scope it as a black-box test across the shape/form-gate/emit sub-sequence) to match the Integration Contracts section's already-correct three-node description, in the same document.

**What the practitioner is building on without this grounding:** WP-C8 and WP-D8 will scaffold actual code modules, files, and tests named after Serving Registry and Accept Gate exactly as currently drawn. If the warrant probe concludes a different boundary is cleaner (registry-as-config, gate-as-inline-seats), un-drawing it later means restructuring built code and tests rather than editing two paragraphs of a design document now — the same asymmetry the marshal catch was itself an example of avoiding.

**Not flagged as Grounding Reframe candidates (noted for completeness):**
- Classify/seat/marshal invisibility to the Tool User: consistent with the "endpoint is a model" abstraction, not a gap.
- Serving Registry's parts-vs-shapes vocabulary as a concept (distinct from its module-boundary question, above): traced to ADR-047's operator-tool-grounded provenance, not an unexamined ARCHITECT-time invention.
- The marshal three-node split itself: earned via a real warrant probe with an attached falsifiable fitness claim; sound.
- WP-F8's accumulated scope: consistent with the corpus's own deferral discipline and already consciously tracked (graduate-time note); worth a size estimate before it opens, not a reframe now.
