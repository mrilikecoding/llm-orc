**Cycle 7 question framing (provisional, for RESEARCH entry):**

Three linked architectural questions:

1. **Where does the routing decision live?** Options: (a) framework-driven routing-planner ensemble; (b) `tool_choice` constrained decoding forcing `invoke_ensemble` when a capability match is detected; (c) hybrid where a small classifier decides which mechanism applies per request. The chain-handling commitment is already framework-driven (Spike δ + belief-map); the routing-decision is the open territory.

2. **How are I/O contracts enforced?** Output_schema currently documentary (per Cycle 6 WP-D moderate advisory 1 resolution); claim-extractor's form drift demonstrates documentary schemas don't deliver compliance. Options: (a) schema-as-enforcement with reject-and-retry; (b) tool-call-as-output-format (agents call a `submit_results` tool whose params are the schema); (c) deterministic shaper after the agent (extra dispatch per ensemble with declared schema). Connects to form-vs-content drift collapse question — schema-as-enforcement might address both.

3. **What's the fallback shape for non-capability-matched requests?** Options: (a) general-completion fallback ensemble (preserves infrastructure, adds dispatch overhead); (b) direct LLM completion as residual through the orchestrator-LLM (preserves latency, loses infrastructure); (c) lightweight shim that wraps direct completion with minimal infrastructure (calibration verdict + audit log but no full ensemble overhead). The must-delegate-as-bounded framing means the fallback shape is design territory, not a forced choice.

These three questions are linked: the routing-decision mechanism + the contract-enforcement mechanism + the fallback shape together determine the cycle 7 architecture.

## BUILD-surfaced finding: client-tool-action terminal gap (2026-05-24)

**What surfaced.** At the WP-A BUILD reflection gate, the handler swap made concrete that ADR-027's pipeline emits only `ContentDelta` + `Completion` — the chat-completions surface now **never** returns `finish_reason: "tool_calls"`. The practitioner's north-star is: use OpenCode to invoke the RDD skill, with the actual work (writing files to the *local* machine, running commands) **delegated to ensembles**. The only coherent mechanism for "ensemble does the work, file lands on the local machine":

1. OpenCode sends a request carrying its client tools (`write_file`, `bash`, `read`).
2. The endpoint dispatches the generation to an ensemble (e.g., `code-generator` produces file content).
3. The endpoint returns the deliverable to OpenCode **as a `tool_calls` response** (`write_file(path, content=<ensemble output>)`).
4. OpenCode executes its local tool and writes to the developer's machine.
5. The loop continues (OpenCode returns the tool result; the endpoint decides the next step).

Ensembles run server-side and write to `.llm-orc/agentic-sessions/`, **disjoint** from the developer's filesystem (essay E4.2.1; PLAY note 22). So the deliverable must return through the client's local tools — the endpoint must emit `tool_calls`. ADR-027's pipeline has no stage that does this and is single-turn (no agentic loop).

**Diagnosis — how the cycle missed it (not a clarity failure on the practitioner's part):**

- **C1 conflation.** The cycle framed its problem as "NL-routing-to-ensembles ≈ 0 under tool-rich clients" and treated *both* direct-completion *and* client-tool-delegation as the failure. That conflated (a) the orchestrator *avoiding* ensemble dispatch (the real problem) with (b) client-tool-calls as the *mechanism* for applying work to the local filesystem (a needed capability). ADR-027 fixed (a) by removing the orchestrator-LLM, which killed (b) as a side effect.
- **ADR-022 §line 48 already described the split it dropped:** *"client-declared tools remain correct for actions on the tool-user's filesystem... when the user asks to write code, `code-generator` covers it."* That is the two-step flow (ensemble generates, client tool writes); ADR-027 dropped the second half without replacement.
- **No Cycle 7 spike tested the real-client round-trip.** Cycle 6 γ/λ ran against real clients but tested the *orchestrator-LLM* (the thing being removed); note 22 and λ.5-paid were literally the dispatch→client-write handoff failing. The Cycle 7 pipeline spikes (ζ/ε/ε'/μ/ν) ran against Python harnesses producing text — never through OpenCode. **The architecture being kept was never run through the client it exists to serve.**
- **The scope-of-claim partition covered routing + synthesis, never delivery-to-client.** No essay node named the client-tool-application surface, so no audit (5-round research, 7-round DECIDE) could catch its absence. The DISCOVER "rapid compounding" snapshot warning pointed at this risk.

**What still holds.** AS-9 (structurally-bounded roles produce reliable output) and AS-10 (capability matching from request content alone) are unaffected. ADR-027's architecture is **incomplete, not wrong**: it built the answer-a-question terminal but not the do-work-on-the-client terminal. The fix is *aligned* with ADR-027's philosophy — a framework that *deterministically* wraps an ensemble deliverable into a client tool-call would be more reliable than the orchestrator-LLM that failed at it (note 22).

**WP-A disposition.** WP-A stands as committed (`e538264` + `0a7a822`) — a faithful, green, reversible increment of ADR-027 as written. The `ClientToolCall` chunk plumbing removed in `0a7a822` (handler + `_collect_non_streaming` branches + `_build_completion_body` tool_calls shaping) comes back cleanly when the pipeline gains a client-tool-action stage; the `OrchestratorChunk` union still includes `ClientToolCall`.
