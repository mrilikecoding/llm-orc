# Capability Floor and Observability for Local-First Agentic Orchestration

*2026-04-29*

## Abstract

This essay reports findings from a backward research loop in the agentic-serving scoped cycle. The loop investigated five questions: the empirical capability floor for an orchestrator driving llm-orc's cascading-tool model, what signals reliably distinguish a profile that meets the floor from one that does not, what observability surfaces support diagnostic truthfulness without imposing coordination burden, what default configuration ships with llm-orc to make first-session experience honest about its competence, and how the dual contracts of user-facing outcome and project-facing method are reconciled. The process consisted of nine empirical spikes against a live llm-orc-plus-OpenCode deployment on consumer hardware, augmented by a parallel literature scan covering tool-use evaluation benchmarks, observability patterns in shipped agentic systems, tool-design literature, multi-agent orchestrator patterns, and Beck's prior posts on AI-augmented development. The empirical work produced four substantive conclusions: capability is family-specific rather than parameter-count-driven, prompt steering is the cheapest sufficient intervention at the qwen3:8b tier, the inference platform's per-model tool-calling support varies in ways class-level capability flags cannot reflect, and several failure modes (notably "fast-confabulation" — a model producing confident text claiming tool use that did not occur) are unmeasured in current public benchmarks. Three production changes ship from the cycle: a biased default system prompt, dispatch-result logging that surfaces error reasons actionably, and typed-error promotion for per-model tool-calling rejection. Implications for the four-layer architecture, ADR-011, and the project's default-configuration story are discussed.

## The Starting State

The agentic-serving cycle entered its PLAY phase having built nine work packages of architecture: a four-layer system (serving, runtime, ensemble engine, knowledge graph) with a closed five-tool orchestrator surface, calibration gating for composed ensembles, mandatory result summarization, autonomy policy enforcement, turn-boundary delegation for client-declared tools, and a structural reservation for Plexus integration. The architecture had been validated through unit, integration, and structural fitness criteria tests. PLAY ran a real OpenCode TUI session against the live `llm-orc serve` endpoint to inhabit two stakeholders — a Pure Tool User and an incidental Operator — and observe the system in situated encounter.

PLAY surfaced two next-cycle fundamentals. First, a bilateral observability gap: across a four-turn session, neither the OpenCode response stream nor the `llm-orc serve` operator terminal carried diagnostic signal. The session terminated on budget exhaustion at fifty thousand tokens after approximately six minutes wall-clock with zero useful deliverable. Second, an undefined orchestrator-capable Model Profile capability floor: the default `orchestrator-local` profile (mistral-nemo:12b at the time of PLAY) fabricated a TypeScript project structure for a Python repository, narrated tool execution that did not happen, and described file writes that left no file on disk. The Pure Tool User verdict was that the system would not be likely to use again — a verdict that applied to the default configuration as encountered, not to the architecture as designed.

The research loop's task was to characterize the floor empirically, identify what defaults make the first-session experience honest, and feed the findings back into the cycle as deployable changes.

## Method: Empirical Loop with Headless Tooling

The loop ran nine spikes between 2026-04-27 and 2026-04-29, plus a parallel literature scan. Each spike followed a uniform pattern: configure a specific orchestrator profile in the operator-side `config.yaml`, restart `llm-orc serve` with minimal-but-real instrumentation patches, fire a fixed user request (typically about ensemble availability and a code-review task on the project README) through OpenCode, and observe four surfaces — the OpenCode TUI rendering, the SSE traffic on the wire, the server-side log, and the post-session state.

A small but significant tooling shift occurred mid-loop. The first three spikes ran interactively, with the practitioner driving OpenCode by hand and the agent observing logs. After verifying that `opencode run` is a fully headless one-shot invocation with `--format json` for parseable output, subsequent spikes ran as background commands. This shift made A/B comparison feasible and dramatically cut the per-spike cost. It also meant findings were reproducible: each spike's exact invocation is recorded in the research log, and any spike could be re-run unchanged.

The instrumentation patches applied during the spike phase are themselves part of the finding. The default `llm-orc serve` operator-side log surface is structurally empty: `src/llm_orc/web/server.py` declares one logger that fires only on unhandled exceptions, and the entire `src/llm_orc/agentic/` tree has no logger references at all. Three logger lines plus three diagnostic print statements were sufficient to surface the orchestrator's per-turn decisions in real time. The cost-to-value ratio of operator-side instrumentation is highly favorable; the absence in the default code path is a structural omission, not a configuration choice.

A literature scan ran in parallel as an isolated subagent task across six gaps: tool-use capability evaluation benchmarks, observability patterns in shipped agentic systems, tool-design literature on schema verbosity and surface composition, multi-agent orchestrator patterns, Beck's prior Genie posts, and recent Ollama-available small models. The scan returned partial results on the first dispatch (interrupted by usage limits), then completed fully on a second dispatch with expanded scope. Citation provenance is recorded in the lit-review artifact.

Two scope conditions on the empirical findings deserve naming explicitly. First, the spike trial counts are small. CAP-7 was n=2 (run-and-rerun against qwen3.5:9b); every other spike was n=1. The cycle's findings about specific failure modes are characterizations from small samples; their stability across additional trials would benefit from re-runs that the cycle did not perform. Second, every spike fired through OpenCode as the client. The 16-tool surface that the orchestrator navigated — five internal plus eleven OpenCode-declared client tools — is partly an OpenCode property. Other agentic clients (Cline, Roo Code, Continue, etc.) declare different tool sets, may use different system prompts on top of the orchestrator's, and may produce different floors. The cycle's findings about the specific competing-tools dynamic that caused CAP-1's failure should be read with that scope in mind.

## The Capability Gradient Observed

The spike battery exercised five orchestrator models against the same task class. The aggregate result is summarized below; full per-spike trajectories are recorded in the research log.

The qwen3:14b orchestrator drove the full internal cascade — `list_ensembles` followed by `invoke_ensemble` followed by mid-flight client-tool delegation — across seven turns spanning approximately twenty-two minutes of wall-clock time. The architecture's mandatory-summarization path fired twice with `summarization_failed` errors that the orchestrator handled gracefully via text fallback. Path-accuracy on the produced response was five real of seven cited; two paths were fabricated as plausible-sounding but non-existent specific files inside real directories.

The qwen3:8b orchestrator with the default neutral system prompt did not engage the internal cascade. When asked about ensembles available in the instance, the model selected `skill` — a client-declared tool whose name semantically resembled "what's available in this system" — and produced empty content after the skill returned. The session closed with a silent-giveup pattern: zero useful output to the user.

The same qwen3:8b orchestrator with a biased system prompt — explicitly directing the model to call `list_ensembles` first for capability queries and warning against confusing similarly-named client tools — drove the cascade through `list_ensembles` and `invoke_ensemble` retries, eventually producing a graceful text fallback when summarization failed. After the summarizer failure was diagnosed and unblocked, the same setup produced the cycle's first end-to-end successful agentic response: `list_ensembles` to discover, `invoke_ensemble` to execute the code-review ensemble on the README, and a substantive synthesis based on the actual ensemble output.

The mistral-nemo:12b orchestrator with the same biased prompt did not engage the cascade. Across two trials it produced confident-sounding text claiming `list_ensembles` had been called and listing one fabricated ensemble (`ai-detect`) with an invented description and an invocation syntax (`/ai-detect README.md`) that does not match llm-orc's tool surface at all. No tool calls were emitted. The biased prompt's directives could not override the model's tendency toward confabulation.

The qwen3.5:9b orchestrator — newer and larger than qwen3:8b — produced a different failure mode. It correctly called `list_ensembles` once but stopped after the result returned, without invoking any ensemble or producing a useful synthesis. The pattern reproduced across two trials. Three candidate explanations are consistent with the observation: an Ollama packaging bug in the qwen3.5 chat template (QwenLM/Qwen3 issue 1831, "fix chat template for Qwen 3.5 — fixes tool calling crash, parallel calls, thinking bleed"), output-length tuning differences in qwen3.5 that bias toward earlier stopping, or generation-parameter mismatches against Qwen's recommended values (notably `presence_penalty=2.0`). The chat-template hypothesis has the strongest symptom-cause correspondence but the cycle did not run the experiments needed to distinguish it from the other two.

The deepseek-r1:8b orchestrator could not be tested for cascade behavior because Ollama's `/v1/chat/completions` endpoint returns 400 with the message "registry.ollama.ai/library/deepseek-r1:8b does not support tools" on every request. The model has no tool-calling capability exposed via Ollama. The orchestrator runtime's session-start validation passed (because `supports_tool_calling` on the OpenAICompatibleModel class is True for any Ollama-served model), and the failure manifested only at the first generation call.

## Findings

### Across the families tested, tool-use training quality dominates parameter count

The evidence from these spikes places the empirical floor for full internal-cascade engagement on consumer hardware in this deployment between qwen3:8b (above the floor with biased prompt) and mistral-nemo:12b (below the floor regardless of prompt). The qwen3:8b model — smaller than mistral-nemo:12b in parameter count — engaged the cascade and produced useful output; the mistral-nemo:12b model fabricated content rather than calling tools, with the same biased prompt that succeeded for qwen3:8b. The qwen3.5:9b model, newer and larger than qwen3:8b, sits above the floor for first-tool selection but fails on continuation. Across these families and parameter counts, parameter count and release recency are insufficient signals for orchestrator suitability. The cycle did not isolate "family" as a single variable from training recency, architecture, and tool-calling training volume; the conclusion is appropriately scoped to the families tested rather than a general claim about model lineage.

External corroboration comes from a published practical evaluation. An independent Docker Engineering Blog evaluation across 3,570 test cases reported F1 scores in the small-model class — Qwen3-14B at 0.971, Qwen3-8B at 0.933, with specialist tool-calling models xLAM-8B at 0.570 and Watt-tool-8B at 0.484. This evaluation is methodologically separate from the Berkeley Function-Calling Leaderboard's own AST-based scoring (Patil et al., ICML 2025) but reaches a directionally consistent conclusion: the qwen3 family substantially outperforms specialist tool-calling models in similar parameter classes. The cycle's qwen3:8b-clears-the-floor finding has external benchmark validation.

### Prompt steering is the cheapest sufficient intervention for this task class at this tier

The CAP-1 to CAP-3 contrast is decisive on cascade engagement: the same qwen3:8b model on the same hardware against the same user request fails one way and succeeds another, and the only variable that changed is the orchestrator's system prompt. The biased prompt directs the model to call `list_ensembles` first for any question about ensembles, available capabilities, or what the system can do, and warns explicitly against picking client-declared tools whose names resemble capability queries (the failure mode the model exhibits without the bias). The "sufficiency" claim is bounded — the bias was crafted against this task class (capability-query routing on the OpenCode 16-tool surface), and other task classes may need different bias content.

Two implications follow. First, the default `DEFAULT_ORCHESTRATOR_SYSTEM_PROMPT` shipped with llm-orc as of the start of the cycle was insufficient: it listed the five tools and explained Option C and the retry convention but did not bias the model toward internal tools when the question semantics warranted. The biased version is now the default in the codebase. Second, structural composition of the orchestrator role into a router-plus-executor ensemble — a candidate intervention considered after CAP-1 and prototyped in CAP-2 — is not empirically warranted. The CAP-2 router-executor pattern works as designed but adds approximately 2.5x latency overhead for an outcome that the cheaper prompt-steering intervention also produces. ADR-011's commitment to a single-Model-Profile orchestrator holds at this capability tier.

The literature on multi-agent orchestrator patterns is consistent with this finding. LangGraph documentation reports supervisor-routing overhead in excess of 30%; AutoGen's GroupChat costs one full LLM call per agent turn. The recommendation across documented frameworks is to apply structural composition only after cheaper interventions have been exhausted. The cycle's empirical conclusion matches the literature's prescriptive guidance.

### Per-model tool-calling support is platform-specific and class-level checks are insufficient

The deepseek-r1:8b case identified a real validation gap. The session-start check at `_default_orchestrator_llm_loader` correctly verified that the model interface class advertises tool-calling support, but Ollama's per-model tool-calling capability metadata varies, and a class that supports tool-calling in general can wrap a model whose specific weights have no tool-calling adapter. The first turn fails at the provider with a clear-text error; the orchestrator's session start cannot have caught the failure with the existing class-level check.

The smaller fix shipped from this cycle promotes the provider's "does not support tools" 400 response into a typed `ToolCallingNotSupportedError` — the same error type the session-start validator raises for class-level mismatches. Downstream code paths (the SSE stream's error formatting, OpenCode's retry logic) can now treat both surfaces as a configuration error rather than a transient transport fault. A larger pre-flight fix — querying Ollama's `/api/show` endpoint at session start to read the configured model's capability metadata — is not in this cycle's scope but is a clear follow-up.

### Confabulation as a failure mode is unmeasured in tool-calling benchmarks

The mistral-nemo:12b run produced confident-sounding text claiming a tool call had occurred when no dispatch event fired. The Berkeley Function-Calling Leaderboard's methodology evaluates the structural correctness of tool calls that are emitted; it does not evaluate the case where a model claims to have called a tool without actually emitting the call. ToolBench, ToolEval, and tau-bench similarly measure structural and outcome dimensions of tool calls that occur, not the presence of tool-call claims absent dispatch. The "fast-confabulation" failure mode — the most insidious of the failure modes the cycle observed because it appears authoritative on superficial inspection — sits in the tool-calling-benchmark literature's blind spot. Whether broader hallucination benchmarks capture the same phenomenon under different terminology is unexplored by this cycle.

The VILA-Lab Claude Code architecture paper (Liu et al., arXiv:2604.14228, 2026) explicitly names "Silent Failure and the Observability-Evaluation Gap" as an open research direction and notes that agents tend to respond by confidently praising work even when quality is mediocre. The framing aligns with the cycle's empirical observation but the operationalization (a flag for tool-call-claims-without-dispatch) is not implemented in any system surveyed. This is a class of operator-side observability work where the cycle's empirical findings are ahead of the published patterns.

### Latency is the binding constraint at consumer hardware

The capability findings are necessary but not sufficient for product viability. Across the spike battery, full-cascade wall-clock time was approximately twenty-two minutes for qwen3:14b at the S0 trial and approximately six minutes for qwen3:8b at the CAP-3b trial. Per-turn latency for the orchestrator on consumer hardware ranged from sixty to one hundred seventy seconds against fifteen-to-twenty-three-thousand-token contexts; the tool schema overhead alone (five internal plus eleven OpenCode-declared client tools across each turn) accounted for roughly fifteen thousand tokens of input per turn. Cascade depth compounds this: a hypothetical session that calls `list_ensembles`, then `get_ensemble`, then `invoke_ensemble`, then synthesizes — four turns plus client-tool delegations — could exceed ten minutes on this hardware before producing the first useful sentence to the user.

The implication for the user contract is direct: even when the cascade engages correctly and produces a useful response, the latency at consumer hardware is unworkable for interactive coding workflows. A user who waits six minutes for a response receives a degraded experience even if the eventual output is correct. The architecture is honest about *what* it produces but not about *when*. Production deployments at this hardware tier face a real tradeoff: accept the latency, change the hardware, change the model selection, or change the session pacing (e.g., by making cascade engagement explicit and asynchronous rather than implicit and inline). None of these are addressable through prompt engineering alone.

### A failure-mode taxonomy emerged

Across the spike battery seven distinct failure modes appeared. Slow-useful describes the qwen3:14b S0 and qwen3:8b CAP-3 outcomes — the cascade engages, struggles through summarization or other intermediate failures, and eventually produces a useful response. Fast-confabulation describes mistral-nemo:12b's behavior in PLAY and CAP-5 — the model produces confident fabricated content quickly without engaging tools. Fast-giveup describes qwen3:8b's CAP-1 behavior with the unbiased prompt — the model recognizes it is not equipped to handle the request and produces empty content rather than fabricating. Summarization-broken describes the DIAG-1 root cause — the cascade engages but is gated by a missing model dependency in the operator's local Ollama. Premature-stop describes qwen3.5:9b's CAP-7 behavior — the model engages the first tool correctly but stops mid-cascade. Unsupported describes deepseek-r1:8b's CAP-8 case — the provider rejects tool calls at the protocol level. Argument-confabulation describes a behavior observed during CAP-3 under summarization-retry stress, within the validated qwen3:8b configuration: when `invoke_ensemble` returns the same `summarization_failed` error across multiple retries, the model emits structurally-correct subsequent calls with fabricated argument keys (`skip_summarizer`, `force_output`) that are not part of the tool's real schema. Tool Dispatch silently ignores unrecognized fields, so the dispatch proceeds, but the orchestrator believes it has invoked behavior that does not exist. This is a sub-form of confabulation that exists *within* an otherwise-correct cascade — observed under stress, not as baseline behavior — and is therefore present in the configuration the essay validates as a known limitation that future work should address. See the convergence-conditions finding that follows for how the validated configuration manages such failure modes when they appear.

Each failure mode has different observability requirements. Slow-useful needs latency communication; fast-confabulation needs dispatch-event verification; fast-giveup is honest but unhelpful; summarization-broken needs error-reason surfacing; premature-stop needs cascade-completion tracking; unsupported needs pre-flight or first-turn typed-error handling; argument-confabulation needs schema-conformance enforcement at dispatch time. Lumping them under a single "errors" category obscures the diagnostic information operators need.

### The dual contracts converge when three configuration conditions are satisfied

The dual-contract framing — a user-facing outcome contract (the user gets a useful result, or knows clearly why not) and a project-facing method contract (the hypothesis that orchestration with non-frontier models can deliver good results) — was hypothesized to require seam-navigation between divergent design choices. The empirical work shows that the contracts converge when three conditions are simultaneously satisfied: the orchestrator profile is qwen3:8b (or comparable family-tested model), the system prompt biases toward internal tools per the CAP-3 directive, AND the summarizer model dependency (`qwen3:0.6b` per the project-tier config in this cycle) is locally available. With all three, the user gets a useful response and the orchestration cascade engages through internal tools. With any one missing, the cycle observed a different failure mode at a different point in the pipeline. The seam was navigated through configuration, not through architectural reorganization.

The contracts diverge at the latency boundary, as the preceding latency finding documents. The convergence above is conditioned on the configuration; the divergence is conditioned on the hardware. Both are real and persistent.

## What Ships

Three production changes ship from this cycle, recorded as commits on the `agentic-serving` branch. The first updates `DEFAULT_ORCHESTRATOR_SYSTEM_PROMPT` in `src/llm_orc/agentic/orchestrator_config.py` to the biased version that empirically clears the floor at qwen3:8b. A new test pins the bias directives — a "first" cue near `list_ensembles`, and a negative directive against confusing client tools for capability queries — so future revisions that drop the bias surface as visible regressions rather than silent re-emergences of the CAP-1 failure mode. The second adds INFO-level result logging to `OrchestratorToolDispatch.dispatch` in `src/llm_orc/agentic/orchestrator_tool_dispatch.py`, with success paths logging name and kind only and error paths logging the full reason field. Two new tests pin both surfaces. The third detects the specific Ollama 400 "does not support tools" response in `src/llm_orc/models/openai_compat.py` and promotes it to a typed `ToolCallingNotSupportedError`. A new test pins the type promotion against the documented provider error message.

## Open Questions

Several questions remain that the cycle did not answer and that future work could address. Whether qwen3.5:9b's premature-stop is genuinely a chat-template bug (and whether updating Ollama's packaging fixes it) is empirically unresolved; the symptom-cause match against GitHub issue 1831 is strong but not verified. Whether tool-tuned specialist models in the same parameter class as qwen3:8b (such as Salesforce xLAM variants) cross the floor is unmeasured in this cycle and the BFCL data suggests they may not — but the cycle's task class is narrower than BFCL's. Whether real multi-turn workflows (a full RDD phase, a coding session, a debugging interaction) reproduce the single-spike findings is the most important unanswered question for the project's user contract; every spike in this cycle was a single ask, single response, and the failure modes that matter for sustained work may differ. Whether confabulation detection — a structural check that an asserted tool call appears in the dispatch log — is implementable with acceptable false-positive characteristics is an open design question that the literature does not yet answer.

The lit review's spike-actionable model recommendations included candidates not tested in this cycle (qwen3.5:4b, tool-tuned 8B-class models). Pulling and testing them would extend the capability gradient map but is unlikely to reshape the cycle's central findings unless one of them substantively outperforms qwen3:8b on cascade fidelity at lower latency.

## Implications for the Architecture

The four-layer architecture (ADR-002), the closed five-tool surface (ADR-003), the mandatory summarization path (ADR-004), and the orchestrator-as-Model-Profile commitment (ADR-011) are empirically exercised by the cycle's work. CAP-3b's end-to-end successful cascade fired the Serving Layer, Session Registry, Orchestrator Runtime, Tool Dispatch, Ensemble Engine, and Result Summarizer Harness. ADR-007's Calibration Gate and the Composition Validator (WP-G) were structurally present but not entered during the cycle's task class — `compose_ensemble` was not called and no composed ensembles were invoked, so the calibration path remained dormant. Their architectural integrity is verified by the existing fitness-criteria test suite but not by this cycle's empirical work; future cycles that exercise composition would close that gap.

ADR-011's commitment to a single-Model-Profile orchestrator was tentatively flagged as an unexamined commitment at the start of the cycle but is now empirically supported for this task class at the qwen3:8b tier: prompt steering achieves what structural composition would have, at lower complexity and lower latency.

Two amendments are warranted at follow-up. The first is the per-model tool-calling validation gap that CAP-8 surfaced: the session-start check should validate the specific configured model's tool-calling capability, not just the wrapper class's. The second is the operator observability surface: the structural absence of loggers in `src/llm_orc/agentic/` is a design omission that should be remediated, with a default INFO level that surfaces ReAct loop turns, tool dispatches, and result kinds. The cycle's instrumentation patches in the spike worktree are a reference implementation; the production version is a bounded follow-up.

## Conclusion

The cycle's central question — what does it take for a non-frontier local model to drive llm-orc's cascading-tool model effectively, and what defaults make that work — has empirical answers. A qwen3:8b orchestrator with a biased system prompt and the summarizer model dependency satisfied delivers useful agentic responses end-to-end on consumer hardware, at latencies that are workable for asynchronous tasks where multi-minute response times are acceptable (batch summarization, scheduled analysis, background research) but unworkable for interactive coding workflows. The validated model selection is narrower than the model list might suggest: qwen3:8b is in, qwen3.5:9b is out (premature stop after first tool call, three candidate causes unresolved), and deepseek-r1:8b is out at the platform level (no tool-calling support exposed via Ollama). The default configuration shipped before the cycle was insufficient at multiple points: the system prompt did not bias toward internal tools, the operator log surfaced no diagnostic signal, and a runtime model dependency was assumed to be locally available. Three small production changes shipped from this cycle close the most consequential of those gaps. The remaining gaps, including the latency boundary, the multi-turn workflow question, the argument-confabulation behavior in the validated configuration, and confabulation detection more broadly, are sized correctly for follow-up cycles rather than for further extension of this one.

The dual-contract framing — Beck's outcome stance applied to the user, the orchestration-with-non-frontier-models hypothesis applied to the project — held through the empirical work. The contracts converge under correct configuration. They diverge at latency, which a production deployment must address through hardware, model selection, or session pacing, not through prompt engineering. The framing remains a useful navigation tool for follow-up work where the seam may bind differently.
