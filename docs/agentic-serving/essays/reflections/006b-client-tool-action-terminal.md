# Reflection — Cycle 7 BUILD → RESEARCH loop-back: client-tool-action terminal

*2026-05-30. Corresponds to essay-outline-006 Amendment B and research-log archive `006b-client-tool-action-terminal.md` (pending phase-exit archival per ADR-036).*

A short re-entry. The cycle had reached BUILD WP-A and a reflection-gate surfaced that the dispatch pipeline emits text only and is single-turn — it cannot give a tool-driven client's local filesystem the work the ensembles produced. The loop-back named the gap as "missing terminal." That was the symptom. The deeper shape took most of the re-entry to find.

**The constraint-removal step earned its keep, twice.** Once at entry: bracketing ADR-027 surfaced the practitioner's co-location point ("`llm-orc serve` on the same machine as the client → local write should be very possible"), which split the loop-back finding's fused premise into delivery (trivial under co-location) and parity (which co-location does *not* resolve). That single split corrected the original justification (the terminal is required because of the client's *execution model*, not filesystem *geography*) and ended up doing more work than the methods-reviewer dispatch that followed. The reviewer was independently valuable — it caught that the sharpened question still embedded `tool_calls` as the answer, and reformulating to test necessity-first (direct-write Phase A) is the reason this cycle has a finding about *why* the terminal is needed, not just whether it works.

**Architectural Isolation caught a verification error in the in-context agent.** F-ρ.2 — the claim that `agentic-tier-cheap-general` was undefined as a model profile — was an in-context conclusion drawn from `llm-orc list-profiles` (which enumerates only `config.yaml`-embedded profiles) without checking the `.llm-orc/profiles/` directory. The citation auditor, running fresh on the amendment, surfaced the error against the actual file. The finding was retracted with a PROVENANCE CORRECTION; the production `code-generator` resolves its cheap tier on the free tier after all. This is the methodology working as designed (ADR-058 / ADR-067): a fresh context treats prior output as external input and reactivates correction the in-context agent could not generate alone. Worth recording because it's a clean example, not because it was unusual.

**The pattern the practitioner caught at gate-time is the one worth naming.** After the spikes were folded into the amendment and the audits ran clean, the agent posed the reflection-time challenge with a framing — "loop-driver as orchestrator-LLM returning in a grounded seat" — that was presented as discussion-shaping but was actually unflagged extrapolation. The spikes had not tested whether anything about the orchestrator-LLM role in `llm-orc`-architecture terms returns; σ used a generic local model in OpenCode's model slot. The practitioner's reflective pushback ("I'm not sure — what do the spikes indicate?") rather than agreement was what corrected the framing. The C8 sub-tree now records the seat-filler candidates explicitly as *candidates, not findings* and the grounded-loop story as a *working inference, NOT a spike finding* — and the post-gate refinement (W8.6) made that distinction load-bearing across all four sections of the outline. The pattern is the one the susceptibility framework expects (content selection at synthesis moments without competing frames independently surfaced) but it was not caught by any audit — it was caught by practitioner reflection during gate dialogue. The framing audits at R1 surfaced related concerns (alternatives A/B/C) but did not name the specific extrapolation. The reflection that did catch it was the right kind: a refusal to grant the framing without checking what the evidence actually supports.

**The reframe that matters going into DISCOVER.** The original loop-back finding read "ADR-027 is incomplete (missing a terminal)." After σ, the more accurate reading is "ADR-027 is structurally under-specified for the multi-turn agentic behavior the north-star requires, not merely missing a stage." Layer A — a per-turn agentic-loop-driver — is a *role* no current ADR-027 component holds; the terminal (emit a tool_call) is one mechanism the loop-driver uses, not the whole gap. This is the reframe the practitioner pushed for (you said "an agentic loop driver seems important"); DISCOVER update mode should make the "delegate work, apply locally" loop explicit in the Tool User and Skill Orchestration User mental models, and the assumption-inversion on "endpoint-returns-text is sufficient" should land as a settled rejection given C8's evidence.

**A note on cost and discipline.** Four spikes (π, ρ, σ.1, σ.2) ran in-loop, $0 across all of them, against a real OpenCode session via a disposable scratch server pointing at local Ollama. The free-tier preference made the spike scope tractable; the spike-artifact-retention directive means the scratch directory stays as a corpus reference rather than being deleted. The validation-spike investment was clearly worth it: π's Phase-A-first ordering converted a sufficiency test into a necessity test; ρ separated delegation-decision from terminal mechanism; σ revealed layer A; the integrated σ.2 confirmed the north-star pattern composes at the mechanism level. Without them, the amendment would have committed to "add a tool_calls stage to ADR-027" — true but narrow, and missing the structural reframe.
