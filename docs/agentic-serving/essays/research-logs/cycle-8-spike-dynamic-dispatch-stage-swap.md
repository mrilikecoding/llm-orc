# Spike (Cycle 8 DECIDE-entry grounding) — Dynamic Dispatch + Swappable-Stage Lifecycle

**Date:** 2026-07-01
**Cycle:** 8 (agentic-serving; the declarative-ensemble collapse)
**Phase:** DECIDE — grounds the target-architecture ADR before it is drafted
**Cost:** $0.00 target (local Ollama; all-8b / 32GB-sized). Ask-before-spend on any paid arm.
**Retention:** retained until corpus close per the spike-artifact-retention discipline.
**Methods review:** intentionally skipped — this is a feasibility/mechanism spike with pre-stated falsification criteria (below), not a bias-prone measurement spike. Recorded per the practitioner's confirmation 2026-07-01.

---

## Question

Can a minimal fixed declarative lifecycle skeleton serve a real OpenCode turn with **full model-parity capability**, where a stage's seat is a **dynamically-dispatched** ensemble that can be swapped for a structurally different strategy at **zero skeleton change**?

## Setup (what is given vs. what is built)

- **Given — reuse, do NOT re-prove (a2, Cycle-7 PLAY):** real `opencode run` serving the ensemble transparently is already validated (`scratch/spike-omega-serve/serve_ensemble.py`: SSE, tool-less aux-call short-circuit, 8b config for the 32GB rig). This spike builds on that foundation and does not re-spike the serving transport.
- **Build — the gap:** **dynamic dispatch** as an L0 engine primitive (Phase 4 of the control-flow arc) — a node whose `ensemble:` target is resolved at runtime from a key the prior stage produces. Built TDD (Red→Green→Refactor), shippable, matching the guard/loop primitive pattern; NOT throwaway scaffolding.
- **Validate — the architecture:** the swappable-stage lifecycle (classify/decider seat → capability seat → deterministic marshal), with the capability seat filled via dynamic dispatch.

## Pre-registered success criteria

1. **Dynamic dispatch works:** a node resolves its `ensemble:` target at runtime from the prior stage's output and executes it, through the real `EnsembleExecutor`.
2. **Swap-ability (the pass/fail):** the capability seat is filled by seat strategy **A** (e.g., a single capable local model), then swapped for a structurally different strategy **B** (e.g., a small-model verify-heavy / multi-agent ensemble), with **zero edit to the skeleton**; both flow through the serve end to end.
3. **Generality (not the build path again):** at least one **non-build turn** (explain-a-file or fix-in-existing-file) routes and completes through the same skeleton, alongside a build turn. Ω-P3 only ever exercised build-new-files; this is the full-capability probe.

## Falsification (what would say the model is wrong)

- The swap requires skeleton edits → seats are not truly interchangeable; the inter-stage envelope (ADR-024) is insufficient as the seam.
- Three stages cannot handle a non-build turn → the minimal stage-set is wrong (a fourth stage, e.g. a grounding/gate, is forced — the §6.2b question surfacing here).
- Runtime seat resolution cannot be an engine primitive without a heavier adapter → the dynamic-dispatch shape needs rethinking (would reopen an AS-11 tension).

## Method

- Build the dynamic-dispatch primitive TDD in the engine (schema + controller/runner + wiring + tests), mirroring guard (`when:`) and bounded loop (`loop:`).
- Assemble the classify → seat → marshal skeleton as one declarative ensemble on the shipped primitives + dynamic dispatch.
- Reuse the Ω-serve harness for the serving side; drive with real `opencode run` at the earliest runnable point (per validate-against-real-client).
- Seat strategies A and B differ structurally so the swap is a real interchangeability test, not a config tweak of one strategy.

## Artifacts

- Spike code + configs: `scratch/spike-dynamic-dispatch/` (+ `.llm-orc/ensembles/`, `.llm-orc/scripts/` as needed).
- Engine primitive: shipped to `src/llm_orc/` via TDD commits (structural/behavioral separated), like guard/loop.
- Findings appended to this log; feed the target-architecture ADR.

**Live-model drive result (2026-07-01, real qwen3:8b, $0-local).** All three success criteria met with real models (`scratch/spike-dynamic-dispatch/drive.log`):
- **#1 dynamic dispatch:** the skeleton routed and produced through real model seats.

**So the spike's central question resolves YES:** a minimal fixed declarative lifecycle skeleton serves real turns with full model-parity capability (build and non-build), where the seat is dynamically dispatched and swappable at zero skeleton change. The falsification conditions did not fire: the swap needed no skeleton edits (the `results_dict` envelope is a sufficient seam), three stages handled a non-build turn (no forced fourth/grounding stage), and runtime seat resolution is an engine primitive with no heavier adapter (A2).

**But the latency wall is real and biting (Tension 23 / Q3 §6.3), which is the honest headline.** Per-turn wall-clock: solo build **63.5s** (cold load + 1 inference), verified build **81.8s** (2 inferences, warm), explain **13.3s** (short output). Warm generation is roughly 13–40s per inference depending on output length; thinking-mode qwen3:8b is not disabled. A ~60–80s build turn is not a daily-driver on the 32GB rig. Two grounded consequences for DECIDE: (a) F2's "decider/seat cost is a dial" is now measured, a verify-heavy multi-agent seat costs ~2x a single-model seat, so on this rig interactive turns want cheap single-model seats with thinking-mode control and verify-heavy seats reserved for non-interactive use; (b) the obvious untested lever is disabling qwen3 thinking mode and re-measuring (the "thinking-mode + sampling control" Q3 named). Speed is now the load-bearing open constraint, not the architecture.

**Thinking-mode-off re-drive (2026-07-01, qwen3 `/no_think` seat variants) — no latency win, and that itself is the finding.** Thinking-off vs thinking-on wall-clock: solo build **135.2s** vs 63.5s (slower, but this is the cold-load first turn and is noisy), verified build **79.7s** vs 81.8s (flat), explain **12.0s** vs 13.3s (flat). Warm build inference stayed ~40s either way; the `/no_think` soft switch did not move latency. Outputs stayed correct (both wrote valid `is_prime.py`; the fast verified seat even emitted a 6k±1 wheel form). Two candidate explanations, not yet distinguished: (a) the `/no_think` prompt soft switch is not taking effect through llm-orc's Ollama chat path (llm-orc does not pass Ollama's native `think` param — `ollama.py` builds only model/messages/options/format), so thinking may still be running; or (b) thinking tokens are not the bottleneck and the ~40s/inference is raw qwen3:8b throughput on the 32GB rig. A direct Ollama probe (default vs `think:false` param vs `/no_think` prompt, comparing `eval_count`/`eval_duration`) is the next step to distinguish, and it will tell whether wiring the native `think` param into `ollama.py` is a real speed lever. Speed remains the open constraint; the thinking-mode lever as applied here did not help.

**Direct Ollama probe (2026-07-01) — decisive: thinking is the entire cost, and llm-orc sends the wrong lever.** Same coding prompt, three conditions, reading actual `eval_count` (`scratch/spike-dynamic-dispatch/probe.log`):
- default: **101.7s**, 2975 output tokens, 29.4 tok/s, `thinking` field present.
- `think:false` param (Ollama native): **3.1s**, 85 tokens, 30.8 tok/s, no `thinking`. **33x faster, clean code output.**
- `/no_think` prompt (what the seat variants used): 59.6s, 1778 tokens, `thinking` still present. The soft switch is NOT honored by qwen3:8b through Ollama's chat API.

Conclusions: (1) throughput is healthy (~30 tok/s), so the rig is not the problem and model-sizing is not the lever; (2) ~97% of default latency is reasoning tokens (2975 generated for an 85-token function); (3) the real, shippable lever is Ollama's native top-level `think` field, which `ollama.py` does not send (it builds only model/messages/options/format). Wiring `think` through turns a ~101s build turn into ~3s. This is per-request, so it becomes a per-agent / per-seat control (thinking off for interactive seats, on for hard problems), exactly the "thinking-mode control" Q3/§6.3 named, and another swappable seat dimension. **Speed is not an architecture problem and not a wall; it is one un-wired param.** Candidate fix (separate from the dispatch primitive): expose `think` on the LLM agent (via `options` or a first-class field) and lift it into the Ollama chat call; TDD-able small change to `ollama.py` + the config→model path.

**Design findings (2026-07-01 session, selection-policy + contract grounding).** These refine the pre-registered plan against the real engine. The primitive's shape and the seat architecture moved beyond the pre-reg; recorded here to retarget the build and feed the target-architecture ADR.

**F1 (seam / falsification #1, resolved).** A runner receives only the enhanced input string, which for non-script nodes is prose (`dependency_resolver._compute_agent_input`), not a parseable dict. So the pre-registered "runner resolves the `${ref}` against parsed upstream input" does not hold inside the runner. The structured upstream data lives one layer up in `results_dict`, where the guard primitive already resolves `${dep.field}` (`guard_evaluator.py:45`). Resolution therefore belongs at the phase layer, sibling to `_partition_by_guard`. Architectural read: guard decides IF a node runs, dispatch decides WHICH ensemble it runs, and both are deterministic phase-layer resolutions of `${dep.field}` against the common envelope (ADR-024). Dispatch composes guard's resolution with ensemble's child execution. Recommended primitive shape (A2): a phase-layer resolver sets a runtime-only resolved-target field (fan-out precedent: `BaseAgentConfig` excluded runtime fields) that a thin `DynamicDispatchRunner` reads and child-executes. Dispatch refs are dep-qualified (`${classify.target}`), same grammar as guard. Status: recommended, one open confirm before the runner TDD resumes.

**F2 (architecture: swappable seats behind contracts; the decider is a seat).** The fixed skeleton is a sequence of typed seats and their contracts; dispatch fills each seat with any contract-satisfying ensemble. The routing decision is not privileged: the decider is itself a seat, fillable by one frontier call or a multi-stage deliberation ensemble (input-decomposer, component formatter, prose analyst, voting bloc of micro-ensembles, confidence weigher, judge), same contract, different internals. Consequence: the decider's cost is a dial (one cheap classifier up to a deliberation bloc), and it is where per-turn latency (Tension 23) and the ensemble depth guard (limit 5) get spent, so the decider seat most wants a cheap default. Candidate invariant: the seat sequence plus contracts are the fixed point; everything filling them is fluid. Open: is the seat sequence itself swappable, or is it the floor.

**F3 (the seat contract is existing core capability, not new machinery).** `core/validation/` (built, per the practitioner, to enable exactly this dynamic composition) provides `ValidationConfig` with five layers (structural, schema, behavioral Python assertions, quantitative, semantic) and a pass/fail `ValidationResult`, wired inline (`ensemble_execution.py:868`, `_run_validation`) and as CLI (`validate_ensemble`, `validate_ensemble_category`, `validate_all_ensembles`). It survives the `agentic/` removal. Three constraints for use as a seat contract:
  1. Seat-owned, not candidate-owned. As wired the `validation:` block lives on the ensemble and validates it against its own assertions (the §6.2b independence trap relocated to the registry). But `ValidationEvaluator.evaluate(name, results, validation_config)` takes the config as a separate argument, so seat-owned is a convention change, not a rebuild: attach the contract to the seat, run the candidate's results against it.
  2. Black-box only. The structural layer names internal agents (`required_agents`), which is white-box; asserting it would couple a seat to candidate internals and break interchangeability. Seat contracts use the I/O-facing layers (schema, behavioral-on-output, quantitative).
  3. Deterministic signal preferred. The semantic layer is LLM-as-judge (gameable gate plus per-check cost); seat eligibility rests on the deterministic layers, semantic stays optional and soft.
  `validate_ensemble_category(category)` is a natural attachment point for the topaz-keyed registry. This grounds Q2 (§6.2b): the contract assertions are the independent correctness signal the loop's accept decision lacked.

**F4 (within-category routing splits deterministic vs. semantic).** Language specialization (python / js / rust specialists vs. a larger generalist) is a real within-category routing need, but much of the signal is deterministic (file extension, project manifest, fenced language tag) and need not cost an inference hop; semantic routing (reads intent) needs a decider. Binding-time fork (open): load-time operator curation (deterministic, zero runtime cost, swap by config), turn-time classification (per-turn inference), or hybrid (per-turn category plus load-time candidate default). Operator control is deployment-time curation, distinct from runtime end-user disambiguation (a separate, available, non-north-star lever).

**Spike test matrix (produced this session).** Two structurally-different / same-contract swap probes for the swap-ability criterion: (a) decider seat, single frontier call vs. multi-stage deliberation bloc; (b) capability seat, language-specialist small model vs. larger generalist. The non-build turn (explain or fix) exercises the decider routing to a non-code capability seat.
