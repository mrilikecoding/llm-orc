# Three-Arm Architecture Comparisons: What Cycle 3's Spikes Established and What They Did Not

*2026-05-04 (retrospective synthesis written after Cycle 3's Mode B closure)*

## Abstract

This essay is a retrospective synthesis of Cycle 3 of the agentic-serving scoped corpus, written after the cycle's Mode B closure on 2026-05-01. Cycle 3 ran a combined literature review and four spikes (Spike A, Spike B, Spike C, Spike D pilot) probing the cycle's central question — sharpened mid-cycle by the practitioner from "does orchestration shape matter?" to "does cheap-orchestrator + orchestration compete with a more expensive frontier model?" The cycle produced four substantive empirical findings (cross-tier-uncorrelated-errors on documentation review; multi-turn reliability ceilings do not manifest at the cycle's tested complexity; cheap-orchestrator + ensemble outperforms frontier-bare on cross-file verification 3-of-3 vs 0-of-2; multi-ensemble coordination matches frontier-bare at zero-dollar cost via the production model factory path while the OpenCode CLI shell-subprocess workaround stalls indefinitely) and two methodological findings (F1 turn-by-turn dispatch is the correct method for multi-turn frontier-tier testing under the no-paid-API constraint; F2 single-shot facsimile breaks down via imagined-state bias). The cycle closed Mode B (Research Only) at research-phase end on the practitioner's gate-close direction rather than advancing into DECIDE — a deliberate epistemic-discipline choice grounded in the susceptibility snapshot's three grounding actions, which identified that the late-cycle synthesis (Spike C and Spike D) crystallized architectural framings that exceeded the evidence scope. The cycle's deliverable is the five research logs plus audit trail; this essay is a retrospective rollup written after Cycle 4 began addressing the inherited grounding actions as research-entry inheritance. The cycle's contribution to the corpus is best read as: scope-bounded affirmative evidence that cheap-orchestrator + orchestration competes with frontier-bare on specific bug classes, with three open questions about whether the mechanism is genuinely architectural composition or instead deterministic tool access, autonomous routing or manual staging, and architectural advantage or differential information access.

## The Starting State

Cycle 3 of the agentic-serving scoped corpus opened on 2026-05-01 with Cycle 2's essay 003 ("Multi-Turn Orchestration and the Four-Axis Frame") as the load-bearing inheritance. Cycle 2 had run two literature reviews (28 + 26 sources spanning shape inventory, failure modes, mitigations, ensemble design principles, MARG-style concatenation, heterogeneity-uncorrelated-errors, and Jiang et al.'s April 2026 information-theoretic argument) and three spikes — Spike B refuting small-model substitution as a latency optimization on the existing production code-review ensemble, Spike A showing that the same ensemble's two-stage cascade-collapse summarization design is dominated on every measured axis except dimension coverage by a strong prompt-steered single cloud orchestrator with no cascade, and Spike A3 demonstrating that a *novel* ensemble informed by Loop 4's design principles (script-based deterministic analyzer plus two heterogeneous cloud LLM reviewers from different model families plus harness-enforced MARG-style concatenation) produced capabilities prompt-steering structurally cannot replicate — at a 4× latency cost workable for batch/asynchronous review but not for chat-loop interaction.

Essay 003's most consequential open scientific question carried directly into Cycle 3: "A2 + script input" — a prompt-steered single cloud orchestrator receiving the script-agent's deterministic report as additional input context — as a logically-available alternative to A3's full novel ensemble (script + heterogeneous LLMs + MARG concatenation). The cycle did not refute "A2 + script input" might deliver A3's factual grounding at A2's latency without ensemble overhead; the next cycle should test it directly. Cycle 2's susceptibility snapshot named this as a specific in-cycle grounding action for Cycle 3's research entry.

Cycle 2 also closed with a refined reading of ADR-011 (orchestrator-LLM-is-a-Model-Profile): the single-Model-Profile commitment is defensible *as a default* and as the right choice for tasks fitting Anthropic's three threshold conditions (parallelizable, context-window-exceeding, cost-justifying), but is *not a ceiling* for task classes where factual grounding via deterministic checks is part of the success criterion. The cycle had characterized the boundary empirically — "ensembles whose topology adds structural capability prompt-steering structurally lacks" (A3's territory) versus "ensembles whose topology only re-implements what prompt-steering does at higher latency" (A1's failure mode). DECIDE phase would need to record this distinction in the ADR review.

Beyond essay 003's specific carryforwards, Cycle 3 inherited the four-priorities frame as the operative design-priorities lens (performance × environmental cost × local-first preference × token cost), the four-layer architecture commitment (orchestrator + script-models + ensembles + plexus, per ADR-002), eleven accepted ADRs, and the operating frame *"outcomes over an agentic session — agent shape is means; the cycle's evaluation axis is outcome quality per session, not architectural fidelity to any pre-existing shape."* The operating frame mattered because it positioned the cycle's evidence horizon at outcomes-per-session rather than at architectural commitments — which is what later allowed the cycle's Mode B closure to remain epistemically defensible despite the substantive empirical findings the spikes produced.

## RQ-1 Isolation: A2 + Script Input Does Not Match A3's Grounding

Cycle 3's combined literature review (`004a-lit-review-agent-design.md`) confirmed that no published paper directly compared "A2 + script input" against MARG-style heterogeneous ensembles on documentation-review task classes. The lit review surfaced 23 sources spanning multi-turn reliability (τ²-Bench, HORIZON-tier follow-ups, Khanal et al. 2026), evaluation frameworks (CLEAR, PAE-aware procedural independence per Cao et al. 2025, MAST taxonomy, Topaz routing), single-agent-with-skills replacements (OneFlow, Lee et al. 2026), tool-output processing (`How Good Are LLMs at Processing Tool Outputs?` arXiv:2510.15955), information-fidelity for sequential tool chains (Fan et al. arXiv:2602.13320), and script-as-orchestrator patterns (Routine, Compiled AI, Chen et al. 2026 workflow optimization survey). The script-as-orchestrator literature is sparse but substantive — none of the surveyed publications frame their work as the dominant pattern in the field; the dominant pattern remains LLM-as-orchestrator. RQ-1's spike thus sits in a literature gap.

Spike A (`004b-spike-a-cycle3.md`) operationalized the RQ-1 isolation test as a seven-configuration probe on Cycle 2's README-review fixture. Five configurations ran on the cheap tier (MiniMax M2.5 Free via OpenCode Zen, accessed through llm-orc's production `ConfigurationManager` + `ModelFactory` path); two ran on the frontier tier (Sonnet 4.6 facsimile via Claude Code's Agent tool dispatch). The seven arms tested: arm1 (A2 baseline reconstruction — single-LLM code review on README only); arm2 (A2 + script input — same prompt with the deterministic script's report prepended); arm3 (script-as-orchestrator with the failure category named in the system prompt — flagged as CONTAMINATED at trial-results review); arm3_debiased (script-as-orchestrator with failure-mode hints removed); arm4 (script only, no README — structural test of whether bug-finding requires README ground); arm-frontier-bare (Sonnet 4.6 single-shot review of the README); arm-frontier-with-script (Sonnet 4.6 with both README and script-output as input).

The RQ-1 isolation finding on this fixture is unambiguous. Arm 2's 0-of-3 detection of the undefined-profile bugs (`default-local` and `ollama-gemma-small` referenced in the README's YAML examples but never defined in the Configuration section) replicates Cycle 2's A2-without-script behavior. Adding the script as input context did *not* shift the LLM's analytical behavior. The arm 2 trials treated the script's findings as checklist confirmation ("✅ All 5 canonical sections present, ✅ 16/16 external URLs valid, ✅ 10/10 code blocks parseable") and proceeded to general code review without doing the semantic analysis that catches undefined-profile references. Median latency dropped substantially (25.86 seconds vs arm1's 108.95 seconds) but bug detection remained 0-of-3.

The mid-spike practitioner reframe sharpened the cycle's central question from "does orchestration shape matter?" to "does cheap-orchestrator + orchestration compete with a more expensive frontier model?" The cycle absorbed the reframe quickly and ran two frontier-tier arms (arm-frontier-bare and arm-frontier-with-script) in response. The susceptibility snapshot at cycle close identifies this as a mild-to-moderate framing-adoption signal: the comparison baseline ("frontier-bare single-shot with no file access") is partially favorable to the architecture by design — frontier-bare cannot reach other files, while the architecture's script-agent guarantees cross-file access. Cycle 4's research-entry framing names this as scope condition rather than as neutral reference point.

The most novel finding in Spike A is cross-tier and unanticipated: cheap-orchestration with directed prompting and frontier-model-alone find *different and largely uncorrelated* bug classes on this fixture. The cheap-with-direction arm (arm3 contaminated; arm3_debiased after the contamination was caught) caught the undefined-profile bugs that frontier-bare and frontier-with-script both missed; the frontier arms caught configuration-hierarchy ordering inversions, encryption-claim ambiguities, model-ID staleness, and operational-documentation gaps that the cheap-tier arms did not surface. The heterogeneity-uncorrelated-errors mechanism Cycle 2 observed *across heterogeneous LLM reviewers within a single tier* extends *across tiers* as well on this fixture. The four-priorities frame produces a frame-divergent recommendation against a performance-only frame: complementary multi-tier review captures bug coverage that neither tier achieves alone. Single-fixture scope is load-bearing throughout — the broader reading requires additional task classes, which the cycle did not run.

A subtler finding from Spike A's debiased arms is consequential for ADR-011's boundary refinement. Arm 3's contaminated reliability (3-of-3 detection of both bugs with line citations and recommended fixes) versus arm 3_debiased's 1-of-3 detection of only one of the two known bugs reveals that **explicit prompt direction toward the failure category** is the dominant driver of bug detection within the cheap tier — not the architectural framing alone. The script-as-orchestrator framing produces marginal lift over generic-review framing (arm2's 0/3) but does not approach reliable detection without explicit prompt direction. The implication is that "well-architected processes" need to include prompt-direction discipline as a first-class design surface alongside ensemble composition and tool integration.

## Multi-Turn Reliability Probe: Ceilings Do Not Manifest at the Cycle's Tested Complexity

Spike B (`004c-spike-b-cycle3.md`) probed RQ-3 across two fixtures and three arms to test whether multi-turn reliability ceilings from the literature (tau-bench's 34–74% pass@1 single-and-dual-control range; HORIZON's 19% meltdown rate at very-long horizons) manifest at the cycle's deployment configurations and whether Spike A's cross-tier-uncorrelated-errors finding extends to multi-turn. The two fixtures: a tau-shape custom multi-turn library checkout (six deterministic Python tools, three scenarios — book available, book checked out, patron has fines — graded programmatically per Decision A's PAE-aware procedural-independence protocol), and a real-session haiku-generator ensemble authoring task (read existing ensemble examples, author a new YAML + a syllable-counter script, hand-test with valid and invalid haiku).

The primary substantive finding: on the cycle's tested complexity, neither tier hits a reliability ceiling. Tau-shape: cheap-tier 12-of-12 graded success across four trials post-fixture-fix; frontier-tier 6-of-6 via F1 turn-by-turn dispatch. Real-session: 6-of-6 trials produced working `haiku-generator` ensembles with valid 5-7-5 syllable-counter scripts, across three arm × trial pairings (frontier subagent native; cheap-tier `opencode run` against `llm-orc serve`; cheap-with-script preprocessor context). All six real-session trials produced files where each `haiku_syllables.py` correctly returns `valid: true` for a canonical 5-7-5 haiku and `valid: false` for a deliberately broken one when given the matching dependency key.

The cross-tier complementarity from Spike A does NOT replicate at multi-turn on these fixtures. Both tiers reach equivalent outcomes — the same correct actions on tau-shape, the same working ensembles plus scripts on real-session. **The complementarity is task-class-dependent, not universal.** Spike A's mechanism (heterogeneity-uncorrelated-errors across tier-and-capability) requires bug classes where the tiers have different blind spots; multi-turn protocol navigation and agentic-coding-task execution have less of that asymmetry on these fixtures. Cycle 4's lit-review on long-horizon reliability later established a literature-grounded explanation for this scope-dependence: failure modes are task-class-conditional, with software-engineering tasks showing the steepest degradation curve of any domain tested but with the bounded fixtures Spike B used not reaching the regime where degradation manifests.

A substantive methodological finding emerged from Spike B's F1-vs-F2 comparison. F2 single-shot facsimile testing — one subagent dispatch per scenario × trial, where the subagent reasons through full multi-turn dialogue in one shot with imagined tool results — broke down via imagined-state bias. Five of six F2 frontier traces imagined fine balances that did not exist in the actual fixture, then planned for that imagined scenario. The plans were protocol-correct given the imagined data, but the data was not real. Strict-replay outcome was 1-of-6. F1 turn-by-turn dispatch — each turn is one fresh subagent dispatch receiving conversation-so-far plus tool definitions and outputting one turn's structured response, with the harness parsing JSON, executing tools deterministically, appending results to conversation, and dispatching the next turn — produced clean reactivity (6-of-6). For RQ-3-style reliability testing under the no-paid-API constraint, **F1 is the methodologically correct approach**; F2 measures planning capability under self-imagined conditions, which is a different question.

The mid-Spike-B practitioner observation — *"MiniMax is going to perform similarly to Sonnet on simple tasks. That's more or less what you determined, yes?"* — triggered the methods-reviewer dispatch that reshaped the cycle's mid-trajectory. The observation crystallized an emerging concern that Spike B's confirmations on bounded multi-turn fixtures did not test the central question (whether cheap-orchestrator + orchestration competes with frontier on the harder regimes where reliability ceilings would manifest). The methods reviewer (`research-design-review-cycle-3-mid.md`) confirmed two P1 findings: regime mismatch (Spike B's fixtures were in the easy regime where 100 percent pass rates told the cycle little about the central question) and the `+ orchestration` gap (the architecture's `invoke_ensemble` and `compose_ensemble` primitives had not been exercised in any Cycle 3 spike, while the architecture's central claim is precisely that the cheap orchestrator's ensemble-dispatch primitive adds value over cheap-tier-alone). The reviewer recommended a three-arm code-review fixture with pre-spike fixture validation requiring at least one issue category where cheap-orchestrator-alone is expected to struggle.

## Three-Arm Architecture Comparison on Cross-File Verification (Spike C)

Spike C (`004d-spike-c-cycle3.md`) operationalized the methods reviewer's recommendation as a three-arm comparison on a synthesized 90-line Python module diff with five deliberately-injected ground-truth issues spanning categories: ISSUE-1 (off-by-one in `check_limit` — semantic), ISSUE-2 (api_key in `logger.warning` — security), ISSUE-3 (`limit: int = None` annotation — type-safety), ISSUE-4 (test deferral stated in the diff itself — process), and ISSUE-5 (cross-file value drift between `DEFAULT_BUDGET_LIMIT = 100_000` in the new module and `DEFAULT_MAX_TOKEN_LIMIT = 50_000_000` in `orchestrator_config.py`, a 500× mismatch the comment-warning in the diff calls out without importing). Pre-spike fixture validation in `ground-truth.md` named ISSUE-1 and ISSUE-5 as the two architecture-favorable cases where the script-agent's deterministic capabilities are expected to add structural value, ISSUE-4 as the floor (any tier should catch it), and ISSUE-2 and ISSUE-3 as intermediate cases.

Three arms ran. Arm A (cheap-bare): `opencode run -m llm-orc/orchestrator-minimax-m25-free --print-logs` with full Read/Edit/Bash tool access. Arm B (cheap-with-ensemble): `llm-orc invoke spike-c-code-review -f <diff>` dispatching a custom 2-agent ensemble — a script-agent (`spike_c_diff_analyzer.py` performing symbol extraction, security-pattern detection, type-annotation flagging, test-presence detection, and cross-file value verification) plus an LLM reviewer (MiniMax with system prompt instructing it to use script findings as anchor evidence). Arm C (frontier-bare): Sonnet 4.6 via subagent dispatch with the same review prompt plus diff inline, single-shot. Eight total trials (3 + 3 + 2) at zero-dollar cost across all arms.

At surface-detection resolution all eight trials caught all five ground-truth issues by keyword match. **At concrete-verification resolution — the discipline Decision A's PAE framing requires — the architectural-value finding is unambiguous: Arm B caught the actual cross-file mismatch (`100_000` vs `50_000_000`) in 3-of-3 trials; Arm A caught it in 1-of-3 trials (only when MiniMax happened to use the Read tool autonomously); Arm C caught it in 0-of-2 trials.** Sonnet single-shot has no access to other files; Sonnet conceptually flagged the drift and recommended import-from-config in both trials but did not cite the actual values. The cheap-orchestrator + ensemble configuration cleanly outperforms frontier-bare on this bug class — 3-of-3 versus 0-of-2 concrete verification at zero-dollar cost across both configurations.

The cycle's Spike C synthesis at the time framed the mechanism as deterministic-vs-probabilistic complementarity — script provides guaranteed cross-file verification; LLM reviewer provides semantic analysis on top. Spike A had established cross-tier heterogeneity-uncorrelated-errors as one mechanism the literature predicts; Spike C extended the abstraction to deterministic-vs-probabilistic complementarity within a single ensemble. Both fit a unifying frame: the architecture composes components with different error distributions; the composition's coverage exceeds any single component's.

The susceptibility snapshot at cycle close identifies a load-bearing scope condition the spike's headline framing did not propagate with the same force as the underlying evidence. The mechanism on ISSUE-5 is the script-agent's deterministic file access — not the orchestrator's ensemble-routing decision. A guaranteed cross-file file read is what produced Arm B's advantage. A script that simply ran and reported (without `invoke_ensemble` orchestration) would plausibly have produced the same result. The susceptibility snapshot identifies "the architecture's `+ orchestration` primitive is load-bearing on this bug class" as exceeding what the evidence supports; the more directly supported framing is "deterministic tool output is the mechanism on cross-file verification." This distinction is Cycle 4 territory, recorded as the snapshot's first grounding action.

A second scope condition from Spike C: N=3 on a single synthesized fixture supports a fixture-level claim, not an architecture-level principle. The "deterministic-vs-probabilistic complementarity" frame appearing fully formed in Spike C's synthesis well-supports a real empirical pattern but exceeds single-fixture scope when read as a general architectural principle. The Limitations section of `004d` named "single fixture" as item 1; headline framings did not propagate the scope qualifier with the same force.

A third scope condition from Spike C — the Arm A confound — is recorded honestly. The methods reviewer's intended Arm A design was "cheap orchestrator working alone" (single-shot LLM call, no tool access). What was actually run was `opencode run` with full Read/Edit/Bash tools available. This is closer to "cheap-tier with general agentic tools" than to "cheap-tier-alone." A purer Arm A would have used `llm-orc invoke` with a single-LLM-only ensemble. For the cycle's central-question evidence, the confound is mostly favorable to cheap-bare — Arm A had the OPPORTUNITY to use cross-file verification via Read; the architecture's advantage is that Arm B GUARANTEES it. A purer Arm A would likely score 0-of-3 concrete (matching Arm C); the variability observed is generous to cheap-bare.

## Multi-Ensemble Pilot: B1 Ties C1 at Zero Cost, with the OpenCode CLI Stall Finding

Spike D (`004e-spike-d-pilot-cycle3.md`) was scoped explicitly as a Cycle 4 priming experiment, not a primary central-question test. The spike's intent was to test whether the cheap orchestrator could coordinate a multi-stage code-review-and-fix pipeline (review → propose fix → verify) where Stage 1 and Stage 3 dispatch specialized ensembles and Stage 2 generates the fix via LLM. Three arms: A1 cheap-bare-with-tools (`opencode run` with Read/Edit/Bash, no ensemble dispatch), B1 cheap+multi-ensemble (script-agent ensemble for review, MiniMax via direct model factory for fix, script-agent ensemble for verification), C1 frontier-bare (Sonnet 4.6 via subagent, all stages collapsed into single-shot response). Verifier scoring on the proposed fixed files: A1 3-of-5 resolved (verifier false-positive on api_key hash), B1 4-of-5, C1 4-of-5. **B1 ties C1 at zero-dollar cost** — the architecture's value claim from Spike C extends to multi-stage workflows on this single trial.

The pilot's *primary* finding is unexpected and substantive: **the OpenCode CLI deployment shape stalls indefinitely on substantial code-containing prompts at the cheap tier.** Three sustained attempts to dispatch the multi-stage workflow via `opencode run` (the originally-intended Arm B1 design plus Stage 2 retry attempts) all stalled the same way — with the orchestrator process alive but consuming nearly zero CPU, the Bash tool capture showing only the initial INFO log line, and the stdout redirect file at zero bytes. Attempts ranged from 14 minutes to 14 hours before manual termination. The same MiniMax model called via the production `ConfigurationManager` + `ModelFactory` path (the Spike B harness pattern) completed the same Stage 2 prompt (4,682 characters including the original file content) in 24.8 seconds. **This isolates the failure to the OpenCode CLI deployment shape, not to MiniMax or to OpenCode Zen's API.**

The architecture's intended primitive is `invoke_ensemble` dispatched by the orchestrator's internal closed-five-tool surface (per ADR-003). When the orchestrator runs in `llm-orc serve`, this tool is part of its native action space. The OpenCode CLI deployment shape uses a different path: the orchestrator (running through opencode → llm-orc serve) gets opencode's Bash tool exposed to it via the Client Tool Surface Commitment's turn-boundary delegation pattern. Multi-stage workflows via opencode require the orchestrator to dispatch shell commands like `uv run llm-orc invoke <ensemble> -f <file>` — using a SHELL SUBPROCESS as the dispatch primitive instead of the orchestrator's internal `invoke_ensemble` tool. **This is not the architecture as designed.** The shell-subprocess substitute is a workaround for testing without standing up the full `llm-orc serve` direct-API-call deployment. The pilot's stall finding establishes that this workaround is fragile.

The susceptibility snapshot identifies two embedded-conclusion risks in `004e`. Spike D's "architecture works at multi-stage workflow level" finding rests on a manually staged pipeline that bypassed the orchestrator's actual `invoke_ensemble` autonomous primitive. The intended B1 arm (cheap orchestrator with autonomous routing intelligence) never ran as designed; a direct model factory script substituted. Cycle 4 entry should explicitly distinguish Spike D's manually staged pipeline from autonomous routing via `llm-orc serve`. The opencode CLI stall is a deployment-shape finding worth carrying forward; whether it is also a cycle finding in its own right or a sign of infrastructure noise that masked the intended pilot is a reading the synthesis presented as compatible but did not foreground.

The pilot deliberately did *not* test: autonomous routing intelligence (does the orchestrator pick the right ensemble for each sub-task without explicit instruction?); iteration loops (review → fix → verify → re-fix when issues remain); N>1 variance (single trials per arm); heterogeneous reviewers (Arm B1 used MiniMax for the LLM reviewer slot, not the full A3 architecture); higher-complexity fixtures (same 90-line single-file fixture as Spike C). All five are recorded as Cycle 4 hooks.

## The Susceptibility Snapshot's Three Grounding Actions

The phase-boundary susceptibility-snapshot dispatch at Cycle 3's research-end produced three grounding actions for Cycle 4 entry. The snapshot examined the cycle's signals across the AID gate conversations and identified two embedded-conclusion risks at the synthesis boundary — the "architecture works at multi-stage workflow level" reading from Spike D and the "deterministic-vs-probabilistic complementarity" framing from Spike C, both of which the snapshot judged as exceeding the underlying evidence's single-fixture scope when read as architecture-level principles.

The first grounding action (mechanism isolation) names the cycle's most consequential framing question: distinguish "deterministic tool output is the mechanism" from "ensemble orchestration is the mechanism" on cross-file verification. The recommended test compares cheap-bare with the script's output as input context (Spike A arm 2 pattern applied to cross-file verification) versus cheap+ensemble dispatched via `invoke_ensemble`. If the two configurations match on concrete-verification, the lesson is "deterministic tool output is the mechanism," not "ensemble orchestration is the mechanism." If the cheap+ensemble configuration outperforms cheap-with-script-input, the architectural-orchestration claim is supported beyond the deterministic-tool-access reading.

The second grounding action (autonomous routing distinction) names the evidence base for autonomous routing as Cycle 1's CAP-9 baseline plus Spike C's single-stage evidence, and names the evidence gap as multi-stage autonomous coordination via `llm-orc serve`. Without this entry-point grounding, Cycle 4 risks inheriting Spike D's headline confidence markers ("architecture works at multi-stage workflow level") and building against a foundation the evidence does not yet support. Spike D's manually staged pipeline does not establish autonomous routing; the architecture's autonomous coordination primitive across multiple stages is unsupported by Cycle 3's evidence.

The third grounding action (frontier comparison baseline) names "frontier-bare single-shot has no access to other files" as a scope condition rather than a neutral reference point. The recommended test examines what happens when frontier has matched information access (frontier + cross-file extraction script as input context). If the architecture's advantage disappears under matched access, the mechanism is "information access" not "architectural composition." If the advantage persists, the mechanism is genuinely compositional.

The three grounding actions enter Cycle 4 as research-entry inheritance, not as Cycle 3 gaps. The cycle's commitment-gating outputs at gate-close named four settled premises (Spike A's cross-tier finding on the cycle-2 README fixture is genuine, scope-bounded; Spike B's F1 vs F2 methodological finding is settled; cross-tier complementarity does not replicate at multi-turn on the library-checkout fixture; Spike C's concrete-verification result on the synthesized cross-file fixture is genuine, scope-bounded), four open questions (the three grounding actions plus whether the four-priorities frame survives a measured-divergence test or remains rhetorical at the cycle's evidence resolution), and four specific commitments carried forward (Cycle 4 territory named as supported design methods for orchestrator + ensembles; the three grounding actions as research-entry inheritance; the comparison-baseline-as-scope-condition discipline; spike artifacts retained until corpus close per practitioner directive).

## Why Mode B Was the Right Close

Cycle 3 closed at research-phase end with Mode B (Research Only) declared at cycle close on the practitioner's gate-close direction. The decision to close at research without producing an essay was a deliberate epistemic-discipline choice rather than a procedural concession. The cycle's late-cycle synthesis (Spike C and Spike D) crystallized framings the susceptibility snapshot identifies as exceeding the evidence scope — the "deterministic-vs-probabilistic complementarity" frame appearing fully formed at single-fixture scope, the "architecture works at multi-stage workflow level" framing resting on a manually staged pipeline, the "frontier-bare baseline" treated as neutral reference rather than as scope condition. An essay written at research-phase end against those framings would have propagated the embedded-conclusion risks into downstream phases (DECIDE's ADR drafting, ARCHITECT's responsibility allocation, BUILD's implementation) at higher resolution and lower revisability than research-log artifacts permit.

The practitioner's gate response named the right reframe: ground the next cycle in supported design methods for orchestrator + ensembles, with the three grounding actions as research-entry inheritance before any new architectural claim is advanced. This points toward a Cycle 4 entry that resolves the mechanism-isolation and autonomous-routing questions before any cycle's evidence is essay-grade — which is what Cycle 4 explicitly did, opening with the constraint-removal exchange and the methods-reviewer's seven-flag audit before any research dispatch.

The cycle's Mode B close should not be read as the cycle producing nothing essay-grade. The cycle produced: a confirmed cross-tier-uncorrelated-errors finding on documentation review (Spike A), a settled F1-vs-F2 methodological finding for multi-turn frontier-tier testing under the no-paid-API constraint (Spike B), a refuted cross-tier complementarity on bounded multi-turn fixtures (Spike B), a scope-bounded concrete-verification result on cross-file drift (Spike C 3-of-3 vs 0-of-2 at zero-dollar cost), and a deployment-shape finding (the OpenCode CLI shell-subprocess workaround stalls indefinitely on substantial code prompts; the production model factory path completes in 24.8 seconds). The five research logs plus audit trail are the cycle's deliverable; this essay is the retrospective rollup written after Cycle 4 began addressing the inherited grounding actions.

## Implications for the Architecture

Cycle 3's findings have specific implications for the architecture's existing commitments without forcing changes — though several recommendations from the cycle should inform downstream cycles' DECIDE-phase work.

ADR-002 (four-layer architecture) holds. The cycle's evidence does not motivate architectural reorganization. The script-agent slot in the ensemble model — exercised in Spike C and Spike D — is the structural feature that produced the architecture's value on cross-file verification, and the script-models layer is the load-bearing layer for that bug class (per the susceptibility snapshot's first grounding action). The four-layer commitment continues to support the cycle's four design priorities natively.

ADR-003 (closed five-tool internal action surface) is partially exercised by Cycle 3's evidence. The cycle's spikes did not fully test the autonomous `invoke_ensemble` dispatch path — Spike C used `llm-orc invoke` directly (CLI dispatch, not orchestrator-tool-call); Spike D's intended autonomous-routing arm could not run through the OpenCode CLI workaround and substituted manual staging via the production model factory. The closed five-tool surface remains a load-bearing design commitment, but the cycle's evidence does not yet validate it end-to-end at the orchestrator-autonomous level. This is the gap the susceptibility snapshot's second grounding action names.

ADR-004 (Result summarization mandatory) is not directly exercised by Cycle 3. The cycle's spikes used direct model-factory or single-shot subagent dispatches that do not pass through the Result Summarizer Harness. Cycle 4's behavioral spike on the research-loop dogfood fixture surfaced specificity-loss at the harness's interposition point on a single-iteration trial; that finding empirically motivates ADR candidate #5 (Harness reconsideration) in Cycle 4's DECIDE territory.

ADR-007 (Calibration Gate for composed ensembles) is not directly exercised by Cycle 3 either.
