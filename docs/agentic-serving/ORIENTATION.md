# Agentic Serving -- Orientation

## What this system is

Agentic serving extends llm-orc (a declarative DAG-based LLM orchestration engine) to serve as the backend for agentic coding tools via OpenAI-compatible endpoints. **Cycle 7 introduces the framework-driven dispatch pipeline as the primary architecture for the chat-completions surface**: every request to `/v1/chat/completions` flows through a deterministic three-stage pipeline (Plan → Dispatch → Synthesize) where the orchestrator-LLM is removed from the routing-decision and post-dispatch-synthesis surfaces. A routing-planner ensemble produces a JSON dispatch plan; the framework executes the plan deterministically; a response-synthesizer ensemble composes the user-facing response from structured input under strict-fidelity rules. The Plexus knowledge graph, when available, provides cross-session memory and design knowledge accumulation. The baseline product ships without Plexus; the learning-system value is an upgrade.

Configures by base-URL only; trusts the endpoint to behave as a transparent OpenAI-compatible proxy honoring the configured model ID; degradation signal is configuration dishonesty (per OQ #18 — Cline #10551 + OpenCode #20859). - **Tool user / Population B** — developer/script client with access to alternative llm-orc surfaces (`llm-orc invoke` CLI; direct ensemble HTTP API). On the chat-completions surface, Population B receives a structured advisory per ADR-032 §"Mechanism: Structured advisory for Population B" pointing toward the alternative surfaces where explicit capability identifiers are the normal mode. Reading path: `ORIENTATION.md` → `product-discovery.md` (Population B in Stakeholder Map) → `interaction-specs.md` (Tool User stakeholder, Population B partition) → ADR-032.

## Key constraints

Drawn from `domain-model.md` (AS-1 through AS-10) and project-level Invariants 1-14. The constraints below most shape downstream decisions:

- **AS-9 (Cycle 7) — Structurally-bounded LLM roles produce reliable output on single-decision-shaped tasks where the orchestrator-LLM-as-decider failed.** An LLM role constrained to a single-decision-shaped task (produce JSON plan from request; produce response from structured input) produces reliable output where bundled-reasoning surfaces (routing + multi-dispatch narration in one LLM) failed. The invariant names the **structural property** — role-shape, not model-shape; both candidate Cycle 7 mechanisms (framework-driven dispatch pipeline per ADR-027; Tier 1 hybrid with `tool_choice` interception) produce structurally-bounded roles. Empirical basis: 13 tests + 4 confabulation modes at qwen3:8b (Spike ε + ε' + μ). FC-29 verifies universal satisfaction on the chat-completions surface.
- **AS-10 (Cycle 7) — Capability matching on the agentic-serving chat-completions surface works from request content alone.** The routing decision is computed from the chat-completions request body alone (`messages[]`, `model`, optional `tools[]`, optional OpenAI-protocol-native `tool_choice` per ADR-030); no client-side opt-in mechanism is required, accepted, or used (no llm-orc-specific HTTP header, no skill-framework manifest field, no metadata convention). The agentic-serving endpoint presents as a transparent OpenAI-compatible endpoint at the configured base URL. AS-10 is scoped to the chat-completions surface; `llm-orc invoke` CLI + direct ensemble HTTP API accept explicit capability identifiers as their normal mode. FC-30 verifies no out-of-band signals influence routing.
- **Plexus is optional (AS-8).** The orchestrator works statelessly — serving layer, Dispatch Pipeline (Cycle 7), budget enforcement, result summarization, and ensemble composition all function without Plexus. Plexus is an upgrade to a learning system, not a prerequisite.
- **Result summarization is the default; substrate-routed ensembles skip content summarization (AS-7 amended, Cycle 6 — ADR-025).** Full ensemble result dictionaries are summarized before entering the orchestrator's context — *unless* the ensemble routes its deliverable through artifact-as-substrate per ADR-025. For inline-response ensembles (system ensembles per ADR-025; future ensembles opting out via `output_substrate: inline`), summarization remains correctness-required to prevent context rot.
- **Budget enforcement is a control plane concern (AS-3).** Turn limits and token budgets are enforced at the session level, checked at each iteration of the ReAct loop (for surfaces that use it), regardless of what the orchestrator LLM decides. Under ADR-027 on the chat-completions surface, the pipeline does not invoke a ReAct loop; budget enforcement applies to the per-dispatch ensemble invocations (FC-10 + AS-3).
- **Configuration honesty (Cycle 7 — ADR-032 §Sub-promise (1)).** Every chat-completions response declares its served-by path at three layers — `X-LLM-Orc-Served-By` header + `metadata.served_by` body field + content-layer Rule 5 framing on `action: "direct"` responses per ADR-029. The Population A degradation signal (configuration dishonesty per Cline #10551 + OpenCode #20859) is structurally prevented. FC-38 verifies three-layer consistency across paths.
- **Cost-distribution accountability (Cycle 7 — ADR-032 §Sub-promise (2); honest-residual-uncertainty grounding).** Capability-matched requests result in ensemble dispatch (strict-dispatch-when-capability-matched); operator-observable degradation signaling via `direct_completion_fallback` event + `direct_completion_rate` rolling metric per ADR-032 §"Operator-observable degradation signaling"; deployment-relative refutation threshold (sustained `direct_completion_rate` > ~15pp above operator-estimated baseline over 24-hour rolling window). The sub-promise is project-developer-lens grounded with Population A voice silent on direct corroboration — honest-residual-uncertainty per OQ #18. FC-41 verifies operationalization.
- **`tool_choice` bridge mechanism (Cycle 7 — ADR-030, provisional).** Cycle 7 BUILD ships a bridge: HTTP 200 with three-layer `tool_choice_handling: "deferred"` advisory (headers + body metadata always; content-layer Rule 5-adjacent acknowledgment on `action: "direct"` responses only). Full disposition (i) `tool_choice` honoring is deferred to follow-on cycle after ADR-027 reaches production deployment. The bridge addresses AS-10 configuration-honesty footgun (Spike κ D0 — Pydantic silently strips the parameter pre-Cycle 7) without blocking ADR-027 BUILD.
- **Orchestrator's internal action space remains five tools (ADR-003).** `invoke_ensemble`, `compose_ensemble`, `list_ensembles`, `query_knowledge`, `record_outcome` on surfaces that use the ReAct loop. Under ADR-027 on the chat-completions surface, the orchestrator-LLM ReAct loop is replaced by the framework-driven pipeline; the five-tool surface remains operative as architectural commitment for future surfaces adopting `OrchestratorRuntime` per ARCHITECT disposition (a).
- **Client tool surface commitment (Cycle 1 — Option C).** Client-declared tools (bash, file_edit, etc. from OpenCode / Roo Code / Cline) flow through the *response surface* via turn-boundary delegation; reserved TOOL_NAMES enforced. Under ADR-027 the contract is preserved structurally — the synthesizer reads multi-turn `messages[]` including prior assistant tool-calls per Spike ε' C1/C2.
- **Dispatch events route through one substrate to two destinations (ADR-023, Cycle 6 — Inversion N+2; extended in Cycle 7).** The unified Dispatch Event Substrate at L1 fans out to two registered sinks. Cycle 7 adds pipeline-stage event types (`PlanEmitted`, `DispatchFired`, `SynthesizerCompleted`, `DirectCompletionFallback`); the existing substrate is structurally extensible (Protocol-based; FC-24 unchanged). The `dispatch_id` correlation identifier remains single-source-of-truth per FC-22.
