# Agentic Serving -- Orientation

## What this system is

Agentic serving extends llm-orc (a declarative DAG-based LLM orchestration engine) to serve as the backend for agentic coding tools via OpenAI-compatible endpoints. **Cycle 7 introduces the framework-driven dispatch pipeline as the primary architecture for the chat-completions surface**: every request to `/v1/chat/completions` flows through a deterministic three-stage pipeline (Plan → Dispatch → Synthesize) where the orchestrator-LLM is removed from the routing-decision and post-dispatch-synthesis surfaces. A routing-planner ensemble produces a JSON dispatch plan; the framework executes the plan deterministically; a response-synthesizer ensemble composes the user-facing response from structured input under strict-fidelity rules.

**The Cycle 7 loop-back adds a second, parallel surface for tool-driven clients.** A BUILD-phase finding surfaced that the single-turn pipeline returns text only and cannot deliver work to a tool-driven client's local filesystem (OpenCode's north-star "delegate the work, apply it locally" flow). A **surface-mode discriminator** on the Serving Layer now routes by request shape: requests with no client tools take the single-turn pipeline (ADR-027, scoped here); requests carrying client `tools[]` engage a **layer-A loop driver** (ADR-033) that decides each agentic step, delegates per-turn content generation to a single capability ensemble (the callee), and returns the deliverable as a `tool_calls` response the client executes locally via a **client-tool-action terminal** (ADR-034). Grounding is a framework guarantee (single-action-per-turn enforcement), not a model assumption. The Plexus knowledge graph, when available, provides cross-session memory and design knowledge accumulation. The baseline product ships without Plexus; the learning-system value is an upgrade.

## Key constraints

Drawn from `domain-model.md` (AS-1 through AS-10) and project-level Invariants 1-14. The constraints below most shape downstream decisions:

- **AS-9 (Cycle 7) — Structurally-bounded LLM roles produce reliable output on single-decision-shaped tasks where the orchestrator-LLM-as-decider failed.** An LLM role constrained to a single-decision-shaped task (produce JSON plan from request; produce response from structured input) produces reliable output where bundled-reasoning surfaces (routing + multi-dispatch narration in one LLM) failed. The invariant names the **structural property** — role-shape, not model-shape; both candidate Cycle 7 mechanisms (framework-driven dispatch pipeline per ADR-027; Tier 1 hybrid with `tool_choice` interception) produce structurally-bounded roles. Empirical basis: 13 tests + 4 confabulation modes at qwen3:8b (Spike ε + ε' + μ). FC-29 verifies universal satisfaction on the chat-completions surface.
- **AS-9 §Scope boundary (Cycle 7 loop-back) — per-turn agentic-loop-driving is conditionally validated, not covered by AS-9's single-decision basis.** AS-9's empirical basis is single-decision-shaped tasks; it does NOT extend to the layer-A loop-driver (deciding each turn's next step across a multi-turn loop) without validation. The grounded-loop hypothesis is **conditionally accepted** (ADR-033 / ADR-097 pattern) along two axes: **axis 1 (grounding)** is validated *conditional on framework-enforced single-action-per-turn* (Spike τ falsified the unconstrained driver; Spike τ′ passed under enforcement); **axis 2 (sequential-composition drift over a long horizon)** remains the BUILD/PLAY validation target the short probes cannot settle — the cycle's recorded load-bearing risk. FC-43/45 verify axis-1 enforcement; FC-51 makes an axis-2 failure diagnosable (split-incorrect vs callee-incorrect).
- **Parity is the bar for the tool-driven surface (Cycle 7 loop-back) — behavioral, not latency.** Driving a tool-rich client (OpenCode) against agentic-serving produces the same *kind* of working agentic session as against a normal single model: the client executes and observes its own tool calls, with permission gates, diffs, and tool-result feedback intact, and the work lands on the client's filesystem because the client executed it. Local-model latency (and the extra turns single-action-per-turn enforcement adds) is an accepted tuning tradeoff, not a parity failure. The deliverable returns via a client-executed `tool_call` (ADR-034), never a server-side write — Spike π established a co-located server-side write fails parity. FC-47/48/49/50 + the live-end-to-end Acceptance Criteria layer verify it.
- **AS-10 (Cycle 7) — Capability matching on the agentic-serving chat-completions surface works from request content alone.** The routing decision is computed from the chat-completions request body alone (`messages[]`, `model`, optional `tools[]`, optional OpenAI-protocol-native `tool_choice` per ADR-030); no client-side opt-in mechanism is required, accepted, or used (no llm-orc-specific HTTP header, no skill-framework manifest field, no metadata convention). The agentic-serving endpoint presents as a transparent OpenAI-compatible endpoint at the configured base URL. AS-10 is scoped to the chat-completions surface; `llm-orc invoke` CLI + direct ensemble HTTP API accept explicit capability identifiers as their normal mode. FC-30 verifies no out-of-band signals influence routing.
- **Plexus is optional (AS-8).** The orchestrator works statelessly — serving layer, Dispatch Pipeline (Cycle 7), budget enforcement, result summarization, and ensemble composition all function without Plexus. Plexus is an upgrade to a learning system, not a prerequisite.
- **Result summarization is the default; substrate-routed ensembles skip content summarization (AS-7 amended, Cycle 6 — ADR-025).** Full ensemble result dictionaries are summarized before entering the orchestrator's context — *unless* the ensemble routes its deliverable through artifact-as-substrate per ADR-025. For inline-response ensembles (system ensembles per ADR-025; future ensembles opting out via `output_substrate: inline`), summarization remains correctness-required to prevent context rot.
- **Budget enforcement is a control plane concern (AS-3).** Turn limits and token budgets are enforced at the session level, checked at each iteration of the ReAct loop (for surfaces that use it), regardless of what the orchestrator LLM decides. Under ADR-027 on the chat-completions surface, the pipeline does not invoke a ReAct loop; budget enforcement applies to the per-dispatch ensemble invocations (FC-10 + AS-3).
- **Configuration honesty (Cycle 7 — ADR-032 §Sub-promise (1)).** Every chat-completions response declares its served-by path at three layers — `X-LLM-Orc-Served-By` header + `metadata.served_by` body field + content-layer Rule 5 framing on `action: "direct"` responses per ADR-029. The Population A degradation signal (configuration dishonesty per Cline #10551 + OpenCode #20859) is structurally prevented. FC-38 verifies three-layer consistency across paths.
- **Cost-distribution accountability (Cycle 7 — ADR-032 §Sub-promise (2); honest-residual-uncertainty grounding).** Capability-matched requests result in ensemble dispatch (strict-dispatch-when-capability-matched); operator-observable degradation signaling via `direct_completion_fallback` event + `direct_completion_rate` rolling metric per ADR-032 §"Operator-observable degradation signaling"; deployment-relative refutation threshold (sustained `direct_completion_rate` > ~15pp above operator-estimated baseline over 24-hour rolling window). The sub-promise is project-developer-lens grounded with Population A voice silent on direct corroboration — honest-residual-uncertainty per OQ #18. FC-41 verifies operationalization.
- **`tool_choice` bridge mechanism (Cycle 7 — ADR-030, provisional).** Cycle 7 BUILD ships a bridge: HTTP 200 with three-layer `tool_choice_handling: "deferred"` advisory (headers + body metadata always; content-layer Rule 5-adjacent acknowledgment on `action: "direct"` responses only). Full disposition (i) `tool_choice` honoring is deferred to follow-on cycle after ADR-027 reaches production deployment. The bridge addresses AS-10 configuration-honesty footgun (Spike κ D0 — Pydantic silently strips the parameter pre-Cycle 7) without blocking ADR-027 BUILD.
- **Orchestrator's internal action space remains five tools (ADR-003).** `invoke_ensemble`, `compose_ensemble`, `list_ensembles`, `query_knowledge`, `record_outcome` on surfaces that use the ReAct loop. Under ADR-027 on the chat-completions surface, the orchestrator-LLM ReAct loop is replaced by the framework-driven pipeline; the five-tool surface remains operative as architectural commitment for future surfaces adopting `OrchestratorRuntime` per ARCHITECT disposition (a).
- **Client tool surface commitment (Cycle 1 — Option C).** Client-declared tools (bash, file_edit, etc. from OpenCode / Roo Code / Cline) flow through the *response surface* via turn-boundary delegation; reserved TOOL_NAMES enforced. Under ADR-027 the contract is preserved structurally — the synthesizer reads multi-turn `messages[]` including prior assistant tool-calls per Spike ε' C1/C2.
- **Dispatch events route through one substrate to two destinations (ADR-023, Cycle 6 — Inversion N+2; extended in Cycle 7).** The unified Dispatch Event Substrate at L1 fans out to two registered sinks. Cycle 7 adds pipeline-stage event types (`PlanEmitted`, `DispatchFired`, `SynthesizerCompleted`, `DirectCompletionFallback`); the existing substrate is structurally extensible (Protocol-based; FC-24 unchanged). The `dispatch_id` correlation identifier remains single-source-of-truth per FC-22.
- **Delegation is won by composition, not coerced — and the win is stack-scoped (Cycle 7 loop-back #3 — ADR-036).** No model-layer coercion exists on this stack (`tool_choice` silently ignored by Ollama+qwen3 — third negative surface; narrowed-role prompts inert; tool-list restriction breaks the turn with empty responses). The seat-filler's delegation decision is won by composing the guidance into the **user-turn region** (the system slot loses the attention contest: baseline 0/10 vs 55/55 measured), and the win is a (composition × qwen3:14b × OpenCode 1.15.5) property — Spike ψ′ Arm D showed it does not transfer across models. The **delegation-rate meter** (deterministic turn-shape classifier denominator × `TurnDecision` numerator; provisional threshold ≥0.9) is the load-bearing safety net that makes losing the win visible; seat-filler profile swaps require recorded re-validation (FC-60). FC-58..FC-62.
- **Session termination is a framework-guaranteed judgment, not a model disposition or a deterministic policy (Cycle 7 loop-back #5 — ADR-037).** The trailing delegation guidance is two-sided (Finding F / F-ψ″.3): termination-suppressing on work-complete tails, delegation-carrying on work-remaining tails — and distinguishing them requires a model judgment (task-completeness is semantic; every deterministic source examined fails). The committed mechanism: every trailing tool-result tail opens with a **bare-form termination judgment** over the **framework-owned digest** (Session Action Record — the framework guarantees what the judgment gets to see; Spike θ round 1 vs round 2: 0/10 → 29/30 on evidence-base and standard changes alone) under the **deliverable-accounting standard** (a successful write counts as produced; code correctness out of scope). COMPLETE finishes protocol-clean (the model is the stop mechanism — the client ends the loop on a no-tool-calls response); REMAINING falls through to the ADR-036 action call **carrying the judge's own remaining-work statement + a fixed imperative as a next-step anchor** (loop-back #6 — ADR-038): routing forward the signal ADR-037 discarded, so multi-file sessions advance to the next unproduced deliverable instead of re-revising the first (the Finding G fix; Spike ρ: unanchored advance 0/10 → anchored 19/20, with a content-neutral control at 0/10 isolating the remaining-work content as the cause). The digest is the **extensible meta-record seam** — write-log first, enriched on false-stop evidence (FC-67). The AS-3 turn cap is the deterministic backstop beneath the mechanism (FC-69 makes its wiring on this surface refutable). FC-63..FC-69 (+ ADR-038's anchor FCs); Conditional Acceptance — the production digest join + a single real-OpenCode run that **advances through all deliverables and converges** discharges the joint ADR-037+038 gate.
