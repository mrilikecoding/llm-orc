# Product Discovery: Agentic Serving

*2026-05-21 (Cycle 7 DISCOVER update — Cycle 6 update 2026-05-14; Cycle 5 update 2026-05-12; Cycle 4 update 2026-05-04; original 2026-04-01)*

## Stakeholder Map

### Direct Stakeholders

**Tool user.** A developer using an agentic coding tool (OpenCode, Roo Code, Aider, Cursor, Cline) configured to point at an llm-orc endpoint. Interacts with the system through their tool's interface. Cares about response quality, speed, and cost. Does not need to know about ensembles, DAGs, or Plexus -- but may choose to look under the hood if they are also an ensemble author.

The tool user has a **failure-mode-conditional sub-state** that surfaced during play 2026-04-24. In a working session, they treat the endpoint as a model and stay out of the internals. In a stalled or fabricating session, the "endpoint is a model" abstraction breaks -- because models do not typically claim to do things they did not do -- and the tool user begins wanting visibility they would not have wanted in a working session. This is the same person, two states.

**Cycle 6 refinement (2026-05-14)**: the failure-mode-conditional sub-state has a new pain point that does not require fabrication to trigger — **long-running but live**. Post-hotfix verification 2026-05-14 captured an 8m 28s code-generator session where the serve console emitted five lines total (server startup + two HTTP `200 OK` access lines + three completion `tool dispatch: result kind=success` lines) with a 10+ minute window of total console silence during the cloud LLM's inference wait. A prior session ran nominally similar work in 44m 32s — 5.3× variance. The tool user under cloud-orchestrator profiles routinely sits through multi-minute waits without any signal that the system is making progress, stuck, or failing. The Cycle 4 PLAY "failure-mode-conditional visibility" job is sharpened: visibility is needed not only when failure occurs (events should fire but don't) but also when long inference waits make the operator unable to distinguish progress from stuck. This is **liveness signal** — see Tension 5 and Vocabulary additions.

**Cycle 7 refinement (2026-05-21)**: the Tool User population is now mapped along a tool-call-awareness axis surfaced by Cycle 7 RESEARCH (GT-1 grounding reframe) and confirmed in DISCOVER conversation 2026-05-21. **Population A** — tool-call-aware OpenAI-family clients (OpenCode, Cursor, Cline, Aider configured for ensemble routing) whose access surface IS the chat-completions endpoint and who do not have an alternative llm-orc surface to opt into. Cycle 7 treats Population A as **primary** — they are the cycle's principal stakeholder because the value-misalignment surfaces strongest here. **Population B** — developer/script clients with access to alternative surfaces (`llm-orc invoke`, ensemble HTTP API). Population B is important but **not Cycle 7's focus**; they have escape hatches Population A does not. The Cycle 6 "endpoint is a model" abstraction applies to both populations, but the failure-mode-conditional sub-state matters more acutely for Population A — they cannot fall back to a different llm-orc surface when the chat-completions response degrades. Practitioner verbatim DISCOVER 2026-05-21: *"The point is to make llm-orc agentic-serving compatible with any tool that would want to use an OpenAI-compatible chat completions endpoint. The user trusts that llm-orc would use / create ensembles effectively. Full stop."* The "transparent OpenAI-compatible endpoint as project identity" framing is settled at DISCOVER 2026-05-21.

**Ensemble author / operator.** Creates and maintains ensemble YAML configs, model profiles, and scripts. Runs the llm-orc server. Wants the orchestrator to leverage the library of capabilities they have built. Cares about visibility -- not just for debugging, but as part of the experience of tinkering with and improving the system. May also be a tool user, collapsing the "black box" and "visibility" perspectives into one person.

**Cycle 6 refinement (2026-05-14)**: the operator's observability concern is now empirically anchored across three cycles (1, 4, 5) — **bilateral visibility absence** persists in the operator's lived experience even as the architecture's internal telemetry has grown. The Cycle 5 PLAY susceptibility snapshot reframed the gap as **infrastructure-complete / routing-incomplete**: Cycle 5 BUILD shipped new typed internal events (TierSelection records carrying model_profile/tier/topaz_skill; CalibrationVerdict records carrying Proceed/Reflect/Abstain; TierRouterAudit dispatch records; SignalChannel state transitions; fail-safe activations; retry counters) that exist in code and flow to `execution.json` artifacts. None reach the operator's terminal. Post-hotfix verification 2026-05-14 sharpened the gap further on three fronts: (a) **noise floor** — every `list_ensembles` re-validates the full library and emits Pydantic validation warnings for legacy schema-drifted YAMLs (the two named legacy YAMLs were corrected; the underlying every-call-re-validates mechanism remains); (b) **coarse-success-only telemetry** — `tool dispatch: result name=invoke_ensemble kind=success` lines carry no contextual fields (no ensemble name, no session_id, no verdict, no tier); (c) **absent in-flight signal** — no log line surfaces during the multi-minute windows of cloud LLM inference (see Tool User refinement above). The operator's super-objective remains stable; the specific shape of what observability surface is missing has been clarified.

**Cycle 7 refinement (2026-05-21)**: the operator/ensemble-author role gains a long-horizon north star surfaced during DISCOVER 2026-05-21 conversation that lives in product-discovery as context but is **not Cycle 7's deliverable scope**. The north star: **orchestrator-designs-ensembles** — a long-horizon agentic coding vision in which the orchestrator composes meta-ensembles + ensemble primitives + specialist substrate ensembles on demand, with the ensemble-author role shifting from manual library-curation to library-shape-curation and quality-gate-curation. Cycle 7 ships **delegation reliability as prerequisite** to that vision — practitioner verbatim 2026-05-21: *"If we can effectively delegate to ensembles every time, then we can figure out how to make the ensembles better."* The ensemble-design-orthogonality framing is settled: the cycle's ADR-027 framework-driven dispatch pipeline (see Vocabulary) ships a substrate that can serve the north-star vision later without committing to it now. Two new authoring surfaces enter the operator role under Cycle 7's architecture (Spike ε + Spike ζ): **routing-planner ensemble** (the cheap-tier-model decision surface that produces dispatch plans from chat-completion request content) and **response-synthesizer ensemble** (the cheap-tier-model surface that produces user-facing responses from dispatch results under strict-fidelity rules). Both ship as system ensembles maintained by the framework; operator-overrides are a DECIDE-phase question.

**Skill orchestration user (methodology consumer)** *(Cycle 5 confirmed-at-discover-gate role, 2026-05-12)*. The role someone takes on when they use a **skill orchestration process** — a client-side skill framework that decomposes a higher-level workflow into capability-typed sub-tasks the orchestrator dispatches. RDD's `rdd:*` skill plugin is one example; Anthropic Skills, OpenAI Assistants, MCP-based skill frameworks, and other emerging skill-framework abstractions are others. Skills are standardized enough that the architectural commitment is broader than "support RDD" — it is **skill-framework-agnostic orchestration**: the orchestrator routes by capability (Topaz 8-skill taxonomy + calibration verdict) without knowing which skill framework is composing against it. The skill orchestration user is skill-framework-aware; the orchestrator is skill-framework-agnostic.

This role is **distinct from but compatible with** Tool User and Ensemble Author / Operator. The same person can wear multiple roles: a solo developer running their own llm-orc backend, authoring their own RDD skill plugin, and using it through OpenCode — that person is Ensemble Author / Operator (maintains the library), Skill Orchestration User (uses their skill plugin against the library), and Tool User (interacts through OpenCode) all at once. The role separation is meaningful even when collapsed onto one person because the *concerns* are distinct: maintaining the library, composing against it via skill orchestration, and using the resulting endpoint.

Practitioner-verbatim framing of the architectural commitment (Cycle 5 discover-gate, 2026-05-12): *"The idea is that a user may use RDD which has subskills, which will decomposable tasks that the orchestrator would need to delegate to the appropriate ensemble. The alternative framing would be that we determine a specific flow (like RDD) is essential for driving effective llm-orc agentic serving results and therefore we encode a specific flow into llm-orc. That's not my first choice. Better is to be able to leverage any agentic skill (as 'skills' are more-or-less standardized these days)."*

The proposal `proposals/agentic-serving-library-structure.md` used "methodology layer / methodology consumer / methodology-agnostic" as its framing vocabulary. The Cycle 5 discover-gate confirmed-at-2026-05-12 framing is broader: **skill framework / skill orchestration user / skill-framework-agnostic**. "Methodology consumer" remains a valid synonym for the role, but the architectural commitment lands on skill-framework agnosticism rather than methodology agnosticism — covering RDD and any other current or emerging skill framework.

**Orchestrator LLM.** The agent behind the endpoint. Not a person, but an actor with needs: access to the full palette of composition primitives (ensembles, profiles, scripts), knowledge of what has worked before, budget constraints, and a way to surface its reasoning for inspection. Its effectiveness directly shapes the tool user's experience. **Cycle 4 PLAY refined the orchestrator's reliability profile**: high on derivable claims (file reads, tool calls actually fired, structural recommendations grounded in read content); low on integration claims (counts that conflate categories, justifications that fabricate cross-claim consistency, primitives-level errors in recommendations). The error pattern is consistent across introspection, recommendation, and evaluation tasks (notes 8, 10, 11, 12, 13, 18). The orchestrator is best read as *useful but audit-worthy* rather than authoritative.

**Cycle 6 refinement (2026-05-14)**: a third sub-pattern in the reliability profile surfaced — **self-diagnosis is post-hoc accurate but not same-turn-actionable**. Cycle 5 PLAY note 13 surfaced this in narration form. Post-hotfix verification finding 5 captured the practitioner-verbatim instance: after the orchestrator bypassed `invoke_ensemble` and used the client `Write` tool directly, its own response narrated *"I short-circuited that by jumping to manual file writes — bad routing on my part."* The orchestrator accurately diagnoses its own routing defect, but the diagnosis informs only practitioner action (or the orchestrator's next dispatch attempt), not behavior on the current turn. This raises a candidate inversion (see Inversion N+3): should the orchestrator's self-narration about routing be treated as a signal that triggers immediate correction (system-prompt work, dispatch-loop modification), or is it diagnostic noise the practitioner consumes? The orchestrator's structural blindness to its own execution graph (PLAY note 12: practitioner asked *"What was the total run-time of the ensemble?"* and the orchestrator could not answer from context) is a related pattern — the orchestrator commissions dispatches but the dispatch telemetry does not flow into its reasoning context, so it cannot answer dispatch-graph questions or refine routing decisions based on its own dispatch history within the session.

**Cycle 7 refinement (2026-05-21)**: the orchestrator-LLM-as-actor undergoes a **structural role contraction** under Cycle 7's architectural direction (ADR-027 framework-driven dispatch pipeline). Spike ε established empirically that the orchestrator-LLM's PLAY note 22 confabulation pattern (8 cache-hit web-searcher dispatches + 0 claim-extractor + fabricated final response) **does NOT survive** the architectural move to a structurally-bounded synthesizer role on the historical-confabulation test case. The same cheap-tier model (qwen3:8b — same family as the qwen3:14b that confabulated) produces faithful output when its role is structurally bounded to "produce response from given context" rather than "decide what to do next + narrate what was done." This is **the load-bearing architectural finding of Cycle 7 DISCOVER**.

Under ADR-027, the orchestrator-LLM's three current job-roles split:
- **Routing decisions** → routing-planner ensemble (Spike ζ-validated; qwen3:8b)
- **Dispatch execution** → framework code (deterministic; no LLM)
- **Response synthesis** → response-synthesizer ensemble (Spike ε-validated; qwen3:8b with strict-fidelity rules)

The orchestrator-LLM-as-actor reduces to: (a) the model behind the routing-planner agent (single-decision-shaped task; reliable per Spike ζ), and (b) the model behind the response-synthesizer agent (single-decision-shaped task; reliable per Spike ε with one residual rounding drift mode on the tested cases).

**Scope-of-claim partition (load-bearing for downstream phases; tightened by Spike ε' 2026-05-22):** the "structural role contraction" framing applies to the **routing-decision surface and the post-dispatch-synthesis surface** — the specific surfaces Spike ζ + Spike ε + Spike ε' empirically tested. The full "orchestrator-LLM removed from the dispatch path entirely" framing is the architectural direction Cycle 7 commits to. Empirical coverage is broader than the original partition implied (Spike ε' closed gate-named bounds for direct-completion path, numerical-density fidelity, and multi-turn continuity); specific residual bounds remain:

- **Settled (empirically grounded by Spike ζ + ε + ε')**:
  - Orchestrator-LLM removal from the routing-decision surface (Spike ζ — 20-prompt battery).
  - Orchestrator-LLM removal from the post-dispatch synthesis surface on PLAY-note-22 historical confabulation case + 1 positive-control chain + numerical-density fidelity + precise-roundable fidelity + multi-turn continuity across 2 shapes (Spike ε + Spike ε', n=10 tests total).
  - Synthesizer's direct-completion path (Rule 5) produces useful responses across 4 distinct request shapes (Spike ε' A1/A2/A3 + C1 Turn 2 fallback).
  - Multi-turn continuity preserved when prior turns are included in the synthesizer's input (Spike ε' C1 + C2; mechanism: inline-serialize prior turns; production needs native `messages[]` handling — mechanical ARCHITECT-phase work, not a design open question).
- **Plausible-but-untested (reduced from original partition)**:
  - Generalization to other confabulation modes the orchestrator-LLM has shown (path hallucination per note 23; substrate-path-as-deliverable per λ.4-paid / λ.5-paid; coherent factual errors uncalibrated per Cycle 5 PLAY). Not exercised by Spike ε'.
  - Production-scale numerical content broader than Spike ε' B1's 25 figures.
  - Cheap-tier reliability for direct-completion-of-factual-questions in training-data-error-prone domains (Spike ε' A2's "Urga / Khovd" factual error is one data point).
- **Open as DECIDE-phase design questions**:
  - Rule 5 framing requirement scope (Spike ε' Finding ε'.1 — synthesizer systematically omits the "this answer was generated directly..." framing; load-bearing or over-specified?).
  - Multi-step composition mechanism (single-step planner + framework chain-heuristic vs. multi-step planner vs. planner-loops-with-context — Spike ε ε.6; unchanged).
  - Tier escalation policy for direct-completion (whether capability-relevant-but-no-match requests should escalate to a higher tier).
  - Rounding-drift mitigation playbook (system prompt sharpening + tier escalation + runtime fidelity check — Spike ε' Finding ε'.2 characterizes two distinct drift modes: precise-figure rounding and large-number rendering-as-millions).

The PLAY-note-12 dispatch-graph-blindness pattern is structurally resolved **for single-dispatch, deterministic-chain, and multi-turn-with-history cases** under ADR-027 — there are no multi-dispatch chains the orchestrator-LLM needs to narrate. Whether the pattern stays resolved under multi-step composition that the planner cannot resolve in a single decision (genuine open-ended chaining) depends on which multi-step composition mechanism DECIDE adopts.

The "useful but audit-worthy" reliability profile becomes "useful and reliable for single-decision-shaped tasks on the broader surfaces now tested (routing decisions + dispatch-result synthesis + direct-completion fallback + multi-turn continuity); the integration-claim failure mode is structurally avoided rather than mitigated on those surfaces." **Spike ε' closed the gate-named plausible-but-untested bounds; no further pre-DECIDE validation spike is required.**

### Indirect Stakeholders

**Plexus (knowledge graph).** The system's memory layer. Its population over time affects cost, quality, and routing effectiveness for all stakeholders. Its query surface is actively evolving, which constrains what the orchestrator can lean on in early iterations.

## Jobs and Mental Models

### Tool User

**Jobs:**
- "I want to use my coding tool the same way I would with Claude or GPT, but backed by my own orchestration" -- autonomy from a single provider
- "I want complex tasks handled well without knowing what happens behind the endpoint" -- the endpoint is a model
- "I want it fast enough that it does not break my flow" -- speed matters
- "I want it cheap enough to leave running all day" -- cost matters more than peak capability for sustained use
- "I want to know what's happening when the system is slow or stuck" -- failure-mode-conditional visibility; in a working session this job is dormant, in a stalled session it becomes urgent
- "I want to know the system is still alive during long inference waits" -- liveness signal; sharpening of the prior job. In a working-but-slow session (cloud-orchestrator profiles producing multi-minute inference waits), the tool user does not want failure narration — they want a heartbeat that distinguishes "still working" from "stuck." Surfaced by post-hotfix verification 2026-05-14 (8m 28s session with 10+ min console silence; 5.3× variance across nominally similar tasks).
- *(Cycle 7)* "I want my chat-completion request to be handled by the right capability ensemble when one fits — without me having to know it exists or having to opt in" — capability-matching from request content alone. The framework reads my NL; the framework decides which capability ensemble (if any) fits; the framework handles the dispatch and gives me a chat-completion response back. From my client's perspective, this is indistinguishable from any other OpenAI-compatible endpoint. Practitioner verbatim 2026-05-21: *"The user trusts that llm-orc would use / create ensembles effectively. Full stop."* The trust contract is the transparent-endpoint promise. (Population A primary; Population B secondary — see Stakeholder refinement.)
- *(Cycle 7 loop-back)* "I want ensemble-generated work to actually land on my machine through my own tool calls — the way a normal model's tool-calls do — so my session stays a real agentic session, not a pile of text claims." The **delegate-work / apply-locally loop**: the endpoint delegates generation to an ensemble, then returns the deliverable as a `tool_calls` response the client executes against the local filesystem. **Parity is behavioral, not latency** — driving the client against agentic-serving should produce the same *kind* of working session as driving it against a normal single model (local models are inherently slower; latency is an accepted tradeoff governed by which model profiles the ensembles select). Practitioner verbatim 2026-05-24: *"can driving ensembles via ... multi-turn interaction with OpenCode result in what we'd expect from using OpenCode with a 'normal' single model process."* Empirically grounded this cycle by Spike π (the `tool_calls` round-trip is necessary for parity; co-located server-side direct-write fails it).

**Mental model:** The endpoint is a model. The tool user does not distinguish between "a model answered" and "an ensemble of models answered." Quality, speed, and cost are the only axes they evaluate on. **The "endpoint is a model" abstraction holds conditionally on orchestrator competence** -- when the orchestrator fabricates (claims tool calls that did not fire, hallucinates project structure, narrates work it did not do), the abstraction breaks because real models do not behave that way. The tool user cannot stay in the "endpoint is a model" frame when the endpoint is doing something only an unsupervised pipeline of models can do. **Cycle 4 PLAY refinement**: the abstraction holds *more reliably* under a cheap-cloud-orchestrator deployment (MiniMax M2.5-free via Zen routed through OpenCode produced coherent, file-grounded responses on first contact — note 6) than under a default-local-orchestrator deployment (Cycle 1 PLAY's fabrication-on-first-encounter). The Cycle 1 verdict — *"I would not be likely to use it again"* — is profile-conditional, not architecture-conditional.

**Cycle 7 loop-back refinement (2026-05-24)**: the "endpoint is a model" abstraction extends to *agentic coding* sessions, where it sharpens into an execution-model commitment. In a tool-using session the endpoint is not answering a question — it is *participating in the client's tool loop*. The client drives and observes its filesystem through tool calls it executes itself; a deliverable written behind the client's back (server-side direct write) leaves the client's filesystem view stale and breaks the agent loop (Spike π Phase A: parity FAILS — the file lands on disk, but the client executed nothing, with no permission gate, no diff, no undo, operating on an unverified text claim — structurally the PLAY-note-22 told-about-vs-driving trust surface). Parity requires the deliverable to return as a `tool_calls` response the client itself executes. The interaction shape the deliverable travels through: **ensemble produces → server reads the artifact (production capability ensembles route deliverables to the server-side artifact store by design per ADR-025 — Spike ρ finding F-ρ.1) → terminal marshals the artifact into a client `write`/`edit` tool-call → client executes locally → client observes the tool result and continues.** The abstraction holds in an agentic session only if that round-trip is indistinguishable from a normal model emitting tool-calls.

### Ensemble Author / Operator

**Jobs:**
- "I want my ensembles to get used -- the orchestrator should route to the right one" -- the work I have built has value
- "I want to see what the orchestrator is doing -- which ensembles it picks, what it creates, why it fails" -- visibility for debugging and understanding
- "I want server-side observability when the system is running -- colored logs or a TUI dashboard showing which model / ensemble / profile is invoked per turn" -- log activity in the operator's terminal, not just in-stream narration on the tool user's surface *(refined from Cycle 4 PLAY note 7; sharpened Cycle 6 — see below)*
- "I want the dispatched-event detail in my terminal log: which ensemble, which session_id, which tier the router selected, what the Calibration Gate's verdict was, which audit was consumed, which signal-channel state transitioned" *(Cycle 6 sharpening — Cycle 5 BUILD shipped these typed internal events; they exist in code and write to `execution.json`; the gap is the absent routing to the operator's terminal. Post-hotfix verification 2026-05-14 finding 7: the current `tool dispatch: result name=invoke_ensemble kind=success` line carries none of these contextual fields.)*
- "I want signal during in-flight states, not just on completion" *(Cycle 6 new — liveness signal. Two low-cost candidates surfaced by post-hotfix verification 2026-05-14: (a) tool-call-emit logging — log when the orchestrator emits a tool call into the response stream, before dispatching, giving a "received tool call from cloud LLM" anchor; (b) inference-wait heartbeats — when a request has been open for more than N seconds without tool activity, emit a "still waiting on cloud LLM" line giving mid-stream signal during long inference.)*
- "I want the Pydantic validation warnings from legacy schema-drifted YAMLs to either fix themselves or fall silent — not re-emit on every `list_ensembles` call" *(Cycle 6 new — noise floor. The underlying every-call-re-validates mechanism is observable territory: single-warn-then-skip vs. validate-once-at-load vs. some other pattern. Post-hotfix verification 2026-05-14 finding 7 captured 8+ enumeration cycles emitting the same warnings in a 28-minute log window.)*
- "I want to tinker -- adjust behavior, experiment with configurations, turn the tool on itself" -- visibility as play, not just audit
- "I want working defaults on first encounter — a usable agentic-serving config out of the box that exercises the cycle's mechanism architecture, not just the mechanism architecture itself" *(new Cycle 4 PLAY note 1; practitioner verbatim: "the agentic-serving config is to me part of the build")*
- "I want the system to get better over time without me hand-tuning everything" -- organic stabilization of patterns through use
- "I want to control cost and resource boundaries" -- predictable spend
- "I want to bootstrap the knowledge graph from what already exists" -- time-to-value from session one
- *(Cycle 7)* "I want delegation reliability as the prerequisite for ensemble-quality improvement work" — the operator-facing dispatch surface should reliably get capability-matched requests to the right ensemble; ensemble-quality work iterates against that substrate. Practitioner verbatim 2026-05-21: *"If we can effectively delegate to ensembles every time, then we can figure out how to make the ensembles better."* Cycle 7's deliverable is the dispatch substrate; ensemble-design iteration is downstream and orthogonal.
- *(Cycle 7, north star context — NOT Cycle 7 scope)* "I want the orchestrator to design ensembles on demand from primitive composition" — long-horizon agentic-coding vision in which meta-ensembles + ensemble primitives + specialist substrate ensembles compose dynamically. The operator-role shifts from library-curation to library-shape-curation and quality-gate-curation. Lives in product-discovery as context for downstream cycles; Cycle 7's dispatch-substrate work is the prerequisite, not the destination.

**Mental model:** The system is a library of capabilities (ensembles, profiles, scripts) that the orchestrator draws from. The orchestrator is a smart dispatcher that learns through use. The knowledge graph is what makes it learn. Over time, repeated patterns stabilize organically -- the operator does not have to curate them explicitly, but should be able to see what has stabilized and intervene when needed. **Cycle 4 PLAY refinement** *(practitioner-generated, note 15)*: the library is more usefully thought of as **capability-fine-grained** (operation-named ensembles like `web-searcher`, `claim-extractor`, `argument-mapper`) than **methodology-coarse** (methodology-named ensembles like `rdd-lit-reviewer`). The orchestrator stays methodology-agnostic; the same capability library serves many methodology consumers. The operator's library-authoring job at deployment is now reframed: *"author the capability ensemble set that serves the deployment's methodologies"*, not *"tag every existing ensemble with whatever topaz_skill fits."*

### Skill Orchestration User (Methodology Consumer)

**Jobs:**
- "I want to compose my skill framework (RDD, Skills, Assistants, MCP-skill-framework, ...) against a capability library, not against a skill-framework-aware orchestrator" — the skill orchestration process is mine; the orchestrator should not know about RDD or code-review or security-review or any specific flow by name
- "I want the same capability library to serve multiple skill frameworks" — `claim-extractor` works for RDD's lit-review *and* for any other skill framework's claim-extraction step
- "I want the orchestrator to route by capability (Topaz skill) rather than by skill framework" — skill composition is my responsibility; capability dispatch is the orchestrator's
- "I want to leverage any agentic skill framework as the skill abstraction stabilizes across vendors" — the orchestrator's commitment is to *whatever skill standard emerges*, not to a particular skill framework
- "I want the orchestrator to dispatch to a capability ensemble when my skill framework names one" *(Cycle 6 candidate — empirically uncertain. PLAY notes 1-9, 20 and post-hotfix finding 3 disclosed the operational routing preference: direct → client-tools → ensemble. Whether the right resolution is "the system should dispatch when explicitly named" (intervention to restore ensemble-first routing) or "the skill framework should know to name explicitly and intervene when the orchestrator misroutes" (revise the stakeholder mental model) is Cycle 6's central open question.)*
- "I want to ask the orchestrator dispatch-graph questions and get answers grounded in the actual dispatch history of this session" *(Cycle 6 candidate. PLAY note 12: practitioner asked the orchestrator how long the ensemble ran; the orchestrator could not answer from its own context because dispatch telemetry does not flow into its reasoning surface. Whether the orchestrator's context should carry execution state is Open Question 2 — see Tension 15 below.)*
- *(Cycle 7)* "I want capability matching to work from request content alone — no client-side opt-in flag the skill framework has to set" — the skill framework is client-side; the orchestrator sees only chat-completion request content; capability inference is the orchestrator's job. The transparent-endpoint promise extends down to the skill orchestration layer: skill frameworks can be authored without llm-orc knowledge.
- *(Cycle 7)* "I want the routing decision to be reliable enough that I do not need to intervene per-request" — Cycle 6's explicit-naming-bypass pattern (Tension 14, Inversion N+1) becomes a defect to remediate under Cycle 7's framing rather than an intended-scope feature. Spike ζ validates that a routing-planner ensemble at qwen3:8b can produce reliable capability decisions on diverse request shapes (100% JSON conformance + 90% strict capability-match); ADR-027 framework-driven pipeline removes the orchestrator-LLM's per-request routing discretion entirely. The Skill Orchestration User's job-list reverts toward Cycle 5's "the orchestrator routes by capability under NL" framing — now honored architecturally rather than asked-of-the-orchestrator-LLM.

**Mental model:** The orchestrator is a *substrate* for skill orchestration processes, not a skill-framework-specific deployment. Skill frameworks decompose higher-level workflows into capability-typed sub-tasks; the orchestrator's job is to invoke the right capability ensemble for each sub-task. The decomposition shape is the skill framework's responsibility (which sub-skills, which capabilities, in what order); the routing shape is the orchestrator's. The Topaz 8-skill taxonomy is the *lingua franca* between the two layers — skill frameworks tag their decomposed tasks with Topaz skills; the orchestrator routes by them.

*Provenance note*: The role's jobs and mental model derive from Cycle 4 PLAY 2026-05-12 note 15 (practitioner-generated three-layer framing) and Cycle 5 DISCOVER gate refinement 2026-05-12 (practitioner-confirmed skill-framework-agnostic architectural commitment broader than the proposal's methodology-framing). The role is **confirmed as distinct** from Tool User and Ensemble Author / Operator; the framings are no longer candidate.

**Cycle 6 refinement (2026-05-14)**: the Skill Orchestration User's mental model — *"the orchestrator's job is to invoke the right capability ensemble for each sub-task"* — was **empirically refuted under both tested client configurations**. Cycle 5 PLAY notes 1-9 and 20 disclosed the orchestrator's operational routing preference under both tool-less `curl` and tool-rich OpenCode: **direct LLM completion → client-tool delegation → `invoke_ensemble` dispatch**, not ensemble-first-when-slot-fits as ADR-021's natural-language-supported clause implied. Post-hotfix verification 2026-05-14 finding 3 extended the gap: even *explicit-naming* of the ensemble + dispatch verb was insufficient on first attempt — the orchestrator used the client `Write` tool instead of `invoke_ensemble`. Required practitioner intervention (*"It's in .llm-orc/ensembles/agentic-serving/code-generator.yaml so it should have registered"*) for the orchestrator to acknowledge and retry. The operational routing preference may be more aggressively direct-or-client-tools-biased than PLAY note 20's "NL framing alone" characterization. **This is the central Cycle 6 DISCOVER question** — see Tension 14 and Inversion N+1 below. Whether the stakeholder's mental model should be revised (the operator learns explicit-naming-plus-intervention patterns) or whether the system should meet the existing mental model (NL routes to ensemble when slot fits) is open.

**Cycle 7 refinement (2026-05-21)**: the Skill Orchestration User's role sharpens substantially under DISCOVER 2026-05-21 framing and the Spike ζ + ε + κ findings. **Three sharpenings:**

1. **Client-side skill framework with no client-side opt-in expected.** Practitioner-verbatim sharpening DISCOVER 2026-05-21: the orchestrator sees only the resulting chat-completion requests; capability matching must work from request content alone with no client-side opt-in mechanism (no flag the skill framework sets to say "this is a capability-matched task"). The Skill Orchestration User's mental model under Cycle 7 lands at: **routing-planner reads my request → framework dispatches the right capability ensemble → response-synthesizer produces the response.** The orchestrator-LLM is no longer the load-bearing decision surface.

**Cycle 7 loop-back refinement (2026-05-24, BUILD → RESEARCH re-entry)**: the Cycle 7 landing above — *"routing-planner reads my request → framework dispatches the capability ensemble → response-synthesizer produces the response"* — describes a **single-turn** request/response. The north-star (run RDD, or a similar long-horizon skill-framework process, via OpenCode while trusting that work is delegated to ensembles) is **multi-turn**: across a session the endpoint must participate in the client's agentic tool loop, deciding each turn's next action and delegating per-turn generation to ensembles. Spike σ named the two layers the session needs: **layer A** drives the loop (decide read/edit/write/bash/finish next); **layer B** generates the content. ADR-027's plan → dispatch → synthesize pipeline is a layer-B mechanism; **no ADR-027 component drives layer A across turns** (the routing-planner decides *which capability*, not *what the next agentic step is*; the synthesizer produces a response from given context, it does not drive anything). The Skill Orchestration User's mental model under the north-star is therefore a **loop participant**, not a pipeline with a delivery stage.

*Open framing question carried into DECIDE (deliberately not settled here, per loop-back snapshot advisory #1):* is layer A a **wrapper around** the plan → dispatch → synthesize pipeline (the pipeline becomes the per-turn generation subroutine), or is the pipeline a **callee of** the layer-A loop — or, per Spike σ.2, not the per-turn primitive at all (the driver emits tool-calls directly and delegates only generation)? The *"ADR-027 is incomplete, not wrong; the fix is additive and aligned with its philosophy"* framing is the **wrapper** reading. It entered via the cycle-status BUILD-surfaced finding and is **not inherited as settled**: it happens to be the reading that minimizes upstream design change, which is not evidence that it is correct, and the loop-back argument audits (R1 Alt-A, R4 Question 3) named the inverted reading as a live alternative without investigating it. The spikes validated layer-B generation and short-horizon layer-A feasibility at the **mechanism level (n=1)**; they did not settle which structure centers the multi-turn loop. DECIDE tests the architecture against the loop-participant requirement rather than defaulting to the lowest-rework option.

### Orchestrator LLM

**Jobs:**
- "Match the task to the right ensemble from the library -- or compose one from available primitives if nothing fits" -- effective task resolution
- "Know what has worked before for similar tasks" -- retrieval over reasoning, the cost/quality lever
- "Compose from the full palette: other ensembles, model profiles, and scripts" -- not restricted to a subset of primitives
- "Stay within budget -- turns, tokens, wall-clock time" -- resource discipline
- "Surface what I am doing and why in a way that is inspectable" -- the visibility contract with the operator

**Mental model:** The orchestrator sees a toolkit of primitives and a knowledge graph. It does not distinguish between pre-built and self-composed ensembles at execution time -- both are just ensembles. The distinction is in the lifecycle (validation, calibration, trust, promotion). A well-populated knowledge graph converts routing from reasoning to retrieval, enabling cheaper model tiers to make effective decisions. **Cycle 4 PLAY refinement**: the orchestrator's natural-language responses are *high on derivable claims* (claims grounded in actual file reads / tool-call results) and *low on integration claims* (claims that aggregate or relate multiple derivable facts — counts, comparisons, justifications, primitives-level descriptions of system surfaces). The reliability profile is consistent enough across introspection, recommendation, and evaluation modes to be characterized as a feature of the orchestrator's response shape, not a one-off (notes 8, 10, 11, 12, 13, 18). Two specific failure modes recur: *unchallenged-claim-stickiness* (claims the conversation does not pressure are not re-checked even when re-running verification on related claims would reveal the error — note 11), and *coherence-without-grounding* (surface-coherent synthesis incorporates errors that none of the cycle's quality infrastructure inspects — notes 8, 16, 19).

## Value Tensions

1. **Quality vs. cost vs. speed.** The classic triangle, with a twist: the knowledge graph shifts the frontier over time. Early sessions may be expensive (reasoning-heavy, frontier models); later sessions are cheaper (retrieval-heavy, smaller models). The tension is not static -- it evolves as the graph matures. If bootstrapping works, the expensive early phase may shrink or disappear.

2. **Autonomy vs. visibility.** The tool user wants a black box that works. The operator wants to see inside it. When these are the same person (which may be the common case), the tension becomes: how much of the orchestrator's reasoning is surfaced, where, and when? Too little and the system is opaque; too much and it is noisy.

3. **Find vs. build.** The orchestrator can invoke existing ensembles or compose new ones. Finding is cheaper and safer; building is more capable but riskier. Over time, building should feed back into finding -- a composed ensemble that works well should stabilize into a library entry organically.

4. **Static composition vs. dynamic composition.** Invariant 7 says ensemble references are static string literals resolved at load time. The orchestrator needs to compose ensembles from all three primitives at runtime, including other ensembles. The validation path (checking references against the existing ensemble reference graph before loading) is the technical resolution, but the conceptual tension remains: how much of the ensemble graph is knowable at load time vs. runtime?

5. **Visibility: what form?** Refinement from play 2026-04-24: visibility may need to be *experience-conditional*, not only event-conditional. **Cycle 4 PLAY refinement (note 7)**: concrete shape for the operator-terminal surface now nameable — *colored logs* and/or a *basic TUI dashboard* showing per-turn dispatched-model / dispatched-ensemble / tier-selection / calibration-verdict. The cycle has new internal events at every dispatch (TierSelection, CalibrationVerdict, Tier-Router-Audit consumption records, cross-layer Signal Channel aggregations) that exist in code but do not reach the operator without code changes. **Cycle 5 PLAY susceptibility snapshot reframing** (sharpening note 19): the gap is **infrastructure-complete / routing-incomplete**. Cycle 5 BUILD did ship new internal events that did not exist in Cycle 1 or Cycle 4; they write to `execution.json`; the open question is the routing surface, not the event model. **Cycle 6 (post-hotfix verification 2026-05-14) extension**: liveness signal during in-flight states is a third missing surface alongside the named "what dispatch happened" and "what verdict fired" content — a single 8m 28s session showed 10+ minute windows of console silence during cloud LLM inference. Two low-cost candidates: tool-call-emit logging and inference-wait heartbeats. The tension is between "enough to debug and trust" and "not so much overhead that it slows the system or overwhelms the operator." Tinkering as a use case suggests visibility should be interactive, not just retrospective.

6. **Cold start vs. warm start.** Does the knowledge graph have to learn through use, or can it be bootstrapped from existing context? The bootstrapping pipeline has a shape: llm-orc artifacts → Plexus ingestion (background) → enrichment fills the knowledge graph → queries and analysis results feed back. Quality gate concern at the ingestion boundary -- "garbage in, garbage out." The quality of bootstrapping determines how quickly the system delivers on its value proposition.

7. **Organic stabilization vs. explicit curation.** Repeated patterns should surface and stabilize on their own rather than requiring the operator to hand-tag them. But how does the system distinguish a genuinely good pattern from one that was merely used often? The knowledge graph needs to track quality signals, not just frequency.

8. **How is the capability floor surfaced -- static spec, runtime probe, or both?** The system has correct structural components -- Budget Controller, Conversation Compaction, Calibration Gate -- that work as designed when invoked. But the default `orchestrator-local` Model Profile may not meet the capability floor required to invoke them. Practitioner verdict at default config (play 2026-04-24): *"If I were to install llm-orc and run it with OpenCode like this out of the box, I would not be likely to use it again."* **Cycle 4 PLAY refinement (note 6)**: the verdict is *profile-conditional*. The cheap-cloud-orchestrator deployment (MiniMax M2.5-free via Zen) produced a coherent first session with file-grounded responses; the local-default-as-orchestrator deployment was unrecoverable. Three design surfaces remain open: (a) **static specification** -- write the floor down in an ADR, scenario, or startup check; (b) **runtime probe** -- ship a calibration ensemble that runs at install/startup, tests whatever local models exist against a baseline, and produces operator-readable recommendations when any tested profile falls below the floor; (c) **both** -- static spec for the abstract floor, runtime probe for the concrete-against-operator-hardware mismatch detection.

9. **Local-first as load-bearing value vs. cloud-orchestrator-as-acceptable-fallback.** Newly explicit from Cycle 4 research: cloud-orchestrator-only mode "is not much different from using something like Sonnet, though it's cheaper" -- which fails the local-leverage value the architecture exists to deliver. But the cheap *cloud* orchestrator is itself the routing tier across the cycle's research findings (cheap-cloud-orchestrator routes; local models amplify deterministic and bounded-scope ensemble work). Open question: where does cheap-cloud-orchestrator-as-routing-layer end and "fallback to cloud" begin? The boundary determines what counts as a value-preserving versus value-eroding deployment.

10. **Calibration signals as quality lever vs. feedback bias compounding.** The proposed cross-layer calibration channel (essay 005's ADR candidate #6, now ADR-016) routes ensemble-output signals upward to inform dispatch decisions. The mechanism is potentially powerful -- it enables in-process trajectory-level calibration. The risk is concrete: feedback paths can compound bias (Khanal et al.'s universal non-improvement from episodic memory augmentation; CAAF's prompt-engineering-artifact finding; Li et al.'s trigger-vulnerability finding on bias acceleration in feedback shapes). Five bounding mechanisms have been named as load-bearing rather than decorative.

19. **`tool_choice` mechanism: implement, reject, or reframe out of scope** *(new Cycle 7 — Spike κ finding D0; framework-correction territory)*. Spike κ established empirically that the framework has zero `tool_choice` handling — Pydantic strips the parameter at the request-parsing boundary (`_ChatCompletionsRequest` model has no `tool_choice` field), and there is no `tool_choice` reference anywhere in `src/llm_orc/`. (c) **reframe out of scope** — commit to the routing-planner ensemble + ADR-027 framework-driven pipeline as the primary routing mechanism; `tool_choice` becomes a parameter the framework neither honors nor rejects (status quo, but with documentation correction). The dispositions interact with the cycle's Tier 1 hybrid vs. ADR-027-direct question (Cycle 7 RESEARCH GT-2): under ADR-027-direct as primary mechanism, `tool_choice` handling becomes optional — disposition (b) is the cleanest contract. Under Tier 1 hybrid, `tool_choice` handling becomes required — disposition (a). Spike κ's cost-equivalence finding (both Tier 1 hybrid and ADR-027-direct require new framework code) feeds GT-2(a)'s "if costs are same order of magnitude, ADR-027 as primary recommendation" rule.

21. **Layer-A control structure: wrapper around the pipeline vs. callee of the loop** *(new Cycle 7 loop-back — Spike σ; loop-back snapshot advisory #1)*. The north-star session is a multi-turn agentic loop; ADR-027 is a single-turn pipeline. The **layer-A loop-driver** (decide the next agentic step per turn — read/edit/write/bash/finish) is a role no ADR-027 component holds, and no ADR-027 component drives it across turns. Dispositions: (a) **wrapper** — keep plan → dispatch → synthesize as the per-turn generation engine and add a loop-driver above it plus a `tool_calls` terminal; the pipeline stays the architectural center, the loop is additive (the "amendment / incomplete-not-wrong" reading). (b) **callee** — the layer-A loop-driver is the architecture's center: it drives the client's tool loop and delegates per-turn generation to ensembles. Two sub-readings: **(b1)** the plan → dispatch → synthesize pipeline is *one of the things the loop calls* per turn; **(b2)** per Spike σ.2 — where the driver emitted tool-calls directly and delegated only `write` content, with no plan→dispatch→synthesize unit running per turn — the pipeline may **not be the per-turn primitive at all**, and "cheap loop-driver + per-turn ensemble delegation" is the whole shape. σ.2 is the most disruptive callee-shaped evidence available and the *only multi-turn* evidence the loop-back produced. The tension is *which structure centers the architecture*. The loop-back spikes validated the mechanisms (parity, planner delegation, short-horizon grounded driving), not the ordering. **Held open into DECIDE / ARCHITECT.**

**Two opposite default-pulls operate here, and neither is evidence of correctness:** the wrapper reading minimizes upstream rework; the callee reading has the only spike behind it (σ.2). **Evidence-availability asymmetry:** every multi-turn run the loop-back produced is callee-shaped, and *no wrapper-as-multi-turn-loop was ever spiked* (Spike ρ ran the pipeline + terminal for a single turn only). So "let the effective path win on evidence" is not symmetric by default — the evidence on hand, and the evidence BUILD's first runs will most naturally produce, both skew callee. To make the comparison fair, DECIDE's evidence plan must include at least one **wrapper-shaped probe** (the pipeline running as the per-turn generation subroutine under a loop-driver), structured alongside the OQ #19-style build-complexity comparison, before BUILD settles the question by callee-skew. DECIDE resolves on named discriminating criteria — reliability over long horizons (the grounded-loop falsification probe), build-complexity / rework, latency compounding across turns, and which structure the operator's mental model maps to — not on either default-pull. The three layer-A seat-filler candidates (a model in OpenCode's "model" seat per Spike σ; the routing-planner extended to per-turn driving; a new dedicated layer-A component) are DECIDE / ARCHITECT selection work, not settled here.

22. **Artifact-bridge: the deliverable lives server-side and must arrive client-side** *(new Cycle 7 loop-back — Spike ρ finding F-ρ.1; loop-back snapshot advisory #3)*. Production capability ensembles carry `output_substrate: artifact` (ADR-025) — deliverables route to the server-side `SessionArtifactStore` *by design*, with the response carrying a summary plus an `ArtifactReference`, not the content itself. So the client-tool-action terminal must **bridge**: read the artifact the ensemble produced, marshal it into the client `write`/`edit` tool-call content. This is the **designed-in** form of the disjoint-filesystem problem (ADR-025 routes deliverables server-side by architecture, not incidentally), and it does **not** dissolve under co-location — even on one machine the deliverable lives in the artifact store, not where the client expects it. The Tool User's mental model assumes the deliverable arrives at their machine in a useful form; the interaction spec must account for the bridge before ARCHITECT decomposes the implementation. Terminal shape: `dispatch → ensemble writes artifact → framework reads artifact → emit tool_call(content=artifact) → client writes locally`. Cycle 7 DECIDE / ARCHITECT territory; the bridge is unbuilt (Spike ρ used an `output_format: text` stand-in ensemble, not a production artifact-substrate one).
