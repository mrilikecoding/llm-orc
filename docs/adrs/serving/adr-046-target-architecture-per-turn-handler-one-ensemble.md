**Status:** **Accepted (2026-07-02)** — orchestrator-actor dissolution confirmed by the practitioner; argument audit CONVERGED at R4 (Convergence-Saturation TRIGGERED: 0 new P1; P1 trend across the ADR-046 track 4→4→1→0); full §3 component-level re-derivation complete; backward-propagation sweep executed at the decision-record + domain-model level (supersession/partial headers on ADR-001/002/007/009/010/014/016/024/025; domain-model Amendment Log #23 + the AS-1/2/3/5/6/7/9 invariant dispositions, practitioner-approved). The current-state doc sweep (system-design / ORIENTATION / roadmap / field-guide) is deferred to Cycle-8 BUILD, landing with the `agentic/` code deletion. Reports R1–R4 under `housekeeping/audits/argument-audit-decide-cycle-8-adr046*.md`.

## Context

ADR-044 codified AS-11 (agentic serving is declarative-ensemble-native; where the engine is inadequate, extend it, never a parallel layer) and left two things open: **dynamic dispatch** was "to be built as an engine primitive," and the **full per-ADR survives-vs-superseded classification** of the removed `src/llm_orc/agentic/` inventory was deferred to "the target-architecture ADR" (ADR-045 §Decision.1 + Appendix inventory). ADR-045 executed the clean-slate collapse and carried the ADR-033-chain's validated requirements forward as design inputs (its carry-forward table), to be **re-homed declaratively and re-validated**, not assumed.

The Cycle-8 DECIDE-entry grounding spike (`essays/research-logs/cycle-8-spike-dynamic-dispatch-stage-swap.md`) closed the dynamic-dispatch gap and validated the target form with real local models on the 32GB rig (via a direct `EnsembleExecutor` harness; real `opencode run` transport is a2-proven separately from Cycle 7, not re-run for this skeleton — see the Provenance check):

- **Dynamic dispatch shipped** as an L0 engine primitive (commits `64c79ee` shared resolver, `c3709a0` primitive): a `dispatch:` node resolves its target ensemble name at the phase layer (sibling to the guard partition, reusing guard's `${dep.field}` resolver against `results_dict`) and executes it as a child. This is the guard/loop pattern extended, not a parallel layer.
- **The serving turn assembled as ONE declarative ensemble** (classify → seat → marshal), reunifying what Ω-dispatch had to split into decide-ensemble + adapter-invoke + finalize (its own docstring notes the engine "cannot resolve a runtime-chosen `ensemble:` target").
- **All three pre-registered criteria met with real models:** dynamic dispatch resolves and runs inside the engine (#1); a structurally different seat swaps in at zero skeleton change (#2, single-model seat vs. coder+reviewer seat); a non-build turn routes through the same skeleton (#3, explain turn returns prose, not a file).
- **Speed resolved as a config lever, not an architecture limit:** the latency wall was thinking tokens (~97% of generation), not throughput or architecture. Ollama's native `think` param, now wired (commit `6943f8e`), cut build turns ~10x (63.5s → 6.1s) with no quality cost on the tested tasks.

This ADR records the target architecture that evidence supports and performs the classification ADR-045 deferred to it.

## Decision

### 1. The per-turn serving handler is one declarative ensemble: classify → seat → marshal.

The general serving turn (not decompose→build→write; the full-capability handler per the Cycle-8 DISCOVER feed-forward) is a single declarative ensemble of typed **seats** joined by the common I/O envelope (ADR-024):

- **classify** (decider seat) — reads the turn, emits the routing decision `{target, ...}`. Deterministic where the signal is structural (file extension, language, fenced tag); model-backed where it must read intent.
- **seat** (capability seat) — a **dynamic-dispatch** node (`dispatch: "${classify.target}"`): the target ensemble name resolves at the phase layer against `results_dict`, and the resolved ensemble runs as a child. Any contract-satisfying ensemble fills it.
- **marshal** (deterministic finalize) — a `script:` node shaping the seat output into the serve outcome (a client tool_call for a deliverable, a prose finish otherwise). This is ADR-034's terminal + ADR-035's form gate re-homed as a script node.

No imperative orchestration graph. The runtime-chosen target resolves inside the engine.

### 2. Seats are swappable behind contracts; the decider is itself a seat.

- The **fixed skeleton is the sequence of typed seats and their contracts**; everything filling a seat is fluid. A seat swaps at zero skeleton change (spike #2).
- **The seat contract is the existing `core/validation/` framework** (`ValidationConfig`: structural/schema/behavioral/quantitative/semantic layers; pass/fail `ValidationResult`), which survives the `agentic/` removal and was built to enable exactly this composition. It is used:
  - **seat-owned, not candidate-owned** — the contract lives on the seat; candidates are validated against it. Candidate-owned assertions are the builder grading its own homework (the §6.2b independence trap relocated to the registry). The evaluator already takes the config as a separate argument (`ValidationEvaluator.evaluate(name, results, validation_config)`), so seat-owned is a convention, not a rebuild.
  - **black-box** — I/O-facing layers only (schema/behavioral/quantitative on the output). The **structural** layer names internal agents (`required_agents`) and would couple a seat to a candidate's internals, breaking interchangeability.
  - **deterministic-first** — the semantic (LLM-judge) layer is optional and soft; seat eligibility rests on the deterministic layers.
  This grounds Q2 / §6.2b: the contract assertions are the independent correctness signal the loop's accept decision needed. **Caveat:** the spike *analyzed* this framework (F3, design analysis); it did not wire `ValidationConfig`/`ValidationEvaluator` as the seat's pass/fail gate — the swap tests asserted correctness by inspection. So seat-contract *wiring* (not just its sufficiency) is unvalidated (§Open).
- **The decider can be a seat too** (design extension, not yet spike-validated). The validated skeleton wired `classify` as a `script:` node, and the decider-swap arm of the spike's test matrix was not run; whether the seat *sequence* itself is swappable is open (spike F2). The design admits filling the decider by one frontier call or a multi-stage deliberation ensemble behind the same contract — routing composed, no privileged router — but that is a proposal pending its own falsification probe, not a spike result.

### 3. Per-ADR classification of the `agentic/` inventory — ADR-045's deferred first task.

**Test (from ADR-045):** an ADR's decided *imperative-orchestration mechanism* is **superseded** (implementation removed, re-homed declaratively); a *contract / invariant / deterministic-work / direction* **survives** (implementation re-homed, decision intact). An extension is not classified more definitively than its base (ADR-045 guidance b).

**Already headered (confident, per ADR-045):** superseded — ADR-033–043 (loop-driver chain), ADR-015 (tier router), ADR-021 (per-capability ReAct dispatch); ADR-027 chained via ADR-043.

**The master frame — the persistent orchestrator actor is dissolved (practitioner 2026-07-01).** The classification turns on one decision this ADR now makes explicit: **the client (OpenCode, or any caller of the serving endpoint) owns the loop; each request is one declarative ensemble pass; there is no persistent internal orchestrator actor** — no runtime ReAct loop, no runtime `compose_ensemble`, no accumulating orchestrator conversation context, no self-composition trust/promotion. **Persistence does not vanish; it relocates** to the substrate the ensembles read and write turn to turn: the artifact substrate (ADR-025), the session registry / structured-handoff artifacts (ADR-013), and, when active, the knowledge graph (ADR-009/010 / Plexus). This resolves OQ #26 (callee / client-owned loop) as a first-class decision. It drives the table: mechanisms belonging to the dissolved actor are **superseded**; the persistence substrate, the seams, and deterministic work **survive** (re-homed); cross-turn state lives in the substrate, not in an actor's memory.

### 4. Thinking-mode is a per-seat routing dimension (the §6.3 / Q3 declarative home).

The spike measured thinking-off as a ~10x interactive-latency lever with no quality cost on the tested (easy) tasks, and confirmed thinking is not needed for those turns while remaining untested on hard tasks. So thinking on/off is a **per-seat control and a routing signal**: easy/interactive turns route to a thinking-off seat; hard turns to a thinking-on seat (or a thinking-routed decider). Speed is a routing/config dimension in this architecture, not a limit. This generalizes §2's "seats are a dial" to the thinking axis (alongside model size and verify-heaviness).

## Rejected alternatives

- **decompose → build → write (the Ω-P3 / narrow-serve shape).** Rejected as the target: the Cycle-8 DISCOVER feed-forward settled full model-parity via composition (the serve does everything a single model does, no capability degradation), so the target is a general per-turn handler, not a build-new-files pipeline. Ω-P3 is a research demo of one sub-case.
- **A dedicated router/dispatch component (the Ω-dispatch adapter shape).** Rejected: AS-11 forbids the parallel layer, and the spike showed the decider is just another seat behind a contract. A privileged router re-introduces the control-flow-in-Python seam AS-11 closes.
- **Classifying the deferred ADRs from ADR-045's manifest alone (no per-ADR read).** Rejected: ADR-045 §Rejected-alternatives already showed the mechanism test states the principle but per-ADR application misfires without deciding the declarative form (it over-classified ADR-012/016). The declarative form is now decided (§1–§2), so the read is enabled; it is still required, not skippable.

## Consequences

**Positive:**
- One architecture, validated end to end with real models and interactive on the target rig.
- The carry-forward requirements (ADR-045 table) get declarative homes (§3): 037/038/036/043 via shipped primitives + the serving ensemble, and 034/035 via the marshal node; 040/041 and 039 are candidates (not re-validated); 033 is out of scope (the client owns the multi-turn loop).
- Contract-testing is existing capability re-pointed (seat-owned validation framework), not new mechanism.

**Negative / cost:**
- The [read] classification pass is real work with corpus-wide supersession stakes; this ADR is not Accepted until it and the argument audit complete.
- Extensibility (registry + binding time) and full grounded acceptance remain open decisions.

**Neutral:**
- The architecture is provider-agnostic; the inference backend (Ollama vs. llama-server) sits below the ensemble layer and is a separate tuning decision.
